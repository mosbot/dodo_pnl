# ПланФакт API — OPERATIONS для AI-агента

Этот файл покрывает все шесть типов операций (справочник), operation-driven сценарии: чтение списка операций, server-side поиск, сводки по операциям, создание / изменение / удаление операций, а также special case `/operations` на рабочем стенде.

`README.md` — policy, router, budgets, cache.
`QUICKSTART.md` — короткий technical index.
`USE_CASES.md` — бизнес-семантика и `method + scope`.
`LOOKUPS.md` — как добывать `accountId`, `contrAgentId`, `projectId`, `operationCategoryId`.
`QUICKFILTERS.md` — быстрые фильтры операций и публичные методы `QuickFilters`.

## Когда открывать

- нужно понять, что такое конкретный тип операции и какие у него нюансы;
- нужен список операций;
- нужен точечный поиск по `comment`, сумме, статье, контрагенту;
- нужна аналитика по статьям, проектам, суммам или датам начисления;
- нужен accrual-расчет по `operationParts`;
- нужно создать, изменить или удалить операцию;
- нужен быстрый фильтр или сохраненный фильтр в разделе **Операции**;
- нужно понять, почему `/operations` не совпадает с UI или ведет себя как full dump.

## Default route для operations

Для operation-driven read-only сценариев путь по умолчанию:

1. Сначала `POST /api/v1/operations/list` с максимально узкими server-side фильтрами.
2. Использовать `GET /api/v1/operations` только если:
   - post-check показал неполноту `operations/list`;
   - задаче действительно нужен полный raw-layer;
   - пользователь явно согласился на широкий full dump;
   - в текущей сессии уже есть актуальный full operations-cache.

`GET /api/v1/operations` не является стартовым маршрутом по умолчанию.

### Full dump — только fallback

Не начинать read-only сценарий с `GET /api/v1/operations` только потому, что:

- нужен список операций;
- период широкий;
- пользователь просит таблицу, HTML или расшифровку;
- агент хочет «подстраховаться» полным raw-layer.

Широкий формат результата не является основанием для полного dump.

## Справочник типов операций

| Тип         | `operationType` | Деньги двигаются? | Дата учёта                            |
| ----------- | --------------- | ----------------- | ------------------------------------- |
| Поступление | `Income`        | ✅ приходят        | `operationDate`                       |
| Выплата     | `Outcome`       | ✅ уходят          | `operationDate`                       |
| Начисление  | `Accrual`       | ❌                 | `calculationDate` (= `operationDate`) |
| Перемещение | `Move`          | ✅ внутри бизнеса  | `operationDate`                       |
| Отгрузка    | `Shipment`      | ❌                 | `calculationDate`                     |
| Поставка    | `Supply`        | ❌                 | `calculationDate`                     |

> **Важно:** колонка `Дата учёта` в справочнике — это базовая дата операции, а не универсальная дата для всех показателей.  
> Для денежных показателей использовать `operationDate`; для accrual-показателей и аналитики по начислению использовать `operationParts[].calculationDate` и подтверждение даты начисления.  
> То есть участие операции в расчете зависит не только от типа операции, но и от **показателя** и **метода**. 

### Поступление (`Income`)

Деньги пришли на счёт бизнеса от контрагента.

**Ключевые поля:**

- `operationDate` — дата оплаты (дата реального зачисления)
- `isCommitted` — подтверждена ли дата оплаты
- `operationParts[].calculationDate` — дата начисления для accrual-аналитики
- `operationParts[].isCalculationCommitted` — подтверждена ли дата начисления.

**Нюансы:**

- Привязана к статьям - Доходы, Активы, Обязательства, Капитал (кроме Дивидендов). Привязана к сущностям - Счёт, Контрагент, Статья, Проект, Сделка.
- В `operationParts` хранятся аналитические разрезы: статья, проект, сумма, дата начисления.
- Для аналитики по статье, контрагенту, проекту, сделке, дате начисления и подтверждению даты начисления использовать `operationParts`, даже если операция не разбита по частям.  
  - Если операция разбита по частям — сумму по проекту / статье / сделке считать по `operationParts[].valueInUserCurrency`, а количество операций — по уникальным `operationId`.  
  - Поступление может участвовать:  
  - в денежных показателях — по `operationDate` и `isCommitted`;  
  - в accrual-показателях — по `operationParts[].calculationDate` и `operationParts[].isCalculationCommitted`.
- Специальных флагов нет; участвует в cash ОПУ, accrual ОПУ и ДДС без условий (кроме подтверждения даты).

### Выплата (`Outcome`)

Деньги ушли со счёта бизнеса контрагенту.

**Ключевые поля:** те же, что у Поступления.

**Нюансы:** зеркальна Поступлению — те же правила дат, флагов и `operationParts`. Не путать: `Income` ≠ `Outcome` при создании через API.

- Привязана к статьям - Расходы, Активы, Обязательства, Капитал (Вложений учредителей). 

### Начисление (`Accrual`)

Операция типа Начисление работает как классическая бухгалтерская проводка и переносит средства с одной статьи учета на другую.

**Ключевые поля:**

- на верхнем уровне `operationDate` равен `calculationDate` — это не баг; у Начисления нет отдельной даты оплаты, поэтому оба поля указывают на дату начисления
- для accrual-аналитики, как и для других типов операций, дату начисления брать из `operationParts[].calculationDate`, а не с верхнего уровня
- `isCommitted` и `isCalculationCommitted` отражают одно и тоже подтверждение дат
- `isOpuCalculation` — флаг «Учитывать в кассовом методе»

**Нюансы:**

- Начисление — это неденежная операция. Оно не попадает в ДДС, БДДС, блок «Движение денег» Платёжного календаря и любые другие показатели, основанные на движении денег.

- В кассовом методе в ОПУ, Платёжном календаре, БДР, показатели внутри Проекта, КА, Сделок продаж, и возможные другие показатели, которые связаны с прибылью начисление участвует **только если `isOpuCalculation == true`**.

- В accrual ОПУ участвует всегда при `isCalculationCommitted == true`.

- `false` у `isOpuCalculation` — значимое значение, не отсутствие поля. Не применять `// true` как fallback.

- У Начисления счёт должен быть служебным. (`account.isUndistributed == true`).

- Начисление всегда должно порождать связанную запись через `boundMoveOperationId`; иначе часть данных потеряется.

### Перемещение (`Move`)

Деньги перемещаются **внутри бизнеса** между двумя счетами. Деньги не теряются и не появляются — меняется только их местонахождение.

**Ключевые поля:**

- `operationDate` — дата перемещения
- `isCommitted` — подтверждена ли дата
- `boundMoveOperationId` — связывает пару записей

**Нюансы:**

- В API одно перемещение = **два объекта** с одинаковым `boundMoveOperationId`: списание и зачисление.
- Нет статьи доходов/расходов. Нет контрагента.
- Не участвует в ОПУ ни по одному методу.
- В пользовательском списке и итоговых карточках сумма перемещения = **0**. Raw `value` / `valueInUserCurrency` у пары можно использовать только как техническую сумму внутреннего перевода между счетами, но не как доход, расход или «общую сумму операции».
- В денежных показателях участие зависит от отчёта:
  - в некоторых агрегированных показателях движения денег перемещения-зачисления могут входить в Поступления, а перемещения-списания — в Выплаты;
  - в ДДС и других отчетах с разбивкой по статьям перемещения обычно показываются отдельной строкой;
  - в Платёжном календаре и БДДС перемещения могут не выводиться как отдельные строки деятельности, но влияют на остатки.
- Для UI-подобного вывода пару записей сворачивать в одну операцию по `boundMoveOperationId`.
- Если у одной из пар счёт `isUndistributed == true` — это, скорее всего, Начисление, а не обычное перемещение.

### Отгрузка (`Shipment`)

Факт **отгрузки товара или оказания услуги** по сделке продаж. Деньги не двигаются — фиксируется только факт выполнения обязательства.

**Ключевые поля:**

- `calculationDate` — дата отгрузки
- `isCalculationCommitted` — подтверждена ли дата отгрузки
- `sellingDealId` — привязка к сделке продаж

**Привязка к статьям:** Доходы (любая), Активы и Обязательства (только статьи, к которым применимо начисление).

**Нюансы:**

- Неденежная: не попадает в показатели по Движению денег (ДДС, БДДС, cash ОПУ, Платёжный календарь) и Прибыли кассовым методом.
- Участвует в **accrual ОПУ** по статьям доходов при `isCalculationCommitted == true`.
- Отгрузки по статьям **Активов / Обязательств** участвуют в Балансе и КА (дебиторка, кредиторка, оборот), но **не в Прибыли**.
- Используется для расчёта задолженности по сделкам продаж: если сумма Отгрузок > суммы Поступлений по сделке — денежная дебиторка.
- В список операций включается, но в «общую сумму всех операций» автоматически не добавлять.
- Если пользователь запрашивает прибыль, уточнить метод: при кассовом методе Отгрузки не участвуют.
- Прибыль сделки и формулы расчёта → `DEALS.md`, раздел «Прибыль сделки».

### Поставка (`Supply`)

Факт **получения товара или услуги** по сделке закупок. Деньги не двигаются — фиксируется только факт принятия обязательства.

**Ключевые поля:**

- `calculationDate` — дата поставки
- `isCalculationCommitted` — подтверждена ли дата поставки
- `purchaseDealId` — привязка к сделке закупок

**Привязка к статьям:** Расходы (любая), Активы и Обязательства (только статьи, к которым применимо начисление).

**Нюансы:**

- Зеркальна Отгрузке, но для стороны закупок.
- Участвует в **accrual ОПУ** по статьям расходов при `isCalculationCommitted == true`.
- Поставки по статьям **Активов / Обязательств** участвуют в Балансе и КА, но **не в Прибыли**.
- В расчётном списке purchase deal поставка увеличивает `providedValue`, но не превращается в `income/outcome/profit` сделки; прибыль и рентабельность сделки есть только для продаж.
- Используется для расчёта задолженности по сделкам закупок: если сумма Выплат > суммы Поставок — неденежная дебиторка.
- Не участвует в ДДС, БДДС, cash ОПУ, Платёжном календаре.
- В список операций включается, но в «общую сумму» автоматически не добавлять.

### Участие в списке, суммах и показателях

| Контекст                    | Отгрузки / Поставки                                                                                                               |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| Список операций             | Включаются                                                                                                                        |
| Количество операций         | Включаются                                                                                                                        |
| Общая сумма операций        | **Не включать автоматически**                                                                                                     |
| ДДС / движение денег / БДДС | **Не включаются** — только реальное движение денег                                                                                |
| Прибыль кассовым методом    | **Не включаются** — только Поступления / Выплаты с подтверждённой датой оплаты; Начисления, у которых `isOpuCalculation == true`. |
| Прибыль методом начисления  | **Включаются** — с подтверждённой датой начисления / отгрузки / поставки                                                          |

### Правило выбора метода

Если пользователь просит доходы, прибыль, рентабельность или маржинальность — сначала определить метод:

- **Кассовый метод** → участвуют:
  - Поступления / Выплаты с подтверждённой датой оплаты;
  - Начисления, у которых `isOpuCalculation == true`.
  - Отгрузки и Поставки не участвуют.
- **Метод начисления** → участвуют:
  - Отгрузки / Поставки и другие релевантные операции с подтверждённой датой начисления;
  - Поступления / Выплаты / Начисления — только если у них подтверждена дата начисления.
  - Поступления и Выплаты с неподтверждённой датой начисления не участвуют.

Если метод не указан, а от него зависит состав операций — задать один уточняющий вопрос до расчёта.

## Contract-first и runtime caveats

Полные правила поведения с `/api/v1/operations` (full-dump caveat, кэш, инвалидация) — в `README.md`, раздел `/api/v1/operations — особое поведение`. Здесь только runtime-специфичные детали.

- Для типовых read-only operations-сценариев сначала использовать `POST /api/v1/operations/list`.
- `GET /api/v1/operations` использовать не по умолчанию, а только как fallback или когда задаче реально нужен полный raw-layer.
- `apidoc.planfact.io` открывать не «сначала», а как contract-reference: write-payload, редкий параметр, непокрытая схема или ошибка contract.
- `GET /api/v1/operations` и `POST /api/v1/operations/list` не считать взаимозаменяемыми:
  - `GET /api/v1/operations?searchString=...` может проигнорировать фильтр и вернуть полный dump;
  - `POST /api/v1/operations/list` — основной narrow-first маршрут для server-side фильтрации списка.
- `POST /api/v1/operations/list` считать read-only endpoint'ом поиска, а не write-запросом.

⚠️ **`POST /api/v1/operations/list` с фильтром по дате возвращает неполный набор:**  
На рабочем стенде `operationDateStart` / `operationDateEnd` в этом endpoint'е отсекают плановые операции с будущими датами — даже если они попадают в запрошенный период (например, запрос `2026-01-01…2026-04-22` не вернёт операцию с датой `2026-04-26`, даже если та относится к 2026 году). Итоги по такому ответу будут занижены.  
Это caveat для post-check, а не разрешение начинать с full dump. Сначала использовать `POST /api/v1/operations/list` с narrow-фильтрами. Переходить к `GET /api/v1/operations` только если post-check показал неполноту, нужен полный слой для явно описанной задачи или пользователь согласился на широкий raw-layer.

## Service methods: сводка vs расшифровка

Эти методы нужны, чтобы не грузить полный список операций, когда пользователю нужен быстрый служебный ответ.

| Endpoint                                                   | Когда использовать                                                             | Что возвращает                                                        | Если нужна расшифровка                                                            |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------ | --------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `GET /api/v1/operations/dates-summary`                     | пользователь не дал период, просит «за всё время», нужно понять границы данных | только границы дат операций и начислений по аккаунту                  | не подходит; для списка идти в `POST /operations/list` с фильтрами                |
| `GET /api/v1/operations/get-undistributed-operation-count` | пользователь спрашивает «сколько нераспределённых операций дохода/расхода»     | одно число: count income/outcome операций по нераспределённым статьям | не подходит; получить ID нераспределённых статей и идти в `POST /operations/list` |
| `POST /api/v1/operations/linked-deals-summary`             | нужна сводка по операциям, привязанным/непривязанным к сделке                  | counts + суммы доходов, расходов, отгрузок, поставок                  | не подходит; для списка идти в `POST /operations/linked-deals`                    |
| `POST /api/v1/operations/linked-deals`                     | нужен список операций, привязанных/непривязанных к сделке                      | список операций с пагинацией                                          | это правильный метод для расшифровки операций сделки                              |

Правило выбора:

- если пользователь просит `сколько`, `итог`, `сводку`, `есть ли данные`, `границы периода` — сначала использовать service/summary endpoint;
- если пользователь просит `покажи операции`, `расшифровку`, `по датам`, `по операциям`, `с комментариями`, `таблицу операций` — нужен list endpoint, а не summary/count;
- если пользователь просит операции из конкретной загрузки/импорта — сначала получить `importLogId` через `LOOKUPS.md` → `ImportLogs`, затем фильтровать операции по `filter.importLogId`;
- не делать fallback в полный `/operations`, если service endpoint не даёт расшифровку. Сначала выбрать узкий list endpoint и фильтры.

### Нераспределённые операции

`GET /api/v1/operations/get-undistributed-operation-count` возвращает только количество income/outcome операций по системным статьям `IncomeUndistributed` и `OutcomeUndistributed`.

Использовать его только для вопроса уровня «сколько нераспределённых операций».

Если пользователь просит расшифровку нераспределённых операций:

1. получить ID статей `Нераспределённый доход` и `Нераспределённый расход` через справочник статей;
2. вызвать `POST /api/v1/operations/list` с `operationType: Income/Outcome`, найденными `operationCategoryId` и периодом пользователя;
3. если период не задан и список может быть большим — сначала уточнить период или предупредить о широком запросе.

Не использовать `get-undistributed-operation-count` для запросов «по датам», «по операциям», «суммы по каждой операции», «выгрузи список».

## Быстрые фильтры операций

Если пользователь спрашивает про быстрый фильтр, сохраненный фильтр или вкладку **«Быстрые»** в разделе **Операции**, открывать `QUICKFILTERS.md`.

Короткое правило:

- быстрый фильтр операций может сохранять только параметры стандартного фильтра, доступные в аккаунте;
- если поле скрыто в стандартной фильтрации, его нельзя использовать в быстром фильтре;
- если в аккаунте скрыты типы **Начисление / Отгрузка / Поставка**, они не должны отображаться в форме быстрого фильтра;
- если параметр недоступен по тарифу или настройкам, результат фильтрации пустой и пользователь видит ошибку / информер;
- для управления быстрыми фильтрами использовать методы `/api/v1/quick-filters*` из `QUICKFILTERS.md`, а не методы `/operations`.

## Как выбирать маршрут

### Decision tree: `/operations` vs `/operations/list` vs service endpoints

1. Если нужен готовый агрегат, а не список или расшифровка операций — не начинать с operations-layer. Вернуться в `README.md → Intent Router` и выбрать отчётный, budget, deal или lookup endpoint.

2. Если нужен список / расшифровка операций:
   - если актуальный `/operations` уже есть в кэше текущей сессии — фильтровать локально, сетевой запрос не делать;
   - если есть точный `operationId`, известный `contrAgentId`, `projectId`, `accountId`, `accountCompanyId`, `searchString` или узкий период — сначала использовать `POST /api/v1/operations/list`;
   - если post-check после `POST /api/v1/operations/list` показал неполноту по будущим плановым операциям или нужен полный raw-layer для дальнейшей локальной аналитики — явно назвать причину и только затем использовать `GET /api/v1/operations` один раз на сессию;
   - если запрос широкий и `/operations` ещё не загружен — предупредить пользователя заранее по правилу из `README.md`.

3. Если нужна только служебная сводка, count или границы периода — использовать service endpoint (`summary`, `dates-summary`, `get-undistributed-operation-count`, `linked-deals-summary`) и не загружать полный список.

4. Запрещён каскад без новой причины:
   - `POST /operations/list` → `GET /operations/summary` → full `GET /operations`;
   - service endpoint → full `GET /operations`;
   - `POST /operations/list` с ошибкой параметра → full `GET /operations`.

Переход к full `GET /operations` допустим только если выбранный endpoint принципиально не покрывает расшифровку, пользователь согласился на широкий raw-layer, уже есть актуальный кэш или нужен полный слой для задачи, явно описанной пользователем.

### Агрегированный показатель (выручка, прибыль, расходы за период, по проектам)

Если нужен **готовый агрегат**, а не список сырых операций — сначала проверить, покрывает ли запрос отчётный endpoint:

- выручка, прибыль, расходы за период → `POST /api/v2/reports/opu` (`REPORTS_V2.md`)
- расходы / доходы по проектам (accrual-ОПУ с разбивкой по статьям) → `POST /api/v2/reports/opu` с `reportGenMethod: Projects`
- план из бюджета + факт по проектам → `GET /api/v1/projects/calculations` (`LOOKUPS.md`); `reports/opu` плана не возвращает
- дебиторка / кредиторка по одному проекту → `GET /api/v1/projects/{id}/summary` (`LOOKUPS.md`)
- список, сводка, прибыль или рентабельность сделок → `DEALS.md`; не начинать с `/operations/list`
- денежный поток по видам деятельности → `POST /api/v2/reports/dds` (`REPORTS_V2.md`)
- cash-динамика по статьям / проектам / контрагентам → `bizinfos` (`REPORTS.md`)
- структура доходов / расходов (базовый) → `bizinfos/incomeoutcomehistorybyoperationcategories` (`REPORTS.md`)
- структура доходов / расходов (ПРО) → `POST /api/v2/dashboardpro/paymentstructure-*` (`DASHBOARDS_PRO.md`)

Грузить сырые `/operations` и считать агрегат вручную — **только если отчётный endpoint не покрывает сценарий** (нестандартная группировка, фильтр, недоступный в отчёте, или нужен одновременно и список, и итог).

### Список операций за период

- по умолчанию сначала `POST /api/v1/operations/list` с narrow-фильтрами периода, статьи, проекта или контрагента;
- после ответа делать post-check фильтров и полноты;
- full `GET /api/v1/operations` допустим только при доказанной неполноте `operations/list`, наличии актуального кэша или явном согласии пользователя на широкий raw-layer;
- если нужен UI-like список, сворачивать связанные записи.

### Точечный поиск малого набора операций

- Если `/operations` уже в кэше текущей сессии — фильтровать локально. Сетевой запрос не делать.
- сначала делать `narrow-first` запрос с самым узким явным периодом;
- если операции свежие, сначала пробовать `lastModificationDateStart` / `lastModificationDateEnd` в JSON body; в official `apidoc` ReDoc эти поля могут отображаться как `filter.lastModificationDateStart` / `filter.lastModificationDateEnd`;
- если нужен именно server-side текстовый поиск, сначала пробовать `POST /api/v1/operations/list`;
- `GET /operations` использовать только если актуальный dump уже есть в кэше или пользователь явно согласился на широкий raw-layer; не считать его автоматическим fallback после ошибки `POST /api/v1/operations/list`.

### Узкий список по известным ID

- если уже известны `contrAgentId`, `projectId`, `accountId` или `accountCompanyId`, сначала пробовать `POST /api/v1/operations/list` с этими фильтрами и узким периодом;
- это предпочтительнее, чем сначала грузить полный `/operations` и потом фильтровать локально;
- если server-side фильтр для нужного поля не сработал или endpoint вернул ошибку параметра — сначала исправить payload, а не переключаться молча на полный dump.

### Грубая сумма по операциям

- сначала проверить, подходит ли `GET /api/v1/operations/summary`;
- помнить, что `summary` считает сырые записи и может не совпадать с UI.

### Accrual-аналитика

- использовать `/operations`, а не `bizinfos`;
- смотреть не только верхний уровень операции, а `operationParts`;
- использовать `calculationDate` и `isCalculationCommitted`.

### Guardrail для таблиц со списком операций и итоговой суммой

Если пользователь просит **список операций** и одновременно хочет видеть **итоговую сумму**, по умолчанию:

- список строк можно оставлять широким и включать `Shipment` / `Supply`;
- если в списке есть UI-свернутые `Move`, их отображаемая сумма = `0`; raw-сумму внутреннего перевода не показывать как сумму операции и не добавлять в итог;
- карточку или колонку `Общая сумма операций` считать без автоматического включения `Shipment` / `Supply`;
- если такие строки есть в списке, их сумму показывать отдельно, например как `Отгрузки вне суммы` или `Поставки вне суммы`;
- если пользователь явно просит не `общую сумму операций`, а именно `сумму доходных частей` или `сумму accrual-строк`, это нужно отдельно проговорить и посчитать как другой показатель.

## Расчёт прибыли из операций

> **Для получения готовой прибыли использовать `POST /api/v2/reports/opu`** (`REPORTS_V2.md`).
> Ручной расчёт из сырых операций нужен **только** если v2 не покрывает сценарий
> (нестандартная группировка, нужен одновременно список и итог, фильтр недоступен в отчёте).
> Раздел ниже — справочник формул для таких случаев.

### Какие статьи участвуют

- Прибыль считается только по операциям, у которых статья относится к **Доходам**
  (`operationCategoryType == Income`) или **Расходам** (`operationCategoryType == Outcome`)
- Для **Нераспределённой прибыли** дополнительно включаются операции по статьям **Дивидендов**
- Все остальные типы статей (Активы, Обязательства, Капитал кроме Дивидендов) — не участвуют

### Кассовый метод

Участвуют три типа операций: **Поступления, Выплаты, Начисления**.
Отгрузки, Поставки, Перемещения — не участвуют.

**Факт** — операции с подтверждённой датой оплаты:

- Поступления / Выплаты: `isCommitted == true`
- Начисления: `isCommitted == true` + `isOpuCalculation == true`
- Нюанс API: на верхнем уровне у Начислений `operationDate` равен `calculationDate` — это не баг; у Начисления нет отдельной даты оплаты, поэтому оба поля указывают на дату начисления.
  Подтверждение проверять по `isCommitted`.

**План** — те же типы операций, статус подтверждения не важен:

- Поступления / Выплаты: `isCommitted` может быть любым
- Начисления: `isCommitted` может быть любым, но `isOpuCalculation == true` обязателен

### Флаг `isOpuCalculation` («Учитывать в кассовом методе»)

- Присутствует только у Начислений
- Если `isOpuCalculation == false` — начисление **исключается** из всех cash-расчётов ОПУ и Платёжного календаря
- Если `isOpuCalculation == true` — начисление участвует наравне с Поступлениями и Выплатами
- `false` — значимое значение, а не отсутствие поля. Не применять `// true` как fallback
- Актуален только при ручной реконструкции из операций; `POST /api/v2/reports/opu` учитывает флаг внутри сам

### Метод начисления

| Тип операции | Участвует | Условие                          |
| ------------ | --------- | -------------------------------- |
| Поступление  | ✅         | `isCalculationCommitted == true` |
| Выплата      | ✅         | `isCalculationCommitted == true` |
| Начисление   | ✅         | `isCalculationCommitted == true` |
| Отгрузка     | ✅         | `isCalculationCommitted == true` |
| Поставка     | ✅         | `isCalculationCommitted == true` |
| Перемещение  | ❌         | всегда исключается               |

### Аналитика по статьям, проектам и суммам

- если аналитика зависит от статьи, проекта, суммы части или даты начисления, контрагента, сделки продаж, сделки закупок смотреть прежде всего в `operationParts`;
- это правило действует не только для accrual, но и для cash-аналитики по разрезам, если операция разбита на части;
- сама операция при этом остается одной бизнес-операцией, а не набором независимых строк.

## `operationDate` vs `calculationDate`

- `operationDate` — дата оплаты / дата движения денег.
- `calculationDate` — дата начисления; для accrual-аналитики канонически смотреть `operationParts[].calculationDate`.
- Одна и та же операция может иметь и `operationDate`, и `calculationDate`; это нормально, не дубликат.
- Одна и та же операция может попасть в разные периоды в `cash` и `accrual`; это нормальная логика учета, а не ошибка данных.
- Базовое правило: не строить fallback-логику `если calculationDate отсутствует, взять operationDate`.
- Если дата начисления нужна для аналитики, брать её только из `operationParts[].calculationDate`.
- Runtime caveat: у части операции, привязанной к сделке через `sellingDealId` / `purchaseDealId`, `calculationDate` может отсутствовать.
  - также возможен случай, когда `operationParts[].calculationDate` автозаполнена датой оплаты, но `calculationPeriodDate == null` и `operationParts[].isCalculationCommitted == false`; для debt-реконструкции это считать отсутствующей явной датой начисления;
  - это не даёт права подменять дату начисления датой оплаты;
  - такую часть нельзя включать в accrual-фильтрацию и в реконструкцию кредиторской задолженности по стороне начисления;
  - до появления `calculationDate` такую строку считать непригодной для расчёта по accrual-стороне.

**Фильтрация по accrual-периоду:** при отборе операций, попадающих в период начисления, сравнивать `operationParts[].calculationDate` с границами периода — **не** `operationDate`. Использование `operationDate` как замены `calculationDate` при accrual-фильтрации даёт неверный результат: операции с оплатой в периоде, но начислением вне него, будут включены или исключены ошибочно.

## `operationParts`

- если в операции есть `operationParts`, использовать их для расчета аналитики;
- для сумм аналитики использовать `operationParts[].valueInUserCurrency`;
- аналитические разрезы брать из частей операции:
  - статья;
  - проект;
  - сумма;
  - сделка;
  - контрагент;
  - дата начисления;
- общие атрибуты брать с верхнего уровня операции:
  - дата оплаты;
  - подтверждение оплаты;
  - счет;
  - юрлицо;
  - валюта;
  - тип операции;
- количество операций считать по уникальным `operationId`, а не по числу частей;
- части операции не трактовать автоматически как независимые бизнес-операции;
- в пользовательском выводе одна операция с несколькими частями должна оставаться одной операцией с внутренним разнесением по частям.
- если аналитика строится по прибыли, выручке или accrual-результату, не включать в такой расчет перемещения как доходный или расходный факт.

## `comment` и текстовый поиск

- пользовательское `назначение платежа` по умолчанию трактовать как верхнее поле `comment`;
- для server-side text search использовать `POST /api/v1/operations/list` с `searchString` в JSON body; в official `apidoc` ReDoc это поле может отображаться как `filter.searchString`;
- поиск работает как **substring-поиск** (case-insensitive): `"Оплата"` найдёт и «Оплата», и «Предоплата», и «Оплата услуг» — всё, что содержит подстроку;
- если нужно **точное совпадение** — после server-side запроса дополнительно фильтровать локально по `comment == "строка"`;
- `GET /operations?searchString=...` на этом стенде может игнорировать фильтр — не полагаться на него для текстового поиска.

⚠️ **Post-check для `POST /operations/list` с `searchString`:**
- после ответа **всегда** проверять `isSuccess`, диапазон дат и что выборка соответствует поисковой строке;
- серверные фильтры в этом API лучше не принимать на веру без проверки.

## Кэш и reload

- Full `/operations` cache живёт только в рамках текущей сессии.
- Новая сессия не наследует operations-cache из предыдущей.
- Если в новой сессии нужен raw-layer, агент должен считать, что `/operations` ещё не загружен.
- В начале новой сессии кэша нет. Любой слой, нужный для ответа, загружать заново.
- `/operations` считать одним полным fetch на сессию; повторно не загружать, если уже есть кэш.
- Исключение: если пользователь сообщил о внешних изменениях или цифры расходятся с UI — инвалидировать кэш и сделать один fresh reload.
- Для инкрементальной подгрузки использовать `lastModificationDateStart` / `lastModificationDateEnd` в JSON body `POST /api/v1/operations/list`; в official `apidoc` ReDoc эти поля могут отображаться как `filter.lastModificationDateStart` / `filter.lastModificationDateEnd`.
- Не передавать полный JSON `/operations` в контекст LLM; сначала сокращать до нужных полей.

## Типовые write-сценарии

### Параллельные lookups перед write

Для создания операции нужны несколько ID из разных справочников (`accountId`, `contrAgentId`, `operationCategoryId` и др.). Эти lookup-запросы **независимы друг от друга** — выполнять их параллельно, а не последовательно.

### Создать поступление

- endpoint: `POST /api/v1/operations/income`
- до запроса нужны: `accountId`, `contrAgentId`, `operationCategoryId`, дата, сумма
- если операция распределенная, сумма может жить в `items[]` или `valueByProjects`

### Создать выплату

- endpoint: `POST /api/v1/operations/outcome`
- не путать `Income` и `Outcome`

### Создать перемещение

- endpoint: `POST /api/v1/operations/move`
- нужны два счета, две даты, суммы списания и зачисления
- успешный ответ может вернуть две операции

### Создать начисление

- endpoint: `POST /api/v2/operations/accrual`
- нужны `companyId`, `currencyCode`, дата, дебетовая и кредитовая статьи, сумма
- не путать статьи дебета и кредита

### Изменить или удалить операцию

- сначала получить и показать текущую операцию;
- до `PUT` / `DELETE` показать изменения или целевой объект и дождаться явного подтверждения.

#### Runtime caveat: парные начисления

Перед `PUT` смотреть не только `operationType`, но и `boundMoveOperationId` / `boundMoveOperation`.

Если у операции есть связанная пара и один из счетов служебный (`account.isUndistributed == true`), это начисление (`Accrual`), даже если текущая запись выглядит как `Income` или `Outcome`.

Для изменения использовать `PUT /api/v2/operations/accrual/{operationId}`, а не `PUT /api/v1/operations/income|outcome/{id}`.

У начисления payload пишется через `debitPart` / `creditPart`; общий контрагент передается верхнеуровневым `contrAgentId`. Чтобы убрать контрагента — передать `contrAgentId: null`.

После `PUT` проверить текущую операцию и `boundMoveOperation`.

#### Runtime caveat: изменение даты начисления через `PUT`

Для `PUT /api/v1/operations/income/{id}` и `PUT /api/v1/operations/outcome/{id}` дату начисления изменять через `items[].calculationDate`.

Не использовать `operationParts[].calculationDate` как write-путь: `operationParts[]` — основной путь чтения и аналитики после сохранения. Если передать изменение только в `operationParts[]`, API может вернуть `isSuccess: true`, но фактически пересобрать части операции и оставить старую дату начисления или вернуть `calculationDate = operationDate`.

Флаг подтверждения даты начисления передавать в том же элементе `items[]`:

- `items[].isCalculationCommitted = true` — дата начисления подтверждена;
- `items[].isCalculationCommitted = false` — дата начисления не подтверждена.

Если пользователь просит изменить и зафиксировать дату начисления, передать:

- `items[].calculationDate = "YYYY-MM-DD"`;
- `items[].isCalculationCommitted = true`.

После `PUT` сделать контрольный `GET /api/v1/operations/{id}` и проверять результат по `operationParts[].calculationDate` и `operationParts[].isCalculationCommitted`.

### Batch write-запросы

- для массовых `POST` / `PUT` / `DELETE` не использовать длинные inline shell-команды с многоуровневым экранированием;
- сначала собирать каждый payload в отдельный `.json` файл;
- затем отправлять write-запросы последовательно, по одному;
- на каждый запрос сохранять raw response и логировать `operationId`, `isSuccess`, `errorMessage`;
- после batch-run не делать full verify по привычке: по умолчанию достаточно выборочно проверить `1–3` первых операций партии и все structurally special кейсы (`split`, `move`, новая валюта, новый счёт, новый тип payload);
- полный verify каждой записи делать только если пользователь явно просит audit trail, если операция high-stakes или если уже есть признаки расхождения между write-response и фактическим состоянием;
- при создании нескольких счетов подряд использовать `accountId` из ответа `POST /api/v1/accounts` как immediate source of truth; не обновлять `/accounts` после каждого create, если ID уже известен;
- batch refresh справочника счетов делать один раз в конце серии create-запросов или только перед следующим шагом, которому действительно нужен свежий `/accounts`;
- если в payload операции не передаётся `contrAgentId`, не делать lookup в `/contragents` только ради имени из чека или платежки; в таком сценарии имя получателя или магазина должно уходить в `comment`;
- если одиночный write проходит, а batch нет — сначала проверять shell-сценарий, сборку payload и логирование, а не обвинять API.

### Ограничения привязки операции к сущностям

Перед отправкой write-запроса агент должен проверить, что все сущности, к которым привязывается операция, доступны для использования.

Порядок проверки должен быть blocking-first: сначала lookup сущностей, которые могут полностью запретить создание операции. Если блокер найден, агент должен немедленно остановиться — не искать остальные ID (`accountId`, `contrAgentId`, `operationCategoryId` и др.), не собирать payload и не делать write-запрос.

#### Проект

- завершённый проект (`closed == true`) — нельзя привязывать новые операции и нельзя отвязывать существующие
- архивный проект — нельзя привязывать новые операции.

#### Контрагент

- если контрагент архивный, к нему нельзя привязать операцию.

#### Счёт

- если счёт архивный, к нему нельзя привязать операцию;
- если счёт связан с архивным ЮЛ, к нему нельзя привязать операцию.

Если есть ограничение, агент не должен отправлять write-запрос и должен сообщить пользователю, какая сущность блокирует действие.

## Запрет на молчаливый fallback к локальной фильтрации

Правило — в `README.md` MUST, раздел «Write-операции и ошибки фильтрации». Алгоритм при ошибке сервера:

1. Прочитать сообщение об ошибке (`errorMessage`, `errors`).
2. Диагностировать причину: неверный формат параметра, неправильное имя поля, недопустимое значение.
3. Исправить параметр и повторить **server-side** запрос.
4. Только если server-side фильтрация принципиально невозможна для данного поля — явно сообщить пользователю и предложить локальную фильтрацию как осознанный выбор.

**Специальный кейс `POST /api/v1/operations/list`:** если этот endpoint вернул ошибку, диагностировать параметры запроса (формат дат, имена полей, недопустимые значения). Не переключаться молча на `GET /api/v1/operations` — это другой endpoint с другим поведением, а не fallback того же поиска.

В official `apidoc` ReDoc часть полей `OperationListRequest` может отображаться с префиксом `filter.`. В request sample этого же метода и в рабочих payload передавать их нужно как **top-level keys** на верхнем уровне JSON body, без обёртки `filter`:

- `filter.searchString` в `apidoc` → `"searchString"` в JSON body
- `filter.lastModificationDateStart` в `apidoc` → `"lastModificationDateStart"` в JSON body
- `filter.lastModificationDateEnd` в `apidoc` → `"lastModificationDateEnd"` в JSON body

Каноническое правило для локальной документации: в prose можно упоминать `apidoc`-лейбл с `filter.`, но в JSON-примерах и payload использовать форму без префикса.

- ✅ `"operationDateStart": "2026-01-01"`
- ✅ `"operationDateEnd": "2026-12-31"`
- ✅ `"searchString": "Капитал"`
- ✅ `"lastModificationDateStart": "2026-04-01"`
- ❌ `"filter": { "operationDateStart": "2026-01-01" }`
- ❌ `"filter.operationDateStart": "2026-01-01"`
- ❌ `"filter.searchString": "Капитал"`

После ответа обязательно сделать post-check фильтрации:

- проверить `isSuccess`;
- проверить `min/max operationDate` относительно запрошенного периода;
- если в ответе есть даты вне периода — считать, что фильтр не применился или payload собран неверно; не считать такой ответ корректной серверной выборкой.

## Known caveats

⚠️ **`/operations` — capped-ответ:** если `items.length == 10000` при `total == 0` — это не значит, что операций нет. Это значит, что ответ обрезан лимитом. Не объявлять об отсутствии данных на основании пустого `total`. Предложить пользователю сузить период.

- после одного full/capped ответа не перебирать в той же сессии broad-фильтры `searchString -> контрагент -> сумма` без нового более узкого периода;
- в accrual-аналитике не подменять `operationParts[].calculationDate` датой оплаты без очень явного документированного основания;
- `/operations/summary` может не совпасть с UI, потому что считает сырые записи.

### Runtime caveat: формат дат

Даты передавать в `YYYY-MM-DD` — несмотря на apidoc. Подробнее → `QUICKSTART.md`, раздел «Формат дат».

- Если ошибка `invalid` по любому полю даты — **первый шаг**: переключиться на `date-only` формат, а не менять endpoint и не грузить полный dump.

### Runtime caveat: ID-фильтры в `POST /api/v1/operations/list`

Поля `contrAgentId`, `projectId`, `accountId`, `accountCompanyId` в contract `OperationListRequest` — это массивы целых чисел. Даже один ID передавать как массив:

- ✅ `"contrAgentId": [11318626]`
- ❌ `"contrAgentId": 11318626`

Если передать одно число вместо массива, сервер может вернуть ошибку конвертации типа.
