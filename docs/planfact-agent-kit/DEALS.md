# ПланФакт API — DEALS для AI-агента

Этот файл покрывает сделки и их связь с операциями.

`README.md` — policy, router, budgets, cache.  
`QUICKSTART.md` — короткий technical index.  
`USE_CASES.md` — бизнес-семантика сделок, долга и `method + scope`.  
`OPERATIONS.md` — список операций и bind target.  
`LOOKUPS.md` — как добывать `contractorId`.

> **Naming note:** API сделок использует `contractorId` для контрагента. В операциях, отчётах и справочниках тот же контрагент передаётся как `contrAgentId`. Это одна и та же сущность, но разные имена параметров в разных endpoint'ах.

## Когда открывать

- нужны сделки продажи или закупки;
- нужен поиск сделки по контрагенту, тексту, периоду или сумме;
- нужно создать, изменить или удалить сделку;
- нужно привязать или отвязать операцию к сделке;
- нужна **прибыль по сделке** или **рентабельность сделки**;
- нужно понять, как использовать сделку в аналитике, не подменяя ей долг.

## Доступ к сделкам и неполный результат

- Если deals endpoint возвращает `403` / access-error, считать, что текущий сценарий работы со сделками недоступен по правам, тарифу или доступу к конкретной сделке.
- Не трактовать такой ответ как отсутствие сделок.
- Если аналитика по сделке построена только по доступной части связанных операций, агент должен явно предупреждать пользователя о неполноте результата.

- `GET /api/v1/deals/purchase`
- `POST /api/v1/deals/list`
- `POST /api/v1/deals/calculated`
- `POST /api/v1/deals/selling/list`
- `POST /api/v1/deals/purchase/list`
- `GET /api/v1/deals/summary/selling`
- `POST /api/v1/deals/summary/selling`
- `GET /api/v1/deals/summary/purchase`
- `POST /api/v1/deals/summary/purchase`
- `GET /api/v1/deals/{dealId}`
- `GET /api/v1/deals/selling/{dealId}`
- `GET /api/v1/deals/purchase/{dealId}`

### Write

- `POST /api/v1/deals/selling`
- `POST /api/v1/deals/purchase`
- `PUT /api/v1/deals/bind/{dealId}`
- `PUT /api/v1/deals/unbind/{dealId}`
- `PUT /api/v1/deals/settings/{dealId}`
- update / delete endpoint'ы смотреть в official `apidoc` только для фактического write-сценария или подготовки write-payload

## Списки и сводки сделок

Для списков, сводок, прибыли, оплаты, отгрузки / поставки и рентабельности сделок сначала использовать готовые deal-endpoint'ы, а не реконструировать показатели через `/operations/list`, если готовый endpoint покрывает запрос.

| Endpoint                                  | Когда использовать                                                  |
| ----------------------------------------- | ------------------------------------------------------------------- |
| `POST /api/v1/deals/list`                 | легкий список сделок, выбор `dealId`, список с количеством по типам |
| `POST /api/v1/deals/calculated`           | сделки продаж и закупок с расчетами в одном ответе                  |
| `POST /api/v1/deals/selling/list`         | сделки продаж с расчетами                                           |
| `POST /api/v1/deals/purchase/list`        | сделки закупок с расчетами                                          |
| `GET/POST /api/v1/deals/summary/selling`  | сводка по сделкам продаж: количество, сумма, прибыль                |
| `GET/POST /api/v1/deals/summary/purchase` | сводка по сделкам закупок: количество и сумма                       |

Минимальные read-only маршруты:

- только список / выбор `dealId` → `POST /api/v1/deals/list`;
- продажи с расчётами → `POST /api/v1/deals/selling/list`;
- закупки с расчётами → `POST /api/v1/deals/purchase/list`;
- продажи и закупки вместе с расчётами → `POST /api/v1/deals/calculated`;
- только сводка без расшифровки → `GET/POST /api/v1/deals/summary/selling` или `GET/POST /api/v1/deals/summary/purchase`.

Для этих read-only маршрутов не открывать `apidoc`, если хватает перечисленных ниже фильтров и не нужен редкий параметр.

Правила выбора:

- если пользователь просит только продажи — использовать `selling/list` или `summary/selling`;
- если пользователь просит только закупки — использовать `purchase/list` или `summary/purchase`;
- `deals/calculated` использовать, когда нужны оба типа сделок сразу;
- `deals/list` не использовать как источник прибыли, оплаты, отгрузки / поставки или рентабельности.

Расчетные списки возвращают готовые показатели сделки: сумму сделки, оплачено / выплачено, отгружено / поставлено, доход, расход, прибыль и рентабельность. Но набор бизнес-метрик зависит от типа сделки:

- для **продаж** (`selling`) `income`, `outcome`, `profit`, `profitability` — рабочие P&L-метрики сделки;
- для **закупок** (`purchase`) основными метриками являются `dealValue`, `paidValue`, `paidPercent`, `providedValue`, `providedPercent`; `income`, `outcome`, `profit`, `profitability` не являются бизнес-метриками закупки и могут быть `0` / `null` даже при корректных позициях, выплатах и поставках.

Прибыль и рентабельность — бизнес-метрики сделок продаж; для закупок не считать их основной метрикой и не диагностировать нулевые `income/outcome/profit` как отсутствие связей.

`AccountingMethod` в сделках: `Cash` — кассовый метод, `Calculation` — метод начисления. Для расчетных списков backend default — `Calculation`; если пользователь просит прибыль / рентабельность сделок продаж без метода, следовать правилу ниже: уточнить метод, но упомянуть default ПланФакта.

`PUT /api/v1/deals/settings/{dealId}` меняет настройку метода учета прибыли сделки и возвращает обновленную сделку с расчетами. Это write-операция: перед вызовом показать `dealId`, новый метод учета и дождаться подтверждения. Не использовать для read-only аналитики.

## Ограничения на write-операции со сделками

Перед любым write-запросом с привязкой к сделке проверить:

| Статус сделки      | Привязать новую операцию | Отвязать существующую | Изменить сделку |
| ------------------ | ------------------------ | --------------------- | --------------- |
| Активная           | ✅                        | ✅                     | ✅               |
| Закрытая / архивная | ❌                        | ❌                     | ❌ (только чтение) |

- Если сделка закрыта или архивна — не отправлять write-запрос (bind / unbind / update). Сообщить пользователю конкретный статус сделки.
- Аналогично проектам: `closed == true` у сделки — это блокер для write. Не искать обходные пути.
- Перед `PUT /api/v1/deals/bind/{dealId}` или `PUT /api/v1/deals/unbind/{dealId}` проверить статус сделки через список или карточку.

## Filter-first для списка сделок

В первую очередь использовать server-side фильтры:

- `filter.searchString`
- `filter.contractorId`
- `filter.entityStatusId`
- `filter.startDateStart`
- `filter.startDateEnd`

## Терминология

|                     | Продажа (`selling`)                       | Закупка (`purchase`) |
| ------------------- | ----------------------------------------- | -------------------- |
| Контрагент          | Клиент                                    | Поставщик            |
| Денежная операция   | Поступление                               | Выплата              |
| Неденежная операция | Отгрузка                                  | Поставка             |
| Расходы в сделке    | Выплаты / Начисления по расходным статьям | —                    |

К сделке продажи можно привязать не только поступления и отгрузки, но и **расходные операции** (Выплаты, Начисления по расходным статьям). Это не ошибка — они учитываются в расчёте прибыли сделки.

### Расчётные поля продаж и закупок

Для сделки продажи:

- `paidValue` — подтверждённые поступления (`Income`, `isCommitted == true`);
- `providedValue` — подтверждённые отгрузки (`Shipment`, `isCalculationCommitted == true`);
- `income` в методе `Cash` — подтверждённые поступления по доходным статьям;
- `outcome` в методе `Cash` — подтверждённые выплаты / расходные начисления по расходным статьям;
- `income` в методе `Calculation` — подтверждённые отгрузки;
- `outcome` в методе `Calculation` — подтверждённые выплаты / начисления по расходным статьям;
- `profit = income - outcome`;
- `profitability = profit / income * 100`, если `income != 0`, иначе `null`.

Для сделки закупки:

- `paidValue` — подтверждённые выплаты (`Outcome`, `isCommitted == true`);
- `providedValue` — подтверждённые поставки (`Supply`, `isCalculationCommitted == true`);
- `dealValue` — максимум из суммы позиций, суммы выплат и суммы поставок в валюте аккаунта;
- `income`, `outcome`, `profit` серверно считаются как `0`;
- `profitability` серверно возвращается `null`, потому что `income == 0`.

Если у purchase deal `dealValue`, `paidValue` и `providedValue` заполнены, а `income/outcome/profit` равны `0`, это штатная бизнес-семантика расчётного списка, а не доказательство, что операции или позиции не привязаны.

`financialSummary` в расчётном списке сделок не является каноническим источником полноты закупки. Для purchase deal позиции читать из `dealProducts[]`, оплату — из `paidValue`, поставку — из `providedValue`; нулевые `financialSummary.dealProducts`, `financialSummary.outcomes` или `financialSummary.supplies` сами по себе не означают, что сделка собрана неправильно.

## Привязка отгрузок и поставок к статьям

Отгрузку можно привязать к статьям **Доходов, Активов, Обязательств**. Поставку — к статьям **Расходов, Активов, Обязательств**. Но не все статьи поддерживают начисление:

| Категория     | Статья                                     | Начисление применимо |
| ------------- | ------------------------------------------ | -------------------- |
| Доходы        | любая                                      | ✅                    |
| Расходы       | любая                                      | ✅                    |
| Активы        | Запасы и подстатьи                         | ✅                    |
| Активы        | Другие оборотные и подстатьи               | ✅                    |
| Активы        | Основные средства и подстатьи              | ✅                    |
| Активы        | Нематериальные активы и подстатьи          | ✅                    |
| Активы        | Другие внеоборотные (кроме Нематериальных) | ❌                    |
| Обязательства | Другие краткосрочные и подстатьи           | ✅                    |
| Обязательства | Платежи третьим лицам и подстатьи          | ✅                    |
| Обязательства | Долгосрочные обязательства и подстатьи     | ❌                    |

Если статья не поддерживает начисление — к ней **нельзя привязать** отгрузку или поставку. Не отправлять write-запрос; сообщить пользователю об ограничении.

**Где учитываются по типу статьи:**

- По статьям **Доходов / Расходов** → в Прибыли (методом начисления), в КА, в Балансе.
- По статьям **Активов / Обязательств** → в КА (дебиторка, кредиторка, оборот) и в Балансе, но **не в Прибыли**.

## Прибыль сделки

Прибыль есть **только в сделке продажи**, не в закупке. Метод по умолчанию — **начисление**.

### Кассовый метод

- **Доходы** = ∑ поступлений, привязанных к сделке, с подтверждённой датой оплаты, по **Доходным** статьям.
- **Расходы** = ∑ выплат и начислений, привязанных к сделке как расходы, с подтверждённой датой оплаты, по **Расходным** статьям.
- **Прибыль** = Доходы − Расходы.

### Метод начисления

- **Доходы** = ∑ отгрузок, привязанных к сделке, с подтверждённой датой отгрузки.
- **Расходы** = ∑ выплат и начислений, привязанных к сделке как расходы, с подтверждённой датой начисления, по **Расходным** статьям.
- **Прибыль** = Доходы − Расходы.

### Суммы в карточке сделки

- Сумма поступлений/выплат = все привязанные с подтверждённой датой оплаты (в валюте аккаунта).
- Сумма отгрузок/поставок = все привязанные с подтверждённой датой отгрузки/поставки (в валюте аккаунта).
- Все суммы пересчитываются в **валюту аккаунта**.

### Правило для агента

Если пользователь просит прибыль по сделке без указания метода — уточнить метод, но упомянуть, что по умолчанию в ПланФакте используется начисление. Не подставлять кассовый метод молча.

## Как читать сделки

- `selling` и `purchase` не путать;
- для selling deal в аналитике использовать `dealValue`, `paidValue`, `providedValue`, `income`, `outcome`, `profit`, `profitability`;
- для purchase deal использовать `dealValue`, `paidValue`, `providedValue` и проценты оплаты / поставки; не использовать `income`, `outcome`, `profit`, `profitability` как показатели результата закупки.
- `profitability` может быть `null`, это не обязательно ошибка данных.

## Сделки и задолженность

- наличие сделки само по себе не означает наличие долга;
- долг по сделке — производная метрика по связанным операциям и их статусам оплаты / начисления;
- если часть операции привязана к сделке, но у неё отсутствует `calculationDate`, такую часть нельзя трактовать как факт поставки / отгрузки и нельзя включать в кредиторскую задолженность по стороне начисления;
- если `calculationDate` в части автозаполнена датой оплаты, но у операции `calculationPeriodDate == null` и начисление не подтверждено, для расчёта долга считать, что явная дата начисления по сделке отсутствует;
- для общей дебиторки / кредиторки по КА сначала идти в `LOOKUPS.md` (`GET /contragents/calculated/{id}` с `calculationMethod = Calculation`; при необходимости `{id}/additional`); если нужен срез именно по сделкам или нестандартный фильтр — в `USE_CASES.md` и `OPERATIONS.md`;
- сделки использовать как группировку, enrichment или фильтр.

## Типовые write-сценарии

### Создать сделку продажи или закупки

- endpoint'ы: `POST /api/v1/deals/selling`, `POST /api/v1/deals/purchase`
- до запроса нужны: `title`, `contractorId`, дата
- не путать продажу и закупку

### Привязать или отвязать операцию к сделке

- endpoint'ы: `PUT /api/v1/deals/bind/{dealId}`, `PUT /api/v1/deals/unbind/{dealId}`
- нужны оба ID: `dealId` и `operationId`
- не угадывать операцию; сначала получить и показать целевую связку

## Known caveats

- сделка не равна долгу;
- `profitability == null` допустимо;
- для текстового поиска сделки сначала пробовать server-side фильтры, а не full list;
- НДС влияет только на отображение колонки товаров в сделке/отгрузке; **не влияет на суммы** по сделкам; не пытаться вычленять НДС из сумм;
- выставляемые счета в сделках продаж — для документооборота с клиентом; **не связаны с операциями**;
- прибыль есть только в сделке продажи, не в закупке;
- для purchase deal нулевой `financialSummary.*` не равен отсутствию связей; проверять `dealProducts[]`, `paidValue`, `providedValue` и связанные операции.
