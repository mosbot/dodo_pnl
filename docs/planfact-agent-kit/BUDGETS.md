# ПланФакт API — BUDGETS для AI-агента

Этот файл покрывает budget entities, lifecycle бюджетов и budget settings.

`README.md` — policy, router, budgets, cache.
`QUICKSTART.md` — короткий technical index.
`USE_CASES.md` — бизнес-семантика и `method + scope`.
`LOOKUPS.md` — если для бюджета сначала нужен `projectId`.

## Когда открывать

- пользователь просит список бюджетов или бюджет по названию / проекту / методу;
- нужен бюджет по `budgetId`;
- нужно создать, изменить, закрыть или удалить бюджет;
- нужно читать или менять настройки бюджета;
- нужен plan/fact сценарий, где plan берётся именно из budget entity;
- нужен `БДДС` или `БДР`.

## Три критичных правила

- Для budget plan/fact факт брать из `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories`, а не из `POST /api/v2/reports/opu`.
- `bizinfos.planValue` не равен плану сохранённого бюджета из `GET /api/v1/budgets/{budgetId}`.
- Не вызывать OPU для контрольной проверки budget fact без отдельной причины: это другой слой данных и лишний запрос.

## Независимость БДР и БДДС

БДР и БДДС — разные budget flows. Не использовать бюджет одного типа как источник структуры, настроек, валюты или payload-примера для бюджета другого типа.

- `Bdr` может использовать как бюджет-источник только другой `Bdr`;
- `Bdds` может использовать как бюджет-источник только другой `Bdds`;
- если бюджета-источника того же типа нет, создавать бюджет с нуля по правилам целевого типа.

## Ограничения доступа и штатный budget-сценарий

- Если budget endpoint возвращает `403` или другой access-error, считать, что текущий штатный budget-сценарий недоступен по тарифу, правам или scope бюджета.
- Не трактовать такой ответ как отсутствие бюджета.
- Если пользователь просит альтернативный расчет после недоступного budget endpoint, явно помечать его как отдельный расчет, а не как штатный budget report.

## Типы бюджетов

### БДР — Бюджет доходов и расходов

БДР — это **плановый слой** для доходов и расходов, а не сам отчёт ОПУ. Готовый ОПУ строится из фактических операций через `POST /api/v2/reports/opu`; БДР хранит запланированные суммы в бюджетной сущности `GET /api/v1/budgets/{budgetId}`.

Используется для планирования:

- доходов;
- расходов;
- прибыли;
- выполнения плана по доходам и расходам.

**Метод расчёта:** БДР может строиться двумя методами:

- **метод начисления** — данные отражаются по дате начисления;
- **кассовый метод** — данные отражаются по дате оплаты.

**Правило для агента:** если пользователь просит БДР, план/факт по доходам и расходам, бюджет прибыли или отклонение доходов и расходов от бюджета, и **не указал метод** — задать один уточняющий вопрос: считать методом начисления или кассовым методом. Не считать автоматически, что БДР = метод начисления.

---

### БДДС — Бюджет движения денежных средств

Используется для планирования:

- поступлений;
- выплат;
- денежного потока;
- выполнения плана по движению денег.

Каждое поступление или выплата фиксируется в том периоде, когда деньги фактически поступили или были потрачены.

**Правило для агента:**

- БДДС — это **денежный слой**, а не прибыль;
- это не ОПУ и не выбор между `cash` и `accrual`;
- это план/факт по поступлениям и выплатам денежных средств;
- для БДДС не задавать вопрос про `cash / accrual` — БДДС строится по движению денег;
- БДДС ≠ прибыль; не подменять cash-flow P&L-метриками.

### Runtime-структура БДДС

БДДС **нельзя** создавать как БДР с заменой `budgetMethod: Bdr` на `Bdds`.

В `BudgetVersionItemModel` нет отдельного поля `cashFlowType` / `activityType`. По `apidoc` строка бюджета хранит:

- `operationCategoryId`;
- `outcomeOperationCategoryClassification`;
- `entityId`;
- `date`;
- `value` / `valueInUserCurrency`;
- `operationType`.

Поток ДДС в UI определяется через справочник статьи: `OperationCategory.activityType`.

- операционный поток;
- инвестиционный поток;
- финансовый поток.

Правильная логика:

- `operationType` в budget item — это направление движения денег (`Income` / `Outcome`, а не поток ДДС);
- `operationCategoryId` должен ссылаться на статью, у которой в справочнике задан корректный `activityType`;
- статьи с `activityType: Operating` попадают в операционный поток;
- статьи с `activityType: Investment` попадают в инвестиционный поток;
- статьи с `activityType: Finance` попадают в финансовый поток.

Набор строк только по статьям `operationCategoryType: Income / Outcome` может сохраниться, но заполнит только операционный поток. Для БДДС это неполная структура, даже если `POST/PUT` вернул `isSuccess: true`.

### Бизнес-логика заполнения нового БДДС от факта

Если пользователь просит создать БДДС, опираясь на фактические операции:

1. Проверить дубль бюджета через `GET /api/v1/budgets` с `filter.title` и `filter.budgetMethod=Bdds`.
2. Если бюджет с точным названием уже есть — обновлять его, не создавать дубль.
3. Если есть существующий БДДС с подходящим scope — использовать его как бюджет-источник структуры и настроек.
4. Если БДДС-источника нет, новый общий БДДС можно создать без live-примера, потому что runtime mapping строится через `operationCategoryId -> OperationCategory.activityType`, а не через отдельное поле в budget item.
5. Для общего БДДС по всей компании использовать собственный общий scope: `budgetType: Common`, `entityIds = [projectId проекта "Не выбран"]`, `item.entityId = undistributedProjectId`.
6. Получить факт по статьям через `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories` за нужный период.
7. Для БДДС **не** ограничиваться `operationCategoryType: Income / Outcome`; включать все денежные движения, которые пришли в fact-side по статьям: `Income`, `Outcome`, `Assets`, `Liabilities`, `Capital`.
8. Для определения потока нужен справочник статей:
   - если в fact-ответе нет `activityType`, получить статьи через `GET /api/v1/operationcategories` или точечные `GET /api/v1/operationcategories/{operationCategoryId}`;
   - не делать повторный lookup, если категории уже есть в кэше текущей цепочки.
9. Отбрасывать строки, у которых статья не имеет валидного `activityType` (`Operating`, `Investment`, `Finance`), кроме случая явной технической нераспределённой статьи, которую продукт использует для ДДС.
10. Для каждой строки бюджета писать:
    - `operationCategoryId` из факта;
    - `operationType` как направление денег (`Income` для поступлений, `Outcome` для выплат);
    - `outcomeOperationCategoryClassification` из статьи, если есть; иначе `None`;
    - `date` = последний день месяца;
    - `value` / `valueInUserCurrency` = плановая сумма, рассчитанная от `factValue` / `factValueInUserCurrency`.

Для БДДС факт должен соответствовать денежному слою ДДС: Поступления, Выплаты и Перемещения по дате оплаты. Начисления, Отгрузки и Поставки не являются ДДС-фактом.

Параметры fact-запроса для БДДС:

- endpoint вызывать как `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories` с query-параметрами; не использовать `POST` с JSON body;
- `filter.periodStartDate` / `filter.periodEndDate` — период бюджета;
- `filter.isCalculation=false` — денежная дата, не начисление;
- `filter.ignoreMove` не передавать или передавать `false`; `ignoreMove=true` запрещён для БДДС, потому что Перемещения участвуют в ДДС;
- `filter.ignoreAccrual=true`, если endpoint поддерживает параметр, потому что Начисления не являются ДДС.

Направление строки (`operationType`) нельзя выводить из `operationCategoryType`: статьи `Assets`, `Liabilities`, `Capital` могут быть и поступлением, и выплатой в зависимости от операции. Использовать тип денежного движения из fact/detail, если он есть; если в ответе доступна только подписанная сумма, то положительное значение маппить в `Income`, отрицательное — в `Outcome`, а `value` писать как модуль суммы.

Правила для write:

- `first-budget flow` ниже относится только к общему **БДР** без бюджета-источника;
- не применять `first-budget flow` БДР к БДДС;
- не создавать БДДС только из строк `Income / Outcome` по статьям доходов и расходов;
- перед `POST/PUT` проверить, что в payload попали строки минимум по тем `activityType`, которые есть в факте за период;
- если в факте есть `Investment` или `Finance`, а в payload все строки относятся только к `Operating`, остановиться и исправить mapping до write.

## Scope бюджета

Бюджет в ПланФакте может быть построен:

- по всей компании (по умолчанию, если пользователь не указал иное);
- по юрлицу;
- по проекту;
- по группе проектов.

Не предполагать только project-scope без проверки данных и задачи.

## Источники данных: план и факт

### Что считать планом

Если пользователь просит план бюджета, бюджет по статьям, значения бюджета по месяцам или budget detail — брать **план** из:

```
GET /api/v1/budgets/{budgetId}
```

Для детализации плана использовать данные budget detail, включая элементы внутри бюджетной версии. Суммы плана брать из budget detail, **а не из агрегированного fact-запроса**.

### Что считать фактом

Если пользователь просит факт по бюджету, plan/fact сравнение или отклонение факта от плана — брать **факт отдельно** из:

```
GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories
```

Использовать поле `factValue` / `factValueInUserCurrency` как факт по статье.

Это специальный budget-сценарий. Для сравнения с сохранённым бюджетом fact-side берётся из `bizinfos`, потому что так устроена продуктовая логика бюджета. Не заменять этот источник на `POST /api/v2/reports/opu` и не вызывать OPU для контрольной проверки без отдельной причины.

### Критичное разграничение: `planValue` в BizInfos ≠ план бюджета

В ответе `bizinfos/incomeoutcomehistorybyoperationcategories` есть поля `planValue` / `planValueInUserCurrency`. Они **не равны** плану уже сохранённого бюджета из `GET /api/v1/budgets/{budgetId}`.

| Поле                                    | Когда использовать                                                        |
| --------------------------------------- | ------------------------------------------------------------------------- |
| `factValue` / `factValueInUserCurrency` | Всегда — как факт по статье                                               |
| `planValue` / `planValueInUserCurrency` | Только в сценарии заполнения бюджета плановыми суммами на основе операций |

Если пользователь сравнивает факт с уже существующим бюджетом — сторону `plan` брать из `budgets/{budgetId}`, а не из `planValue`.

## Два сценария plan/fact

### Источник факта для budget plan/fact

Для budget plan/fact факт брать из `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories`, даже если `/api/v1/operations` уже загружен в кэш текущей сессии.

Operations-cache не является заменой продуктового budget fact. Локальный расчёт из операций допустим только как отдельная диагностическая реконструкция по явной просьбе пользователя или при ошибке budget fact endpoint; в ответе пометить, что это не стандартный budget fact.

### Сценарий 1: сравнение с существующим бюджетом

Пользователь просит сравнить факт с уже созданным бюджетом.

- `plan` → `GET /api/v1/budgets/{budgetId}`, суммы из budget detail
- `fact` → `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories`, поле `factValue`

**Shortcut для проектных бюджетов:** если пользователь просит план+факт **по проектам целиком** (не по статьям), можно обойтись одним запросом `GET /api/v1/projects/calculations` (см. `LOOKUPS.md`, раздел «Projects → Финансовые агрегаты»): он возвращает `incomePlan / outcomePlan` (план из бюджета проекта) и `incomeFact / outcomeFact` (факт в выбранном методе) сразу. Использовать этот маршрут, когда детализация по статьям не нужна.

### Сценарий 2: заполнение бюджета плановыми суммами на основе операций

Пользователь хочет заполнить бюджет плановыми суммами, взятыми из операций.

- источник обеих сторон — `bizinfos/incomeoutcomehistorybyoperationcategories`
- `factValue` → факт; `planValue` → плановые суммы для заполнения бюджета
- один запрос возвращает оба поля одновременно — второй запрос не нужен

Если пользователь формулирует задачу как «построить бюджет, опираясь на фактические данные» или «спланировать рост от факта», базой для плановых строк считать `factValue` / `factValueInUserCurrency`, а не `planValue`. `planValue` использовать только если пользователь явно просит взять готовые плановые суммы из операций.

### Правило метода: план и факт должны использовать один метод

Не смешивать методы между `plan` и `fact`:

| Тип бюджета    | Метод для plan   | Метод для fact   |
| -------------- | ---------------- | ---------------- |
| БДР кассовый   | логика `cash`    | логика `cash`    |
| БДР начисление | логика `accrual` | логика `accrual` |
| БДДС           | движение денег   | движение денег   |

## Продуктовая логика автозаполнения

- Бюджет можно автоматически заполнить, если в системе есть операции за нужный период, по нужному юрлицу и проекту.
- При автозаполнении могут подставляться и плановые, и фактические данные.
- Если пользователь отменяет автозаполнение, но операции в системе есть — в бюджет подставляются только фактические данные.
- Фактические операции подтягиваются в бюджет автоматически при загрузке в ПланФакт; отдельный импорт факта не требуется.
- Повторный импорт заменяет плановые данные бюджета новыми во **всём** бюджете целиком, а не только в изменённых строках.

## Как выбирать маршрут

### Список бюджетов

```
GET /api/v1/budgets
```

- сначала переносить в фильтры название, `projectId`, `budgetMethod`, `budgetId`;
- не грузить полный список по умолчанию, если пользователь уже дал название, проект или метод;
- если пользователь дал проект по названию, сначала получить `projectId` через `LOOKUPS.md`.

Доступные server-side фильтры: `filter.title`, `filter.budgetId`, `filter.projectId`, `filter.budgetMethod`.

### Бюджет по ID

```
GET /api/v1/budgets/{budgetId}
```

Использовать для получения плана и budget detail.

### Настройки бюджета

```
GET /api/v1/budgets/settings/{budgetId}
PUT /api/v1/budgets/settings/{budgetId}
```

Если запрос именно про настройки — не подменять его общим `PUT /api/v1/budgets/{budgetId}`. Settings и базовая budget entity — разные маршруты.

### Закрытие бюджета

```
POST /api/v1/budgets/close/{budgetId}
```

Lifecycle-действие `close` — отдельный маршрут, не частный случай обычного update.

### Создать или изменить бюджет

```
POST /api/v1/budgets
PUT /api/v1/budgets/{budgetId}
DELETE /api/v1/budgets/{budgetId}
```

Для обычного budget write до подтверждения показать пользователю: проект, бюджетный метод, период и ключевые суммы. Если пользователь явно пишет, что бюджет нужно создать или обновить в аккаунте, это уже подтверждённый budget write: не спрашивать второе подтверждение, но зафиксировать метод, scope и ключевые суммы в ответе после выполнения. Точный payload брать из official `apidoc`.

Runtime-контракт создания бюджета:

- `POST /api/v1/budgets` создаёт только оболочку бюджета. В create-payload передавать только top-level поля: `title`, `description`, `startDate`, `endDate`, `budgetType`, `entityIds`, `agentId`, `currencyCode`, `budgetMethod`.
- В `POST /api/v1/budgets` не передавать `versions`, `budgetStatus`, `agentIds`, `projectIds` и строки бюджета. API может вернуть ошибку вида `Бюджет <guid> не найден`, если попытаться создать бюджет сразу со строками.
- Строки бюджета записывать отдельным `PUT /api/v1/budgets/{budgetId}` после успешного `POST`. В update-payload передавать `title`, `description`, `startDate`, `endDate`, `budgetStatus`, `versions`, `currencyCode`.
- Настройки бюджета писать отдельным `PUT /api/v1/budgets/settings/{budgetId}`. Settings не смешивать с create/update budget entity.

### Fast-path: явный запрос на создание бюджета

Если пользователь явно пишет, что бюджет нужно **создать в аккаунте** и ожидает готовый результат, агент должен трактовать это как уже подтверждённый budget write и идти по короткому маршруту, а не по длинной диагностике.

Разрешён максимум `1` уточняющий вопрос:

- только если не указан метод `cash / accrual` для БДР;
- только если без вопроса нельзя однозначно выбрать целевой scope, метод или структуру бюджета.

Минимальный маршрут зависит от того, что именно делает пользователь: обновляет существующий бюджет или создаёт новый бюджет. Существующий бюджет можно использовать как источник структуры, но не как источник факта.

Если `budgetId` уже известен из кэша или предыдущего ответа — поиск бюджета по названию пропустить.

**A. Обновить существующий бюджет**

1. Если `budgetId` неизвестен: `GET /api/v1/budgets` с `filter.title` + `filter.budgetMethod` целевого типа бюджета (`Bdr` или `Bdds`).
2. `GET /api/v1/budgets/{budgetId}` для структуры, контекста и текущих строк бюджета.
3. Получить факт через `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories`, только если факт нужен для расчёта новых строк.
4. `GET /api/v1/budgets/settings/{budgetId}` только если настройки не загружены в кэш или их нужно менять.
5. `PUT /api/v1/budgets/{budgetId}`.
6. `PUT /api/v1/budgets/settings/{budgetId}` только если настройки действительно изменились или без их записи UI-сценарий не будет корректным.

**B. Создать новый бюджет**

Существующий бюджет можно использовать как **бюджет-источник структуры и настроек** только если у него тот же `budgetMethod`. Он не является источником фактических доходов или расходов.

1. `GET /api/v1/budgets` с `filter.title` + `filter.budgetMethod` целевого типа бюджета (`Bdr` или `Bdds`), чтобы проверить точное совпадение имени.
2. Если бюджет с точным названием уже есть — перейти к маршруту A, не создавать дубль.
3. Найти существующий бюджет того же `budgetMethod` с подходящим scope. Бюджеты другого `budgetMethod` игнорировать.
4. Если источника того же типа нет: для общего `Bdr` использовать first-budget flow БДР; для общего `Bdds` использовать flow БДДС от факта (`Common`, проект `Не выбран`, потоки через `operationCategory.activityType`).
5. Если пользователь ожидает, что в бюджете будут видны фактические доходы/расходы или денежные поступления/выплаты, получить факт через `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories` с тем же периодом, методом и scope после выбора корректной структуры строк для целевого типа бюджета.
6. `POST /api/v1/budgets` только с оболочкой бюджета: `title`, `description`, `startDate`, `endDate`, `budgetType`, `entityIds`, `agentId`, `currencyCode`, `budgetMethod`.
7. `PUT /api/v1/budgets/{newBudgetId}` обязательно, если бюджет создаётся со строками плана: все `versions[].info.items` писать только этим запросом.
8. `PUT /api/v1/budgets/settings/{newBudgetId}` обязательно, если нужно перенести настройки или включить видимость факта (`isShowFact = true`, `groupByType = Month`).

### First-budget flow: общий БДР без бюджета-источника

Этот flow применяется **только** к `budgetMethod: Bdr`. Для `budgetMethod: Bdds` он запрещён.

Если `GET /api/v1/budgets?filter.budgetMethod=Bdr` вернул пустой список, а пользователь просит создать общий БДР по всей компании:

1. Найти проект **«Не выбран»** через `GET /api/v1/projects` (`title = "Не выбран"` или ближайший undistributed/system project из ответа).
2. Для общего БДР использовать `budgetType: Common` и `entityIds: [undistributedProjectId]`.
3. Создавать одну версию бюджета: `budgetDirectionType: Up`.
4. Внутри этой версии размещать и доходные, и расходные строки; различать их по `operationType: Income / Outcome`.
5. Не создавать отдельную версию `Down` для расходов.
6. Для строк общего БДР использовать `item.entityId = undistributedProjectId`.
7. `groupByType` для помесячного бюджета должен быть `Month`; если нужен факт в UI — `isShowFact = true`.
8. Для БДР методом начисления в settings должен быть `isCalculation = true`; для кассового метода — `false`.

Минимальные проверки в first-budget flow:

- `GET /api/v1/budgets` — проверить дубль и отсутствие БДР-источника;
- `GET /api/v1/projects` — найти `undistributedProjectId`;
- `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories` — получить факт/базу для плана, если бюджет строится от операций;
- после write выполнить только контрольные `GET /api/v1/budgets/{budgetId}` и `GET /api/v1/budgets/settings/{budgetId}`. Не вызывать ОПУ, `operations/list`, `projects/calculations` или дополнительные сверки без отдельной причины.

### Reuse lookup-context в budget write

В рамках одного непрерывного budget write не повторять lookup-и, которые уже были выполнены и не инвалидированы:

- если `GET /api/v1/budgets` уже подтвердил, что бюджета с точным названием нет, не повторять проверку дубля перед `POST`;
- если ЮЛ уже найдено и активно, не делать повторный `GET /api/v1/companies` перед `POST`;
- если проект / группа проектов уже найдены и активны, не делать повторный `GET /api/v1/projects` или `GET /api/v1/projects/groups` перед `POST`;
- повторить lookup можно только если был write, пользователь сообщил о внешнем изменении, прошла длинная пауза, или API вернул ошибку, указывающую на устаревший ID / статус сущности.

Что не делать в fast-path:

- не выполнять дополнительные `GET` "на всякий случай", если уже найден рабочий бюджет-источник структуры;
- не делать повторную верификацию через несколько одинаковых `GET`, если write завершился успешно и локальный preflight пройден;
- не грузить лишние справочники, если `budgetType`, `entityIds` и валюта уже взяты из бюджета-источника структуры;
- не читать и не писать `settings`, если они уже есть в кэше и не меняются;
- не использовать слово «шаблон» как инструкцию к действию: бюджет-источник даёт только структуру и настройки, но не факт и не плановые суммы;
- не повторять preflight lookup-и перед write, если они уже выполнены в этой же цепочке budget write и не были инвалидированы;
- не спрашивать второе подтверждение после явного пользовательского запроса на создание бюджета.

### Write preflight для бюджета

Перед первым budget write агент должен локально проверить payload:

- даты строк бюджета должны соответствовать реальной месячной сетке бюджета;
- плановые строки бюджета писать на конец месяца, а не на первое число;
- для `POST/PUT /api/v1/budgets` top-level `endDate` передавать как первый день последнего месяца периода; после `GET` API может вернуть конец периода как последний день месяца;
- для `POST /api/v1/budgets` preflight должен отклонять payload с `versions`, `budgetStatus`, `agentIds`, `projectIds` или строками бюджета; create-запрос создаёт только оболочку;
- если новый бюджет должен содержать плановые строки, preflight должен требовать последующий `PUT /api/v1/budgets/{newBudgetId}`;
- для `ByProjects` у каждой строки должен быть `entityId`;
- для `Common` не считать, что `entityIds` пустые, а `item.entityId = 0`: если есть рабочий бюджет-источник, эти поля наследовать из него; если источника нет — сверить payload с official `apidoc` и явно показать пользователю, что структура создана без live-примера;
- наследовать структуру, настройки и валюту можно только из бюджета того же `budgetMethod`;
- для `Bdr` без БДР-источника использовать first-budget flow БДР (`Common`, `entityIds = [undistributedProjectId]`, одна версия `Up` с `Income` и `Outcome` вместе);
- для `Bdds` без БДДС-источника строки строить с нуля через `operationCategory.activityType`;
- существующий бюджет-источник отвечает только за структуру и настройки; фактические суммы для плановых строк брать из `bizinfos/incomeoutcomehistorybyoperationcategories` только после выбора корректной структуры целевого бюджета;
- если пользователь ожидает видеть факт в UI, до записи settings проверить, что `isShowFact = true` и `groupByType = Month`;
- для БДР методом начисления в settings должен быть `isCalculation = true`; для кассового метода — `false`;
- после `POST / PUT / PUT settings` считать бюджет созданным корректно только после контрольного `GET /api/v1/budgets/{budgetId}` и, если писались настройки, `GET /api/v1/budgets/settings/{budgetId}`; проверять структуру и настройки, не делать дополнительные аналитические сверки;
- для БДДС контрольный `GET /api/v1/budgets/{budgetId}` должен подтверждать не только количество строк и суммы, но и сохранение строк по нужным потокам ДДС; если все строки оказались только в операционном потоке при наличии инвестиционного / финансового факта, считать write некорректным.

## Known caveats

- `planValue` в `bizinfos` — не план сохранённого бюджета; использовать только в сценарии заполнения бюджета суммами из операций.
- Суммы плана брать из budget detail, а не из агрегированного fact-запроса.
- Повторный импорт заменяет плановые данные во **всём** бюджете, а не только в изменённых строках.
- `budgetMethod` не равен business-методу `cash / accrual` из пользовательского вопроса.
- `settings` и базовая budget entity — разные маршруты, не смешивать.
- `POST /api/v1/budgets` принимает только оболочку бюджета и может вернуть созданный бюджет с пустыми `versions[].info.items`. Это нормально: строки бюджета должны записываться следующим `PUT /api/v1/budgets/{budgetId}`.
- Ошибка `Бюджет <guid> не найден` после `POST /api/v1/budgets` с `versions` обычно означает неверный create-payload, а не отсутствие уже созданного бюджета.
- `close` — отдельное lifecycle-действие, не обычный update.
- `БДДС` и `БДР` не означают, что весь ответ можно собрать только из budget endpoints.
- Budgets хранят плановый слой; фактические данные подтягиваются отдельной агрегацией по операциям.
- Бизнес-логика Платёжного календаря — в `REPORTS.md`, а не в этом файле.
- Для budget write swagger-поля `startDate` / `endDate` с типом `date-time` недостаточны как спецификация: если payload собран не из live-бюджета или уже успешного write-примера, сервер может вернуть `The field EndDate is invalid`.
- В first-budget flow для общего БДР структура из `apidoc` недостаточна: runtime-структура должна быть `budgetType: Common`, `entityIds: [projectId проекта "Не выбран"]`, одна версия `budgetDirectionType: Up`, внутри одной версии строки и `Income`, и `Outcome`. Отдельная версия `Down` для расходов не нужна и может привести к сохранению только расходной части.
- First-budget flow для общего БДР не является шаблоном для БДДС. Для нового общего БДДС без БДДС-источника использовать отдельную БДДС-логику: `operationCategoryId` из факта + mapping потока через `operationCategory.activityType`.
