## Базовая модель

### Два метода расчёта прибыли

| Метод          | Когда операция попадает в расчёт                        | Основная дата     | Флаг подтверждения       |
| -------------- | ------------------------------------------------------- | ----------------- | ------------------------ |
| **Кассовый**   | Когда деньги фактически поступили или потрачены         | `operationDate`   | `isCommitted`            |
| **Начисление** | Когда зафиксирован факт: услуга оказана, товар отгружен | `calculationDate` | `isCalculationCommitted` |

Одна и та же операция может иметь обе даты и попасть в **разные периоды** в зависимости от метода. Это не ошибка данных.

**Правило:** если метод не указан, а от него зависит итоговая цифра — задать один уточняющий вопрос. Не угадывать. Не подставлять кассовый метод как дефолт: пользователь может вести учёт методом начисления, и кассовые цифры будут ошибочными для его бизнес-контекста.

### Две даты в операции

| Поле              | Смысл                                       | Где брать для аналитики            |
| ----------------- | ------------------------------------------- | ---------------------------------- |
| `operationDate`   | Дата реального движения денег (оплата)      | Верхний уровень операции           |
| `calculationDate` | Дата признания факта (начисление, отгрузка) | `operationParts[].calculationDate` |

Для accrual-аналитики всегда смотреть `operationParts[].calculationDate`, а не верхний уровень — верхний уровень может содержать агрегированную или некорректную дату.
Особый случай — Начисление (`Accrual`): `operationDate` всегда равен `calculationDate`. Это не баг.

### Три главных отчёта

| Отчёт      | Что показывает                                              | Выбор метода?                | Ключевая дата         |
| ---------- | ----------------------------------------------------------- | ---------------------------- | --------------------- |
| **ОПУ**    | Доходы, расходы, прибыль                                    | Да — кассовый или начисление | По методу             |
| **ДДС**    | Реальное движение денег за период                           | Нет                          | `operationDate`       |
| **Баланс** | Состояние бизнеса на дату: Активы / Обязательства / Капитал | Нет                          | as-of date (snapshot) |

- **ДДС ≠ ОПУ кассовым методом.** ДДС строится по денежным операциям (Поступления, Выплаты, Перемещения). ОПУ — только по статьям Доходов и Расходов, без Перемещений.
- **`cash` ≠ ДДС.** «cash» в контексте ОПУ — способ расчёта прибыли по дате оплаты, не отчёт о движении денег.
- **Баланс ≠ остаток на счетах.** Баланс — полная структура: Активы, Обязательства, Капитал.

### Денежные и неденежные операции

| Тип                               | Деньги двигаются? | Попадает в ДДС? |
| --------------------------------- | ----------------- | --------------- |
| Поступление, Выплата, Перемещение | ✅                 | ✅               |
| Начисление, Отгрузка, Поставка    | ❌                 | ❌               |

Неденежные не участвуют в ДДС и БДДС никогда. Подробнее → `OPERATIONS.md`.

### Факт, план, просрочка

| Понятие       | Определение                                 | Ключевой признак                                                                                                                                          |
| ------------- | ------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Факт**      | Операция с подтверждённой датой             | `isCommitted == true` (cash) или `isCalculationCommitted == true` (accrual)                                                                               |
| **План**      | Операция с любым статусом подтверждения     | Включает неподтверждённые и будущие                                                                                                                       |
| **Просрочка** | Неподтверждённая операция с датой в прошлом | Cash: `operationDate` < as-of date и `isCommitted == false`; Accrual: `operationParts[].calculationDate` < as-of date и `isCalculationCommitted == false` |

### Типы статей (`operationCategoryType`)

| Тип           | Назначение    | В ОПУ?                                            |
| ------------- | ------------- | ------------------------------------------------- |
| `Income`      | Доходы        | ✅ доходная часть                                  |
| `Outcome`     | Расходы       | ✅ расходная часть                                 |
| `Assets`      | Активы        | ❌                                                 |
| `Liabilities` | Обязательства | ❌                                                 |
| `Capital`     | Капитал       | ❌ (кроме Дивидендов для нераспределённой прибыли) |

Для accrual-аналитики смотреть `operationParts[].operationCategoryType`, не верхний `operationType`: части одной операции могут относиться к разным типам. Иерархия статей → `OPERATION_CATEGORIES.md`.

### Суммы и `operationParts`

- При любой агрегации использовать `valueInUserCurrency`, а не `value`
- `value` — сумма в валюте счёта, нельзя суммировать между разными счетами и разными валютами
- `valueInUserCurrency` — приведено к валюте пользователя, суммировать можно
- Для аналитики по частям: `operationParts[].valueInUserCurrency`

Одна операция может быть разбита на части (`operationParts`), различающиеся по статье, проекту, сумме, дате начисления:

- Суммы, статьи, проекты, даты начисления → из `operationParts`
- Дата оплаты, счёт, валюта, тип операции → с верхнего уровня
- Количество операций считать по уникальным `operationId`, не по числу частей

## Когда бизнес-термин меняет цифру

Задавай уточняющий вопрос только если без него реально меняется итоговая метрика.

| Термин пользователя         | Что уточнить                                              |
| --------------------------- | --------------------------------------------------------- |
| `выручка`                   | `кассовый метод` или `метод начисления`                   |
| `прибыль`                   | `кассовый метод` или `метод начисления`                   |
| `аналитика по проектам`     | все проекты или конкретный; активные, завершенные или все |
| `наша компания`             | весь бизнес или конкретная компания, если их несколько    |
| `сколько заплатили Иванову` | какого именно контрагента выбрать                         |

Если стандартная трактовка уже зафиксирована ниже — не задавай вопрос ради перестраховки.

## Словарь пользовательских формулировок

- `поступления` — деньги пришли на наши счета, обычно `Income`
- `выплаты` — деньги ушли с наших счетов, обычно `Outcome`
- `сколько заплатили контрагенту` — считать выплаты этому контрагенту
- `сколько поступило от контрагента` — считать поступления от этого контрагента
- `назначение платежа` — обычно верхнее поле `comment`; для точного списка сначала искать точное совпадение
- `дебиторка` — нам должны
- `кредиторка` — мы должны
- `баланс` / `финансовое положение` — состояние бизнеса на дату (Активы / Обязательства / Капитал), не остаток денег на счетах
- `остаток` не равен `изменению остатка`
- `денежный поток` не равен `остатку денежных средств`
- `cash` — способ расчёта прибыли по дате оплаты, не синоним ДДС
- `выручка` не равна всем `Income`-частям API подряд
- статья в кавычках, например `"Зарплата"` — точное название статьи
- `быстрые фильтры` / `сохраненные фильтры` — персональные сохраненные условия фильтрации; бизнес-логика и методы → `QUICKFILTERS.md`
- `план по проекту` — план из сохранённого бюджета проекта; источник — `projects/calculations` (поля `incomePlan / outcomePlan`); не брать из `bizinfos.planValue`, это суммы из операций, а не бюджет
- `факт по проекту` — `projects/calculations` (`incomeFact / outcomeFact`) в выбранном методе; или `POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body для accrual-ОПУ с разбивкой по статьям
- `дебиторка / кредиторка по проекту` — `projects/{id}/summary` (`invoiceBalance / invoiceSupplyBalance`)

## Критичные правила интерпретации

### 1. Проекты и технические сущности

- `closed == true` — завершённый проект; `closed == false` — не завершён
- `active` не использовать как основной признак завершённости проекта
- для отчёта по проектам «в работе» исключать плановые проекты без факта: `sumOfIncomeFact == 0` и `sumOfOutcomeFact == 0`
- технического контрагента `Не выбран` по умолчанию исключать из пользовательских контрагентских отчётов
- технический проект `Не выбран` — служебный проект для нераспределённых частей операций

**Ограничения на привязку операций к проектам:**

- **Завершённый проект** (`closed == true`) — нельзя привязывать новые операции и нельзя отвязывать существующие
- **Архивный проект** — нельзя привязывать новые операции

Перед любым write-запросом с `projectId` проверить статус проекта. Если завершён или архивен — не отправлять, сообщить пользователю.

**Архивные сущности в аналитике:**

- По умолчанию аналитика **включает** архивные проекты, контрагентов и счета, у которых есть данные в запрашиваемом периоде. Архивный проект мог быть активным в прошлом — исключать его значит потерять исторические данные.
- Исключать архивные только если пользователь явно просит «активные проекты», «текущие контрагенты» и т.п.
- Технический контрагент `Не выбран` и технический проект `Не выбран` — исключать из пользовательских отчётов по умолчанию (это служебные сущности, не архивные).

### 2. Операции и связанные записи

- каждая сырая запись не равна отдельной бизнес-операции
- одна операция может иметь и `operationDate`, и `calculationDate` — это нормально, не дубликат
- `boundMoveOperationId == 0` — самостоятельная операция
- если `boundMoveOperationId > 0` и `account.isUndistributed == true` — это начисление (`Accrual`)
- компанию начисления определять по `account.company.title`
- если есть `boundMoveOperation`: одинаковый счёт у пары → `Accrual`; разные счета → `Move`
- если данных недостаточно — не выдумывать тип, называть «связанной»
- для UI-подобного вывода группировать парные записи по `boundMoveOperationId`
- если сгруппированная пара интерпретирована как `Move`, отображаемая сумма операции = `0`; raw-суммы списания/зачисления использовать только как техническую сумму внутреннего перевода между счетами

### 3. Задолженность и просрочка

**Первый выбор — готовый endpoint по КА.** Если пользователь просит дебиторку / кредиторку / прибыль по контрагенту, идти в `LOOKUPS.md`, раздел «ContrAgents → Финансовые агрегаты», и использовать `GET /api/v1/contragents/calculated/{id}` с нужным `calculationMethod`; для типа долга дополнительно `GET /api/v1/contragents/{id}/additional`. Это источник истины для долгов в стандартном срезе.

Для списка контрагентов сначала получить ограниченный список КА через filter-first lookup, затем вызвать `GET /api/v1/contragents/calculated/{id}` по выбранным КА. Не запускать N+1 по большому списку молча; правила лимита и предупреждения → `LOOKUPS.md`.

Реконструкция из сырых операций (алгоритм ниже) — fallback, когда готовый endpoint не покрывает запрос: нужен нестандартный фильтр результата (например, «дебиторка > 100к»), нестандартный grouping, просрочка с произвольной as-of date, долги по сделкам через `sellingDealId / purchaseDealId`, или `calculated` вернул ошибку. В ответе помечать, что данные посчитаны вручную из `operations/list`.

Задолженность в общем виде — это не отдельная сущность API. Рассчитывается как расхождение между датой оплаты и датой начисления.

| Тип операции | Оплата          | Начисление      | Интерпретация                                              |
| ------------ | --------------- | --------------- | ---------------------------------------------------------- |
| `Income`     | подтверждена    | не подтверждена | неденежная кредиторка (деньги получены, услуга не оказана) |
| `Income`     | не подтверждена | подтверждена    | денежная дебиторка (услуга оказана, деньги не получены)    |
| `Outcome`    | подтверждена    | не подтверждена | неденежная дебиторка (заплатили, услугу не получили)       |
| `Outcome`    | не подтверждена | подтверждена    | денежная кредиторка (услугу получили, не заплатили)        |

- просрочка — неподтверждённая дата уже в прошлом (< as-of date)
- считать задолженность как на текущую дату, так и на произвольную as-of date
- наличие сделки само по себе не означает долг

**Пошаговый алгоритм расчёта задолженности по операциям:**

1. Загрузить операции (`/operations` или кэш) за нужный период или по всей истории.
2. Отфильтровать `Income` и `Outcome` (Перемещения, Отгрузки, Поставки — не участвуют).
3. До сравнения статусов проверить дату начисления в части операции:
   - если операция привязана к сделке через `sellingDealId` или `purchaseDealId`, но у части отсутствует `operationParts[].calculationDate`, такую часть нельзя считать подтверждением факта начисления;
   - runtime caveat: если у привязанной к сделке операции `calculationPeriodDate == null`, `operationParts[].isCalculationCommitted == false`, а `operationParts[].calculationDate` автозаполнена датой оплаты, считать, что явная дата начисления отсутствует;
   - при ручной реконструкции просроченной задолженности по контрагентам такую часть не включать ни в дебиторку, ни в кредиторку; вынести её в блок «Исключено / требует проверки» с причиной;
   - `operationDate` в этом случае не использовать как замену даты начисления.
4. Для остальных операций сравнить `isCommitted` и `operationParts[].isCalculationCommitted`:
   - оплата подтверждена, начисление нет → неденежная задолженность;
   - начисление подтверждено, оплата нет → денежная задолженность.
5. Определить as-of date (текущую или заданную пользователем).
6. Для **просрочки**: из неподтверждённых отобрать те, чья неподтверждённая дата < as-of date.
7. Классифицировать: `Income` + денежная задолженность = **дебиторка**; `Outcome` + денежная задолженность = **кредиторка** (и зеркально для неденежной, по таблице выше).
8. Суммировать `operationParts[].valueInUserCurrency` для каждой категории.

**Задолженность по сделкам продаж:**

- сумма оплат > сумма отгрузок → неденежная кредиторка (получили деньги, не отгрузили)
- сумма оплат < сумма отгрузок → денежная дебиторка (отгрузили, деньги не получили)

**Задолженность по сделкам закупок:**

- сумма выплат > сумма поставок → неденежная дебиторка (заплатили больше, чем получили)
- сумма выплат < сумма поставок → денежная кредиторка (получили товар, не заплатили)

Для расчёта использовать `sellingDealId` / `purchaseDealId` и связанные операции.

Если у связанной со сделкой операции или части операции отсутствует явная дата начисления либо raw `/operations` не позволяет надёжно подтвердить её наличие, при ручной реконструкции просроченной задолженности по контрагентам не включать такую часть ни в дебиторку, ни в кредиторку; показывать её отдельно в блоке «Исключено / требует проверки» с причиной.

### 4. Отчёты по операциям

- отчёт «по операциям» строить по сырым операциям, потом применять клиентскую интерпретацию
- начисления на служебных счетах показывать, не исключать автоматически
- парные записи `Outcome + Income` с одинаковым `boundMoveOperationId` в UI-отчётах сворачивать в одну

#### Список операций и итоговая сумма

- список операций и итоговая сумма — не одно и то же
- если пользователь просит список операций, в список включать все операции, попавшие в фильтр
- для UI-свернутых перемещений (`Move`) показывать сумму `0`, даже если raw API-записи списания и зачисления содержат `value`
- если пользователь просит денежный итог, считать его отдельно по правилам показателя
- для итогов по поступлениям, выплатам и денежному потоку учитывать только `Income` и `Outcome`
- `Shipment` и `Supply` могут входить в список и в количество операций, но не входят автоматически в денежный итог
- если операция разбита на части, для проекта, статьи, сделки и даты начисления брать суммы из `operationParts`, а не общую сумму операции
- в таблице можно показывать общую сумму операции, но в итог по проекту или статье включать только релевантную часть операции
- одинаковые суммы в cash и accrual при разном количестве операций допустимы, если дополнительные типы операций попали только в список, но не в денежный итог

### 5. Пустые данные и нулевые результаты

- Если за запрашиваемый период нет операций или отчёт возвращает все нули — это **не ошибка API**. Сообщить пользователю: «За указанный период данных нет».
- Не повторять запрос автоматически с расширенным периодом — сначала спросить пользователя.
- `/operations`: `items.length == 0` и `total == 0` → данных нет. `items.length == 10000` и `total == 0` → capped ответ, не пустота (→ `OPERATIONS.md`).
- Отчёт (OPU, DDS, Balance) с нулями по всем строкам → данных за период нет; не трактовать как ошибку endpoint'а.
- Любой endpoint: `isSuccess: true` + `data.items` пустой — это «данных нет», а не ошибка и не повод повторить запрос с другими параметрами. Исключение — `/operations` с capped ответом (см. выше).

### 6. Остатки, ДДС, прибыль и выручка

- остатки по счетам — snapshot на явную дату; `remainder` справочника счетов не источник истины
- ДДС — отдельный отчёт по движению денег; не выбор метода cash/accrual
- для ДДС использовать все статьи учёта, влияющие на движение денег, и дату оплаты `operationDate`
- `cashflow` ≠ прибыль; не подменять P&L cash-flow метриками
- `прибыль` считается только по статьям Доходов и Расходов
- для нераспределённой прибыли отдельно учитывать статью Дивидендов; не смешивать с ДДС
- если пользователь просит прибыль без метода — уточнить метод; не угадывать
- если пользователь явно указал метод начисления — не добавлять `cashflow`
- если пользователь явно указал кассовый метод — не подменять `calculationDate`

#### Выручка методом начисления — scope по умолчанию

По умолчанию включать: `1.1 Нераспределённый доход`, `1.2 Выручка от продаж` и все подстатьи.
По умолчанию не включать: `1.3 Прочие доходы` (проценты, курсовые разницы), `Assets`, `Capital`, `Liabilities`.
Полная иерархия статей → `OPERATION_CATEGORIES.md`.

#### Выручка по проектам и company-level summary

- `выручка по проектам` и `выручка компании по ОПУ` не обязаны совпадать по scope
- рейтинг проектов по умолчанию строится только по пользовательским проектам, без строки `Не выбран`
- company-level summary может включать нераспределённую выручку из `Не выбран`, если цель — совпасть с ОПУ
- если показываются одновременно рейтинг проектов и итог компании — явно назвать scope каждого
- сумма строк рейтинга не обязана равняться company-level summary; если расхождение — коротко назвать причину

#### Прибыльность проектов

- если пользователь просит **план + факт** по проектам — идти в `LOOKUPS.md`, `GET /projects/calculations`: это единственный endpoint, который возвращает план из бюджета и факт в одном ответе; ни `bizinfos/incomeoutcomehistorybyprojects`, ни `POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body плана из бюджета не дают;
- если пользователь просит **дебиторку / кредиторку** по одному проекту — идти в `LOOKUPS.md`, `GET /projects/{id}/summary`, не реконструировать из операций;
- для остальных срезов «прибыльность проектов» всегда сначала фиксировать метод: кассовый или начисление;
- для accrual: использовать accrual-слой, считать только части с `operationCategoryType in {Income, Outcome}`;
- для cash-динамики по подпериодам (time-series): использовать cash-историю по проектам, считать только `Income` и `Outcome`;
- policy по `isUndistributed` называть явно: `cash UI-aligned` (включать) или `cash strict project-only` (исключать);
- по умолчанию для cash-отчётов по проектам — UI-aligned вариант (включать `isUndistributed = true`);
- в ответе всегда явно указывать вариант: `cash UI-aligned`, `cash strict project-only` или `accrual`;
- если нет отрицательных значений — не называть нижний рейтинг «убыточные проекты»; использовать «проекты с наименьшим результатом».

## Method + Scope defaults для частых запросов

Технический маршрут и endpoint'ы — в доменных файлах. Здесь только смысл, `method` и `scope`.

| Сценарий                                                | Method default                 | Scope default                      | Оговорка                                                                                                 |
| ------------------------------------------------------- | ------------------------------ | ---------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Операции за период                                      | `raw operations`               | период пользователя                | список записей, не агрегат                                                                               |
| Точечный поиск операций                                 | `raw operations`               | узкий период или recent writes     | `назначение платежа` → `comment`                                                                         |
| Операции по контрагенту                                 | `raw operations`               | период + контрагент                | `Не выбран` исключать                                                                                    |
| Дебиторка / кредиторка по КА                            | `contragents calculated`       | as-of date + КА / список КА        | `calculationMethod = Calculation`; → `LOOKUPS.md`                                                        |
| Дебиторка / кредиторка / просрочка (нестандартный срез) | `debt reconstruction`          | company-wide as-of date            | fallback из сырых операций, если `calculated` не покрывает                                               |
| Операции по проекту                                     | `raw operations` или `accrual` | конкретный проект                  | суммы в `valueInUserCurrency`                                                                            |
| Остатки по счетам                                       | `balance snapshot`             | company-wide или выбранные счета   | остаток ≠ изменение остатка                                                                              |
| История остатка по счёту                                | `cash history`                 | один счёт                          | изменение ≠ абсолютный остаток                                                                           |
| Баланс / финансовое положение                           | `balance snapshot`             | company-wide или выбранные ЮЛ      | snapshot, не движение; → `BALANCE.md`                                                                    |
| Денежный поток / ДДС                                    | движение денег                 | company-wide                       | → `POST /api/v2/reports/dds`; не спрашивать cash/accrual                                                 |
| ОПУ / P&L / прибыль                                     | спросить cash или accrual      | company-wide                       | → `POST /api/v2/reports/opu`; Дивиденды отдельно                                                         |
| Выручка                                                 | спросить если не задано        | company-wide                       | выручка ≠ все Income подряд                                                                              |
| Сумма по одной статье                                   | спросить если не задано        | company-wide, точная статья        | статья в кавычках = точное название                                                                      |
| Помесячная динамика                                     | один метод на весь ряд         | company-wide                       | не смешивать cash и accrual в одном ряду                                                                 |
| Сравнение двух периодов                                 | один метод, один вызов         | company-wide                       | один OPU/DDS/Balance с `isPeriodDetail`, дельту локально                                                 |
| Рост выручки по проектам                                | спросить если не задано        | пользовательские проекты           | рейтинг ≠ company-level summary                                                                          |
| Рентабельность проектов                                 | спросить если не задано        | пользовательские проекты           | называть вариант: UI-aligned или strict                                                                  |
| План + факт по проектам                                 | `projects calculations`        | пользовательские проекты           | план из бюджета; факт в одном из 3 методов; → `LOOKUPS.md`                                               |
| Дебиторка / кредиторка по проекту                       | `projects summary`             | один проект                        | `invoiceBalance / invoiceSupplyBalance`; → `LOOKUPS.md`                                                  |
| Топ контрагентов по выплатам                            | `cash`                         | company-wide                       | `Не выбран` исключать                                                                                    |
| Структура доходов и расходов                            | `cash`                         | company-wide                       | структура за период ≠ accrual P&L                                                                        |
| Аналитика по сделкам                                    | `deal analytics`               | selling или purchase               | сначала готовые deal-endpoint'ы; для selling прибыль / рентабельность, для purchase `dealValue` / `paidValue` / `providedValue`; ручная реконструкция из операций только если endpoint не покрывает срез |
| Счета на оплату                                         | `documents view`               | документы за период или по статусу | счёт на оплату ≠ вся дебиторка                                                                           |

| Дашборд собственника                                    | зависит от карточек            | обычно company-wide                | для каждой карточки явно подписать method                                                                |
| Компании                                                | `lookup`                       | конкретная компания или справочник | справочник, не аналитика                                                                                 |

## Что API не покрывает напрямую

- чистая прибыль и полноценный P&L не восстанавливаются автоматически из `cashflow`
- остатки по проекту или контрагенту считаются как производная метрика из операций
- `Справочники → Контрагенты → учёт по денежному потоку` публичный API не воспроизводит 1:1
- если UI расходится с публичным API — честно писать о расхождении, не подгонять цифры
