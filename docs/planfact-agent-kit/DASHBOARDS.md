# ПланФакт — Раздел «Показатели» для AI-агента

Этот файл описывает бизнес-логику раздела **«Показатели»** (базовый, доступен всем пользователям на любом тарифе) и соответствующие endpoint'ы API.

`README.md` — policy, router.
`DASHBOARDS_PRO.md` — расширенный раздел «Показатели ПРО» (платный модуль).
`REPORTS.md` — `BizInfos` (v1): cash-history по статьям/проектам/счетам, Платёжный календарь.
`REPORTS_V2.md` — ОПУ и ДДС v2, profit-метрики, detail-trends.
`OPERATION_CATEGORIES.md` — иерархия учётных статей, нужна для интерпретации сумм.

## Когда открывать

- пользователь просит дашборд, сводный экран финансовых показателей;
- нужен стандартный набор блоков: прибыль + денежный поток + остатки + структура платежей + топ-клиенты + прибыльность проектов;
- пользователь явно ссылается на раздел «Показатели»;
- пользователь просит «как в ПланФакте», «как на главной», «как на дашборде», но **без** углублённой аналитики (ROE, оборотный капитал, ликвидность, БДДС/БДР, долги по контрагентам — это `DASHBOARDS_PRO.md`).

## Граница раздела

- Это UI-dashboard layer для базового раздела «Показатели», а не общий fallback для любых финансовых запросов.
- Если пользователь просит один блок, вызывать только endpoint этого блока.
- Если пользователь просит полный экран, вызывать только distinct endpoint'ы реально нужных блоков.
- Не добавлять `reports/*`, `operations/*` или `bizinfos/*` сверх списка блока только потому, что результат будет в виде дашборда или HTML.

## Доступ

- Доступен всем пользователям, на любом тарифе.
- Дополнительная проверка модуля не нужна.

## Request scope

Слово `дашборд`, `показатели`, `как в ПланФакте` или `локальная страница` не означает автоматически все endpoint'ы раздела.

- Если пользователь просит один блок — вызвать только endpoint этого блока.
- Если пользователь просит полный экран «Показатели» — вызвать только distinct data sources реально нужных блоков.
- Delivery (`HTML`, график, таблица, browser-ready файл) не добавляет API-вызовов.
- Для каждого выбранного endpoint собрать один dataset и переиспользовать его во всех карточках/графиках, которые он покрывает.

## Фильтры раздела

Общие для всего раздела:

- Период
- Юрлица и счета
- Проекты
- Сделки
- Отображение для графиков: **по неделям, месяцам, кварталам, полугодиям, годам**

## Блоки

Все суммы выводятся в соответствии с выбранными фильтрами.

### 1. Прибыль, <Валюта аккаунта>

- Выводится общая сумма Доходов, Расходов, Чистой прибыли, Рентабельности в % и Дивидендов.
  - Выводится как плановая, так и фактическая сумма.
- График:
  - отображает весь выбранный период с детализацией по выбранному Отображению (неделя / месяц / квартал / полугодие / год);
  - Доходы и Расходы — столбцы;
  - Чистая прибыль и Дивиденды — линии;
    - если значение < 0, на графике оно отображается красным;
  - в прошлых периодах — фактические значения;
  - текущий период состоит из двух периодов: 1-й — факт, 2-й — план;
  - будущие периоды строятся на основе плановых значений.
- Метод построения отчёта: **Метод начисления** или **Кассовый метод** (выбирается пользователем).

### 2. Денежный поток, <Валюта аккаунта>

- Выводится общая сумма Поступлений, Выплат и их разница.
  - Выводится как плановая, так и фактическая сумма.
- График:
  - отображает весь выбранный период с детализацией по выбранному Отображению;
  - Поступления и Выплаты — столбцы;
  - Разница — линия;
    - если разница в периоде < 0, она отображается красным;
  - прошлое — факт; текущий период = факт + план; будущее — план.
- Варианты построения: **Общий, Операционный, Инвестиционный, Финансовый**.
  - Общий — в расчёт попадают все операции Поступлений и Выплат со всеми статьями.
  - Остальные — только операции Поступлений и Выплат со статьями, у которых соответствующий вид деятельности.

### 3. Остатки на счетах, <Валюта аккаунта>

- График с динамикой остатка по всем счетам бизнеса.
- Ось X всегда детализирована по конкретным датам — фильтр «Отображение» на этот график не влияет.
- В легенде:
  - общий остаток по всем счетам;
  - полное название счёта с его юрлицом — по каждому отображаемому счёту.
- Скрытие счёта:
  - если по счёту не было ни одной операции — не выводится;
  - если за весь выбранный период остаток всегда был = 0 — не выводится.
- Цвета:
  - положительный остаток — зелёный;
  - отрицательный — красный.
- **Endpoint:** `POST /api/v1/dashboards/accountbalance` — получение остатков на счетах в базовом разделе «Показатели».

### 4. Структура платежей

- Имеет переключатель: **Доходы и расходы** / **Поступления и выплаты**. От выбора зависят блоки ниже.

**Блок «Доходы / Поступления»:**

- Круговая диаграмма:
  - общая сумма по Доходам / Поступлениям;
  - суммы по дочерним статьям, формирующим итог:
    - для **Доходов** — дочерние статьи категории «Доходы», с возможностью раскрыть каждую на 1 уровень вниз;
      - рассчитывается всегда методом начисления;
      - исключение: если в тариф не включён модуль «Начисление» — кассовым методом;
    - для **Поступлений** (берутся только операции поступлений) — суммы по статьям: Доходы, Оборотные активы, Внеоборотные активы, Краткосрочные обязательства, Долгосрочные обязательства, Вложения учредителей, Другие статьи капитала (с раскрытием на 1 уровень вниз).
- График с периодами: столбцы — сумма по Доходам / Поступлениям, составленная из сумм по дочерним статьям (stacked).

**Блок «Расходы / Выплаты»:**

- Круговая диаграмма:
  - общая сумма по Расходам / Выплатам;
  - суммы по дочерним статьям:
    - для **Расходов** — дочерние статьи категории «Расходы», с раскрытием на 1 уровень;
    - для **Выплат** (только операции выплат) — суммы по статьям: Расходы, Оборотные активы, Внеоборотные активы, Краткосрочные обязательства, Долгосрочные обязательства, Дивиденды, Другие статьи капитала (с раскрытием на 1 уровень).
- График с периодами: столбцы — сумма по Расходам / Выплатам, составленная из сумм по дочерним статьям (stacked).

### 5. Самые доходные клиенты, <Валюта аккаунта>

- График отражает принцип Парето 20/80: меньшая часть клиентов приносит бо́льшую часть дохода.
- Контрагенты, дающие суммарно 80% и более, выделяются синим цветом.
- Если контрагентов много, выводится столько, сколько помещается на экран; остальные сворачиваются в одну колонку «Остальные».
- Поверх столбцов — линия **«Доля доходов с накопительным итогом, %»**.
- Для расчётов берутся суммы по статьям доходов.
- Метод расчёта: **Метод начисления** или **Кассовый метод**.

### 6. Прибыльность проектов, <Валюта аккаунта>

- На графике выводится информация по проектам: Прибыль проекта (факт и план), Рентабельность проекта (факт и план).
- Выводятся все проекты, по которым за выбранный период была какая-то прибыль.
- Прибыль проекта может быть отрицательной — такие столбцы уходят вниз от нулевой отметки.
- Метод расчёта прибыли: **Метод начисления** или **Кассовый метод**.
- Сортировка на графике: **По прибыли** или **По рентабельности**.

## Endpoint'ы раздела

| Блок / задача                                                                   | Endpoint                                                         |
| ------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Остатки на счетах в разделе «Показатели»                                        | `POST /api/v1/dashboards/accountbalance`                         |
| Прибыль, структура доходов/расходов, базовая cash-структура платежей по статьям | `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories` |
| Самые доходные клиенты                                                          | `GET /api/v1/bizinfos/incomeoutcomehistorybycontragents`         |
| Прибыльность проектов в UI-дашборде                                             | `GET /api/v1/bizinfos/incomeoutcomehistorybyprojects`            |

В `bizinfos`:

- `factValueInUserCurrency` — факт;
- `planValueInUserCurrency` — плановые суммы из операций, не план сохранённого бюджета.

Если пользователь просит один блок, вызывать только endpoint этого блока. Для полного экрана «Показатели» делать по одному запросу на каждый реально нужный distinct data source; не вызывать category, project и contragent datasets одновременно, если запрошен только один из соответствующих блоков.

## Формат запроса для остатков

`POST /api/v1/dashboards/accountbalance` — endpoint остатков именно для базового раздела «Показатели».

```json
{
  "currentDate": "2026-04-22",
  "periodStartDate": "2026-04-22",
  "periodEndDate": "2026-04-22",
  "standardPeriod": "Day"
}
```

Payload плоский, без `filter`. По contract обязательны `currentDate`, `periodStartDate`, `periodEndDate`; для сужения по счетам использовать `accountIds`.

Для справочника / раздела **«Мои счета»** используется `GET /api/v1/bizinfos/accountshistory`. Для **«Показателей ПРО»** используется `POST /api/v2/dashboardpro/accountbalance`. Не переносить эти routes в базовый раздел «Показатели».

## Правила для агента

- Если пользователь просит «как в ПланФакте» или «как на дашборде», но без ликвидности / оборотного капитала / БДДС / БДР / долгов — это **«Показатели»**, а не **«Показатели ПРО»**.
- Для остатков в разделе «Показатели» использовать `POST /api/v1/dashboards/accountbalance` (payload-based, `--data-binary @payload.json`).
- Для раздела «Мои счета» использовать `GET /api/v1/bizinfos/accountshistory` (накопить factValue локально).
- Для раздела «Показатели ПРО» использовать `POST /api/v2/dashboardpro/accountbalance`.
- Не открывать `/api/v1/accounts` только ради проверки счетов или общего итога; список счетов и total брать из ответа выбранного endpoint.
- Для сумм денежного потока / прибыли / структуры / топ-клиентов / проектов — `bizinfos/*`; собирать один dataset на каждый нужный endpoint и переиспользовать его для всех блоков, которые покрываются этим же ответом.
- Метод расчёта (кассовый / начисление) задаётся пользователем; если он не указан для блоков «Прибыль» / «Самые доходные клиенты» / «Прибыльность проектов» — задать один уточняющий вопрос до первого API-запроса (см. `README.md`, раздел «Уточняющие вопросы»).
- Для ДДС и «Денежного потока» в смысле детализации по деятельности (Операционный / Инвестиционный / Финансовый) — предпочитать **`POST /api/v2/reports/dds`** (см. `REPORTS_V2.md`), `bizinfos` использовать только когда достаточно суммарного cash-history.
- Все блоки используют одни и те же фильтры: Период / Юрлица и счета / Проекты / Сделки — применять их ко всем запросам раздела одинаково.
- Блок «Прибыльность проектов» **в рамках UI-дашборда** собирается из `bizinfos/*` (как выше), включая план на уровне `planValue`. Если пользователь просит «план+факт по проектам» **вне** UI-дашборда (отдельно от раздела «Показатели»), план из бюджета через `bizinfos.planValue` не получить — идти в `LOOKUPS.md`, раздел «Projects → Финансовые агрегаты», и использовать `GET /api/v1/projects/calculations`. Для дебиторки/кредиторки по проекту вне UI — `GET /api/v1/projects/{id}/summary`.
