# ПланФакт API для AI-агента

> 

Этот каталог — минимальный рабочий набор для агента, который читает данные из ПланФакт API и при необходимости готовит write-запросы по подтверждению пользователя.

## Роли файлов

- `README.md` — единственный канонический policy-файл.
- `AGENTS.md` — компактная стартовая инструкция и правила поведения агента: как читать этот набор, принимать решения, оформлять результат, работать с неопределённостью и подтверждением изменений. Не выбирает endpoint и не переопределяет policy.
- `BUSINESS_CONTEXT.md` — опциональный бизнес-контекст компании пользователя: отрасль, масштаб, сезонность, нормы, риски и важные KPI. Используется только для интерпретации аналитики и выводов; не выбирает endpoint, не подставляет данные и не переопределяет API / локальную документацию.
- `QUICKSTART.md` — короткий technical index.
- `USE_CASES.md` — единый смысловой слой: базовая модель (методы, даты, отчёты, факт/план, задолженность, суммы), бизнес-семантика, интерпретация данных, `method + scope`.
- `OPERATIONS.md` — справочник по 6 типам операций (бизнес-смысл, ключевые даты, нюансы) и все operation-driven сценарии.
- `REPORTS.md` — `BizInfos` (v1): cash-history по статьям, проектам и счетам; **Платёжный календарь** (бизнес-логика, блоки «Доходы и расходы» / «Движение денег»).
- `REPORTS_V2.md` — **ОПУ и ДДС**: `POST /api/v2/reports/opu`, `POST /api/v2/reports/dds`; бизнес-логика, формулы, profit-метрики.
- `BALANCE.md` — **Баланс** (табличная форма): `POST /api/v1/accounting/balance/table`; Активы / Обязательства / Капитал, `isBalanced`, `accountCategoryType`.
- `BUDGETS.md` — бюджеты, `БДДС`, `БДР`.
- `DEALS.md` — сделки, готовые расчетные списки / сводки, прибыль и рентабельность сделок, связка сделок с операциями.
- `LOOKUPS.md` — справочники, `filter-first lookup`, финансовые агрегаты по контрагентам (`GET /contragents/calculated/{id}`, `{id}/additional`) и по проектам (`projects/calculations`, `projects/{id}/summary` — план из бюджета + факт, дебиторка / кредиторка).
- `OPERATION_CATEGORIES.md` — иерархия учётных статей (plan of accounts): тип, денежная/неденежная, семантика выручки и P&L.
- `DOCUMENTS.md` — `invoice-documents` и document endpoints.
- `QUICKFILTERS.md` — быстрые фильтры / сохраненные фильтры: бизнес-логика, публичные методы `QuickFilters`, Операции / ОПУ / ДДС.
- `DASHBOARDS.md` — раздел **«Показатели»** (базовый, доступен всем): бизнес-логика блоков + endpoint'ы `bizinfos/*` и `POST /api/v1/dashboards/accountbalance` для остатков в базовых «Показателях». Для раздела «Мои счета» используется `GET /api/v1/bizinfos/accountshistory`. Для раздела «Показателей ПРО» используется `POST /api/v2/dashboardpro/accountbalance`.
- `DASHBOARDS_PRO.md` — раздел **«Показатели ПРО»** (платный модуль): ROE, ROA, ликвидность, оборотный капитал, дивиденды, долги по контрагентам, БДР / БДДС дашборд; 14 endpoint'ов `/api/v2/dashboardpro/*`.
- `LLM_EXAMPLES.md` — prompt templates и короткие execution patterns. Не нормативный источник.

## Что читать

Агенту не нужно читать все файлы на каждом старте.

1. Начать с `AGENTS.md`. Это компактная точка входа: цель агента, базовый порядок действий, правила результата и случаи, когда нужно открыть `README.md`.

2. `README.md` остаётся каноническим policy/router. Для выбора маршрута открыть только нужные секции:
   - `Intent Router` — таблица маршрутизации, `Канонический порядок выбора`, `Fast Path`;
   - `Канонический Policy → MUST` — если есть write-действие, ограничение доступа, спорный метод, fallback или риск неверного расчёта;
   - `Request Budget` — если есть риск лишних запросов, широкий поиск, capped response или full dump;
   - `Запрещённые паттерны` — если сценарий связан с операциями, отчётами, Балансом, ДДС / ОПУ или fallback-расчётом.

3. `USE_CASES.md` открывать для business-кейсов, где результат зависит от типа операции, даты, метода расчёта или scope:
   - прибыль, выручка, рентабельность, ОПУ;
   - ДДС, Баланс, план/факт;
   - операции со списком и итоговой суммой;
   - задолженность, просрочка;
   - проекты, КА, сделки, если считается бизнес-метрика.

4. `USE_CASES.md` можно не открывать для чисто технических сценариев без расчёта бизнес-метрики: PDF по известному ID, быстрый фильтр по ID, lookup одного справочного ID, настройки или шаблон без финансового расчёта.

5. После выбора строки в `Intent Router` открывать только указанный файл по сущности. Не открывать дополнительные файлы «для проверки».

6. `QUICKSTART.md` использовать только для техники: auth, формат ответа, даты, пагинация, shell-safe JSON, 429. Он не выбирает business endpoint.

7. `LLM_EXAMPLES.md` открывать только если нужен готовый prompt или пример execution flow. Это не нормативный источник.

8. `AGENTS.md` использовать как стартовый поведенческий чеклист и при споре о формате ответа / подтверждениях / неопределённости. Не использовать `AGENTS.md` для выбора endpoint'а, метода расчёта или business scope.

9. `BUSINESS_CONTEXT.md` открывать только если он есть и задача требует интерпретации результата:
   - аналитика, выводы, executive summary, HTML-отчёт;
   - оценка нормы / риска / отклонения;
   - комментарии к сезонности, масштабу, KPI, структуре расходов или денежным ограничениям.

   Не открывать `BUSINESS_CONTEXT.md` для чисто технических задач: lookup ID, подготовка payload, PDF / документ по известному ID, schema/debug, выбор endpoint'а или проверка contract.

## Первый запуск

Когда агент впервые прочитал документы, он должен:

1. Проверить `.env` и `PLANFACT_API_KEY`.
2. Если `.env` не настроен, сразу дать короткую инструкцию:
   - скопировать `.env.example` в `.env`;
   - вставить ключ в `PLANFACT_API_KEY=...`;
   - не вставлять ключ в чат.
3. Если `.env` настроен, но задача ещё не задана, предложить 3–5 стартовых сценариев:
   - текущие остатки по счетам;
   - операции за период;
   - денежный поток за месяц или квартал;
   - топ контрагентов по доходам или расходам;
   - payload для создания операции без отправки.

Рекомендуемый тон первого ответа:

- короткий;
- прикладной;
- без пересказа всей документации;
- с готовыми вариантами, что можно сделать прямо сейчас.

## Приоритет источников

Для policy, бизнес-семантики и router-правил ориентируйся так:

1. `README.md`
2. `USE_CASES.md`
3. `QUICKSTART.md`
4. `LLM_EXAMPLES.md`

`BUSINESS_CONTEXT.md`, если он есть, не входит в цепочку policy / routing. Он стоит ниже API и локальной документации и используется только как дополнительный слой интерпретации после выбора data source и получения данных.

Локальные MD-файлы являются первичным источником для business routing: intent, method, scope, canonical endpoint, request budget, forbidden endpoints и known runtime caveats.

`apidoc.planfact.io` использовать только как contract-reference:

- для write-payload;
- для редких или неописанных endpoint'ов;
- если локальная документация не содержит нужного параметра;
- если API вернул ошибку схемы;
- если локальный файл прямо требует сверки с contract.

Для типовых read-only сценариев не открывать `apidoc`, если `README.md` и файл по сущности уже содержат endpoint, минимальный набор параметров и known caveats.

Если реальный ответ API расходится с документами, не выдумывай workaround. Честно назови расхождение, покажи фактический ответ и ограничение.

Если реальный ответ API расходится с `BUSINESS_CONTEXT.md`, приоритет у API. Бизнес-контекст не является источником финансовых фактов и не должен использоваться для заполнения отсутствующих данных.

## Канонический Policy

### MUST

#### API-ключ и безопасность

- Не делать ни одного запроса без API-ключа.
- Читать ключ только из `.env` в рабочей папке, переменная `PLANFACT_API_KEY`.
- Если `.env` отсутствует или ключ пустой, остановиться и попросить пользователя настроить `.env` по образцу `.env.example`.
- Никогда не просить вставить ключ в чат и не выводить ключ в логах или ответах.
- Считать любой успешный целевой API-запрос достаточным подтверждением, что ключ рабочий.
- Квоту и rate-limit можно читать из HTTP-заголовков API-ответа: `X-RateLimit-Limit`, `X-Quota-Limit`, `X-Quota-Used`, `X-Quota-Remaining`, `X-Quota-Reset`.
- `X-Quota-Used` и `X-Quota-Remaining` возвращаются без учета текущего запроса.
- Не делать отдельный API-запрос только ради проверки лимита, если лимит уже можно взять из ответа целевого запроса или пользователь не просил проверить квоту.

#### Уточняющие вопросы

- До первого API-запроса задавать максимум `1` уточняющий вопрос, если без него реально меняется итоговая цифра, метод расчёта или сущность.
- **Когда спрашивать метод cash / accrual:** если пользователь просит прибыль, рентабельность, выручку или БДР без явного указания метода — задать один уточняющий вопрос до первого API-запроса. Не спрашивать для: ДДС (всегда движение денег), Баланса (состояние на дату), БДДС (денежный слой).
- **Явный budget-write prompt = уже подтверждённый write:** если пользователь прямо просит создать/обновить бюджет и явно ожидает, что он появится в аккаунте, это считать достаточным подтверждением на budget write. Не запрашивать второе подтверждение; допускается только один уточняющий вопрос, если без него меняется метод расчёта или сам бюджетный контекст.

#### Выбор endpoint'ов

- Для типового запроса выбирать минимальный набор endpoint'ов, а не делать «полный безопасный обход».
- Если пользователь явно просит `read-only`, использовать только read-only endpoint'ы и локальные расчёты.
- Не путать `POST` со state-changing write по одному лишь HTTP-методу: если endpoint документирован как получение списка или поиск без изменения данных, его можно использовать в `read-only` сценарии.
- Read-only `POST` не требует подтверждения пользователя только из-за HTTP-метода. Примеры: `POST /api/v2/reports/opu`, `POST /api/v2/reports/dds`, `POST /api/v1/accounting/balance/table`, `POST /api/v1/operations/list`, `POST /api/v2/dashboardpro/*`, service-методы быстрых фильтров и отчётные / поисковые endpoint'ы, которые явно не меняют данные.
- Если пользователь просит только подготовить payload, можно грузить справочники, но нельзя отправлять write.
- Не добавлять второй endpoint только для контрольной проверки, если формула уже канонически определена в документах или выведена из первого результата.
- Для агрегатов и типовых аналитик предпочитать специализированный data layer, если он сохраняет бизнес-смысл метрики.
- Для типовых read-only сценариев брать routing и минимальные параметры из локальных файлов. `apidoc.planfact.io` открывать только как contract-reference по правилам раздела «Приоритет источников».
- Для запросов про быстрые фильтры, сохраненные фильтры, вкладку **«Быстрые»** или `QuickFilters` открывать `QUICKFILTERS.md`; не придумывать endpoint применения фильтра, если его нет в публичном `apidoc`.
- Все даты в запросах интерпретировать как UTC+0. Не применять коррекцию на локальный часовой пояс пользователя.

#### Трактовка ограничений доступа

- Если штатный endpoint вернул `403`, `forbidden`, `нет доступа`, `недостаточно прав`, `module unavailable`, `tariff`, `subscription` или аналогичный access-сигнал — считать, что текущий штатный сценарий недоступен по тарифу, правам, роли или доступу к сущности.
- Не трактовать такой ответ как «данных нет».
- Пользователю не показывать сырые `errorCode`, `errorCodeTitle`, число кода или enum-имя как основное объяснение. Сначала сказать, что не получилось сделать в терминах задачи пользователя, затем дать понятную причину и один следующий шаг.
- Если в теле по сути только `errorMessage: "Forbidden"` и нет осмысленного `errorCode` / `errorCodeTitle`, не утверждать однозначно «тариф» или «роль». Сказать, что действие запрещено настройками доступа текущего пользователя или политикой аккаунта.
- Если API вернул `200`, `isSuccess == true`, но `data` пустой — считать, что по выбранным параметрам данных нет. Не считать это тарифным ограничением.
- Если endpoint вернул только доступную пользователю часть данных — использовать доступную часть, но явно предупреждать, что результат может быть неполным.
- Если после недоступного штатного сценария пользователь просит альтернативный расчет — явно помечать его как отдельный сценарий, а не как штатный отчет API.

Клиентские формулировки для частых access-ошибок:

| Условие | Как сказать клиенту |
| ------- | ------------------- |
| `NotEnoughApiPermission`, пользователь просил отчет / аналитику | Не могу построить отчет: текущий тариф или настройки аккаунта не включают доступ к API. Проверьте тариф и доступ к API. |
| `NotEnoughApiPermission`, общий API-запрос | Не могу выполнить запрос: текущий тариф или настройки аккаунта не включают доступ к API. Проверьте тариф и доступ к API. |
| `NotEnoughBalancePermission` | Не могу построить баланс: текущий тариф не включает отчет Баланс. Проверьте тариф и доступ к Балансу. |
| `NotEnoughBudgetsPermission` + `errorMessage` содержит `Доступ ограничен владельцем` | Не могу открыть бюджеты: доступ к разделу План ограничен настройками аккаунта. |
| `NotEnoughBudgetsPermission` без текста про владельца | Не могу открыть бюджеты: текущий тариф не включает работу с бюджетами. Проверьте тариф и доступ к разделу План. |
| `NotEnoughDealsPermission` | Не могу построить отчет по сделкам: текущий тариф не включает раздел Сделки. Проверьте тариф и доступ к Сделкам. |
| `NotEnoughPermissions` или `TariffAccessDenied` | Не могу выполнить действие: текущий тариф не включает эту функцию. Проверьте тариф и доступ к нужному разделу. |
| `Forbidden` без осмысленного `errorCodeTitle` | Не могу выполнить действие: операция запрещена настройками доступа текущего пользователя. Проверьте права пользователя и доступ к нужному разделу. |
| `Unauthorized` | Не могу обратиться к API: текущий API-ключ не проходит авторизацию. Проверьте ключ и доступ к API. |

Технические детали вроде `NotEnoughApiPermission` добавлять только по запросу пользователя или если ответ предназначен для поддержки.

#### Известные путаницы endpoint'ов

- **Дашборды / Показатели ПРО.** Если пользователь просит дашборд, виджет или блок из «Показателей ПРО», первичный источник — профильный `POST /api/v2/dashboardpro/*` из `DASHBOARDS_PRO.md`, включая фильтры периода, проектов, сделок, ЮЛ/счетов, потока и метода учёта. Не начинать с `operations/list`, `reports/*`, `bizinfos/*` или ручной реконструкции до проверки профильного dashboard endpoint и корректности payload.
- **Остатки по счетам.**
  - «Мои счета» → `GET /api/v1/bizinfos/accountshistory` (накопить factValue локально).
  - Для «Мои счета» `accountshistory` — ведущий список строк; `/accounts` можно использовать только как lookup по уже полученным `accountId`, не добавлять отсутствующие в истории счета.
  - «Показатели ПРО» → `POST /api/v2/dashboardpro/accountbalance`.
  - «Показатели» (базовый) → `POST /api/v1/dashboards/accountbalance`.
  - `GET /api/v1/accounts` — только для справочника счетов, не для остатков.

#### Справочники и filter-first

- Не делать preflight через `GET /api/v1/accounts` по привычке.
- Использовать `/accounts`, `/projects`, `/contragents`, `/operationcategories` только если действительно нужны ID или названия.
  - «Действительно нужны»: пользователь указал сущность по имени или ИНН, без `ID` нельзя построить следующий запрос, в кэше `ID` нет.
  - «Не нужны»: агент хочет убедиться, что сущность «существует», для перекрёстной проверки результатов другого endpoint или без прямой связи с итоговой метрикой.
- Для write-flow и точечных read-flow сначала получать только metadata / справочник, который блокирует следующий шаг или нужен для построения payload. После получения ID переходить к точечному data endpoint, а не обновлять весь справочник.
- Справочники кэшировать отдельно по типу и параметрам фильтра в рамках текущей сессии. Если нужный `accountId`, `projectId`, `contrAgentId`, `companyId`, `operationCategoryId` или другой ID уже есть из кэша, create-response или предыдущего list-response, повторный lookup запрещён.
- Не делать refresh справочника после каждого create/update. Использовать ID из ответа write-запроса; общий refresh допустим один раз в конце batch-сценария или когда следующий шаг реально зависит от новых справочных атрибутов.
- Для любых списковых и справочниковых endpoint'ов **кроме `/api/v1/operations`** сначала применять server-side фильтры, если пользователь уже дал сужающие признаки:
  - название / часть названия;
  - ИНН, `externalId`, конкретный ID;
  - дата / период;
  - активность, статус, тип, группа, валюта и похожие поля.
- Не выгружать полный справочник или список по умолчанию, если narrow lookup можно сделать одним фильтрованным запросом.
- Полный fetch списка допустим только если пользователь явно просит весь справочник, если server-side фильтров в contract нет или если узкий запрос уже доказанно не сужает выдачу.

#### `/api/v1/operations` — особое поведение

- `GET /api/v1/operations` на этом стенде может игнорировать `searchString`, `operationDateStart`, `operationDateEnd` и другие параметры. HTTP 200 не означает, что фильтрация сработала.
- Не начинать с full `GET /api/v1/operations`, если задачу покрывает narrow `POST /api/v1/operations/list`.
- Если full `GET /api/v1/operations` всё же нужен после post-check, явного согласия пользователя или из-за недоступности narrow-фильтра, считать его одним полным дампом на сессию:
  - фильтровать локально по нормализованным полям;
  - на `offset` и `limit` как на способ уменьшить ответ нельзя рассчитывать;
  - повторно не загружать `/operations`, если уже есть актуальный кэш;
  - исключение: если пользователь явно сообщил о внешних изменениях или цифры расходятся с UI — инвалидировать кэш и сделать один fresh reload.
- Для server-side текстового поиска использовать `POST /api/v1/operations/list` с `searchString` в JSON body: он выполняет substring-поиск. В official `apidoc` ReDoc это поле может отображаться как `filter.searchString`. Для точного совпадения дополнительно фильтровать локально.
- `GET /api/v1/operations` и `POST /api/v1/operations/list` — не взаимозаменяемы:
  - `GET` может игнорировать `searchString` и отдавать полный dump;
  - `POST /operations/list` — для server-side фильтрации, когда она реально нужна.
- Для точечного поиска малого набора операций:
  - сначала `narrow-first` запрос с самым узким явным периодом;
  - для свежих операций предпочитать `lastModificationDateStart` / `lastModificationDateEnd` в JSON body; в official `apidoc` ReDoc эти поля могут отображаться как `filter.lastModificationDateStart` / `filter.lastModificationDateEnd`;
  - если есть кэш — добирать локальной фильтрацией.
- Если запрос широкий и `/operations` ещё не загружен — предупредить пользователя заранее.
  - «Широкий»: диапазон больше 3 месяцев или нет сужающего признака (статья, контрагент, проект).
- Если `/operations` вернул capped-ответ (`items == 10000` при `total == 0`) — не считать это доказательством отсутствия операций.
- Если обычный списковый endpoint возвращает `total > 20 000` — не догружать молча, предупредить пользователя.

#### Долг и расчётные метрики

- Для запросов `дебиторка`, `кредиторка`, `прибыль по КА`, `должники`, `кредиторы` в стандартном срезе — использовать готовый endpoint `GET /api/v1/contragents/calculated/{id}` с нужным `calculationMethod`; для типа долга дополнительно `GET /api/v1/contragents/{id}/additional`. Для списка КА сначала получить ограниченный список КА, затем точечные `GET /calculated/{id}`; не запускать N+1 по большому списку молча. Подробно → `LOOKUPS.md`, раздел «ContrAgents → Финансовые агрегаты».
- Для дебиторки / кредиторки **по одному проекту** — `GET /api/v1/projects/{id}/summary` (поля `invoiceBalance`, `invoiceSupplyBalance`). Подробно → `LOOKUPS.md`, раздел «Projects → Финансовые агрегаты».
- Для **плана из бюджета + факта по проектам** — `GET /api/v1/projects/calculations`. Не собирать это связкой `BUDGETS.md` + `bizinfos`: `bizinfos.planValue` — это суммы из операций, а не план сохранённого бюджета.
- Ручная реконструкция долга из `operations/list` (см. `USE_CASES.md`, раздел «Задолженность и просрочка») допустима только как fallback: нестандартный фильтр результата (например, «дебиторка > X»), нестандартный grouping, долги через `sellingDealId / purchaseDealId`, или готовый endpoint вернул ошибку. В ответе пометить, что данные посчитаны вручную.
- Долг не подменять списком сделок или `invoice-documents`.
- Для debt-сценариев всегда явно фиксировать `as-of date`: если пользователь не указал дату, по умолчанию брать текущую. Имя query-параметра брать из доменного файла / contract конкретного endpoint'а; для `GET /api/v1/contragents/calculated/{id}` это `dateEnd` (см. `LOOKUPS.md`).

#### Write-операции и ошибки фильтрации

- Для write-операций с привязкой к справочникам сначала выполнять минимальные blocking lookup по сущностям, которые могут запретить запись. Если найден блокер (`project.closed == true`, проект / контрагент / счёт архивны, счёт связан с архивным ЮЛ), немедленно остановиться: не искать остальные ID, не собирать payload и не делать дополнительные справочные запросы. Сообщить пользователю конкретный блокер и какие данные уже проверены.
- Перед любым state-changing `POST`, `PUT`, `DELETE` показывать payload или список изменений и ждать явного подтверждения. Read-only `POST` из отчётных, поисковых и dashboard-сценариев подтверждения не требует.
- Исключение для budget-flow: если пользователь уже явно попросил создать/обновить бюджет в аккаунте, не делать дополнительный stop-for-confirmation. Вместо этого идти по минимальному write-маршруту из `BUDGETS.md`.
- После успешного write-запроса не повторять его автоматически.
- Если server-side фильтр вернул ошибку — диагностировать и исправить параметр, повторить server-side запрос. Не переключаться молча на полный dump с локальной фильтрацией. Если server-side фильтрация принципиально невозможна для данного поля — явно сообщить пользователю.
- Для `/api/v1/operations` full dump допустим только после post-check, явного согласия пользователя или если narrow endpoint принципиально не покрывает сценарий. Для всех остальных endpoint'ов переход на полный dump без сообщения запрещён.

### SHOULD

- Кэшировать результаты в рамках одной сессии.
- Для аналитических endpoint'ов кэшировать ответ по точному набору параметров, если пользователь не просил свежий пересчет.
- После write-запроса, явного сообщения о внешних изменениях или длинной паузы уточнять, нужно ли обновить кэш.
- Если пользователь говорит, что данные в аккаунте уже обновились, или текущее число не сходится с UI, не продолжать аналитику на старом `/operations`-кэше.
- Не передавать полный JSON `/operations` в контекст LLM. Сначала парсить и сокращать до нужных полей.
- Для повторной загрузки использовать инкрементальные параметры, если они доступны:
  - `filter.changesFromDate` для справочников;
  - `lastModificationDateStart` / `lastModificationDateEnd` для операций в JSON body `POST /api/v1/operations/list`; в official `apidoc` ReDoc эти поля могут отображаться как `filter.lastModificationDateStart` / `filter.lastModificationDateEnd`.

### MAY

- Идти во внешний `apidoc`, если кейс редкий, edge-case или не покрыт локальным набором.
- Проверять реальный ответ API только если документов недостаточно и без проверки высок риск ошибиться.

Не делай реальный API-запрос только «на всякий случай».

## Intent Router

Перед выбором endpoint'ов агент должен разложить prompt на 4 слоя:

1. `setup` — чтение документов, проверка `.env`, режим `read-only` или `write`;
2. `business intent` — какую метрику, список или сущность реально хочет пользователь;
3. `transform` — фильтрация, группировка, time series, агрегаты;
4. `delivery` — таблица, короткий ответ, TSV, JSON, `report.html`, локальная страница, график.

Ключевое правило:

- endpoint'ы выбираются только по `business intent` и `transform`;
- `setup` и `delivery` сами по себе не должны добавлять новые API-вызовы;
- длинный prompt не означает большой набор endpoint'ов.

### Что не должно влиять на выбор data source

Инструкции, относящиеся к `setup` и `delivery`, сами по себе не должны менять выбор endpoint'ов.

К `setup` относятся:

- чтение документации;
- анализ локального набора;
- проверка `.env`;
- получение или проверка API-ключа;
- подтверждение режима `read-only` или `write`.

К `delivery` относятся:

- таблица;
- график;
- HTML;
- страница;
- локальный запуск в браузере;
- оформление;
- отчет;
- файл результата.

#### Правила локальных HTML-отчетов и страниц

- Если пользователь просит локальный HTML-отчет, страницу, дашборд или browser-ready файл без отдельного визуального ТЗ, любой такой результат обязан использовать визуальную палитру `app.planfact.io`, если пользователь явно не попросил другой стиль.
- HTML-результат должен быть самодостаточным, читаемым в браузере и без внешних зависимостей.
- Под «палитрой `app.planfact.io`» понимать прежде всего базовые UI-цвета продукта: основной бирюзовый акцент, светлые бирюзовые подложки, белые поверхности, серые границы и нейтральный светлый фон. Не изобретать произвольную тему, если пользователь не попросил иное.
- Это требование относится и к простым табличным HTML-файлам, dump-страницам, lookup-спискам и техническим browser-ready выгрузкам: нейтральный «серый шаблон по умолчанию» считать дефектом delivery.
- В видимой части локального HTML-результата по умолчанию использовать только русский язык: заголовки, подписи, колонки, пояснения, карточки, кнопки, легенды, статусы и подписи к графикам.
- Иностранные слова и англоязычные служебные метки в видимой части HTML не использовать, кроме случаев, когда это неизбежное собственное имя, официальный бренд пользователя или буквальный текст из первичных данных.
- Локализовать только оболочку отчета, а не содержимое данных.
- Все значения, пришедшие из API или исходных файлов, выводить без перевода и без нормализации написания.
- Не переводить названия контрагентов, проектов, брендов, компаний, счетов, `comment`, назначение платежа, URL, email, коды, ID и другие значения из данных.
- Если значение пришло из данных, сохранять его в оригинальном написании. Пример: заголовок таблицы — «Контрагент», значение в строке — `Google`, а не «Гугл».
- Технические идентификаторы, пути endpoint'ов, имена HTTP-методов и другие служебные англоязычные поля по умолчанию выносить в отдельный служебный файл или лог, а не показывать в основной визуальной части отчета.
- Перед выдачей локального HTML-результата быстро проверить delivery-слой:
  - палитра `app.planfact.io` действительно применена;
  - оболочка отчета русскоязычная;
  - служебные API-детали не доминируют в основной визуальной части;
  - таблицы и карточки читаемы без ручной правки CSS.
- Эти правила относятся только к `delivery` и не меняют выбор endpoint'ов, расчётную формулу или набор API-вызовов.
- Если пользователь явно не просит JSON, не отдавать сырой JSON как основной результат. Для списков и структур использовать таблицу, для объяснений — краткий текст, для отчётов и визуализации — HTML.
- Если агент строит локальный HTML, dashboard, browser-ready страницу или другой UI, API-запросы должны выполняться только на этапе подготовки dataset или по явному действию пользователя, которое требует новых данных. Сам `render`, перерисовка карточек, сортировка, раскрытие строк, hover, переключение визуального режима, resize, theme change и повторная генерация HTML не должны заново дергать API.
- Для UI-результатов сначала собрать один нормализованный dataset с ключом кэша по endpoint + параметрам периода + фильтрам + методу расчёта, затем рендерить из него все карточки, таблицы и графики. Если dataset уже покрывает выбранный фильтр или вкладку, использовать локальную фильтрацию.
- Авто-refresh UI запрещён без явного пользовательского требования. Если refresh нужен, он должен быть отдельным действием с понятной причиной: write в текущей сессии, пользователь сообщил о внешних изменениях, данные не сходятся с UI или истёк явно заданный пользователем интервал обновления.

Все такие указания меняют только способ подготовки и выдачи результата. Выбор data source определяется только:

- `business intent`;
- периодом;
- методом расчета;
- целевой сущностью;
- типом аналитики.

### Канонический порядок выбора

0. Проверь кэш текущей сессии. Если нужный слой данных уже загружен — брать данные оттуда, endpoint не выбирать.
1. Определи одну основную бизнес-метрику или один основной список.
2. Проверь, нет ли у неё прямого специализированного endpoint'а.
3. Зафиксируй метод, scope, primary endpoint, ожидаемое число API-вызовов и forbidden endpoints из таблицы.
4. Если без уточнения есть риск смешать разные методы расчёта — задай один короткий вопрос до первого API-запроса.
5. До первого API-вызова не открывай дополнительный файл или `apidoc`, если выбранный маршрут уже покрыт таблицей и файлом по сущности.
6. Для operation-driven read-only сценариев default path = `POST /api/v1/operations/list` с максимально узкими server-side фильтрами; `GET /api/v1/operations` не является стартовым маршрутом по умолчанию.

### Пример разбора prompt'а

**Запрос:** «Покажи выручку по проектам за 2025 год в виде HTML-страницы»

| Слой              | Содержание                                            | Влияет на API? |
| ----------------- | ----------------------------------------------------- | -------------- |
| `setup`           | прочитать README, проверить `.env`, режим read-only   | ❌              |
| `business intent` | выручка по проектам — готовый агрегат ОПУ по проектам | ✅              |
| `transform`       | фильтр периода 2025, `reportGenMethod: Projects`      | ✅              |
| `delivery`        | HTML-страница                                         | ❌              |

**Маршрут:** если метод не указан — 1 короткий вопрос (`cash` или `accrual`), затем 1 × `POST /api/v2/reports/opu` с `reportGenMethod: Projects` → локальная агрегация / рендер HTML.
«HTML-страница» не добавляет endpoint'ов.

### Fast Path для типовых запросов

Если prompt уже задаёт:

- метрику или список;
- период;
- метод расчета, если он важен;
- сущность;

то агент должен выбрать строку из таблицы `Каноническая таблица маршрутизации`, открыть указанный файл по сущности и при необходимости `USE_CASES.md`, затем исполнять маршрут без второй интерпретации prompt'а.

Правила fast path:

- один раз выбрать канонический data source;
- один раз посчитать метрику локально;
- только после этого собирать таблицу, HTML, график или локальную страницу;
- не добавлять второй endpoint только для «контрольной проверки», если формула уже канонически определена;
- не открывать повторно весь набор документов после того, как найден точный сценарий.

Повторная валидация допустима только если:

- в `README.md` и файле по сущности нет канонического сценария;
- фактический ответ API расходится с документами;
- результат конфликтует с явной цифрой пользователя;
- sanity check не проходит: неверный знак, порядок величины явно странный, все значения нули там, где это маловероятно.

`HTML`, `график`, `дашборд`, `страница`, `оформи красиво` не являются поводом для отдельной валидации или нового endpoint'а.

### Latency Budget

Это не hard SLA, а целевой budget для поведения агента.

- если данные уже в локальном кэше и формула канонически определена, числовой ответ или таблицу нужно собирать в один расчётный проход;
- для dashboard/page-сценариев не делать отдельные расчётные проходы под каждую карточку, график и таблицу; сначала собрать один нормализованный dataset, потом переиспользовать его;
- не делать вторую или третью сверку только потому, что результат будет показан в HTML;
- если budget уже потрачен, а red flags нет, остановить дополнительные проверки и отдать текущий результат с короткой пометкой о методе и ограничениях.

Практические ориентиры:

| Сценарий                                                            | Целевое время  |
| ------------------------------------------------------------------- | -------------- |
| warm cache, число или таблица                                       | до `30–60` сек |
| warm cache, локальная страница или dashboard                        | до `60–90` сек |
| cold start с одним тяжёлым fetch `/operations` + локальная страница | до `2–3` мин   |

Если ответ выходит далеко за эти ориентиры, ищи лишний цикл интерпретации, повторный расчёт или вторую валидацию без причины.

### Speed Rules

Чтобы не тратить лишние минуты на типовые аналитические запросы:

- для `html`, `xlsx`, `дашборда` и `страницы` сначала собрать один нормализованный dataset, потом рендерить все форматы из него;
- если метод расчёта и состав метрики уже канонически определены, не добавлять второй endpoint только для внутренней проверки;
- если запрос `по проектам`, не добавлять company-level summary, пока пользователь явно не попросил общий итог компании;
- если в предыдущем аналогичном периоде база равна `0`, не тратить лишний проход на процент роста; помечать такой кейс как `new base`;
- повторный fetch уже загруженного `/operations` или `/bizinfos/*` в той же сессии считать ошибкой маршрутизации, а не допустимой подстраховкой;
- после одного full/capped ответа `/operations` не перебирать в той же сессии несколько альтернативных broad-фильтров вроде `searchString` → контрагент → сумма без нового более узкого периода или окна `lastModification`; это лишняя нагрузка без новой информации;
- в batch write-сценарии по умолчанию не делать `verify` для каждой созданной записи; достаточно выборочно проверить `1–3` первых операций в партии и все structurally special кейсы: `split items`, `move`, новая валюта, новый тип счёта;
- если новый счёт только что создан через `POST /api/v1/accounts`, использовать `accountId` прямо из create-response и не обновлять `/accounts` после каждого create; refresh справочника делать один раз в конце партии или только когда он реально нужен следующему lookup;
- не ходить в `/contragents`, если write-payload не использует `contrAgentId`; имя магазина или получателя можно сохранить в `comment`, а lookup контрагента в таком случае считается лишней нагрузкой.

### Request Budget

- В типовом read-only сценарии начни с `1` primary endpoint.
- Добавляй `2-й` endpoint только если первый реально не покрывает вторую метрику без искажения смысла.
- Не добавляй справочник только «для проверки».

Ориентиры:

- текущие остатки по счетам: `1`
- cash flow за период: `1`
- операции за период: `1`
- операции по проекту: `2`
- рентабельность проектов методом начисления: `1`; максимум `2`, только если нужен отдельный project lookup
- read-only пакет `остатки + операции + ДДС`: `3`
- owner dashboard: `2–4`

## Запрещённые паттерны

Сводный список. Каждый пункт — явный запрет с альтернативой. Примеры неправильного поведения → `LLM_EXAMPLES.md`, раздел «Антипаттерны».

1. **`GET /accounts` без нужды в `accountId`** — использовать id из кэша или create-response.
2. **Повторный fetch `/operations` в той же сессии** — фильтровать кэш. Инвалидировать только при явном сигнале: «данные обновились», «цифры не сходятся с UI», «я добавил операцию», или был write в сессии.
3. **Второй `OPU` / `bizinfos` для «контрольной проверки»** — HTML, дашборд и график не меняют data source.
4. **`GET /contragents`, когда `contrAgentId` не нужен в write-payload** — имя сохранять в `comment`.
5. **Company-level summary без явного запроса** — проектный рейтинг ≠ итог компании; не добавлять второй OPU.
6. **Fallback в DashboardPro для долгов/прибыли КА** — для стандартного среза КА сначала `GET /contragents/calculated/{id}`; DashboardPro не является fallback без отдельного dashboard-сценария.
7. **`includeTrendData: true` без явного запроса тренда** — включать только если пользователь просит спарклайн / динамику.
8. **Verify после каждой write в batch** — достаточно `1–3` первых и всех structurally special кейсов.
9. **Параллельный fetch `bizinfos` + `reports/opu` для одного факта** — один источник истины; выбор → `BUDGETS.md`.
10. **Чтение всего набора документов на каждый запрос** — выбрать маршрут в `README.md → Intent Router`, затем открыть только указанный файл по сущности.

## Поводы пересобрать план до первого API-запроса

> Отличие от «Запрещённых паттернов» выше: эти триггеры срабатывают **до первого API-запроса**, пока агент ещё выбирает маршрут. «Запрещённые паттерны» — запреты, актуальные на протяжении всей сессии.

Если видишь эти паттерны, остановись и собери маршрут заново:

- длинный prompt сам по себе тянет к внешнему `apidoc` или лишним endpoint'ам;
- хочется открыть `/accounts` без явной необходимости в `accountId` или справочнике счетов;
- хочется начать со структурного агрегата (`bizinfos/incomeoutcomehistorybyoperationcategories` или `dashboardpro/paymentstructure-*`), хотя нужен список операций, точный итог по статье или time series;
- хочется смешать `cashflow` и accrual-расчет в одном сравнении;
- хочется считать company-level summary простой суммой строк проектного рейтинга.

## Общий источник истины для составных метрик

Если пользователь просит несколько метрик на одном периоде и в одном методе расчета, агент должен по возможности считать их из одного и того же слоя данных.

Ориентиры:

- несколько accrual-метрик на одном периоде -> по возможности один общий accrual-layer;
- несколько cash-метрик на одном периоде -> по возможности один общий cash/report-layer;
- список операций + несколько срезов по нему -> один общий operations-layer.

Запрещено:

- смешивать `cashflow` и accrual-расчет в одном графике или сравнении без явной маркировки;
- брать «удобный» endpoint, если он меняет бизнес-смысл метрики;
- брать структуру как источник временного ряда, если нужен именно time series;
- для спорных бизнес-метрик вроде `выручки` не придумывать состав самостоятельно; стандартную трактовку смотреть в `USE_CASES.md`.
- для аналитики `по проектам` не считать, что сумма строк рейтинга обязана совпадать с company-level summary или ОПУ; если нужен итог компании, scope смотреть в `USE_CASES.md`.

## Кэш и инвалидация

| Что кэшировать   | Когда загружать                                     | Когда инвалидировать                                                                                    |
| ---------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| lookup-layer     | когда нужны ID, названия, статусы или иерархия      | **начало новой сессии**; новая сессия не наследует кэш предыдущей; после релевантного write, явного refresh, внешних изменений                    |
| operations-layer | когда нужен список операций или accrual-аналитика   | **начало новой сессии**; новая сессия не наследует кэш предыдущей; после write по операциям, явного refresh, внешних изменений, несоответствия UI |
| report-layer     | когда пользователь просит соответствующую аналитику | **начало новой сессии**; новая сессия не наследует кэш предыдущей; после релевантного write, явного refresh, внешних изменений                    |

**Переиспользование отчётных ответов в рамках сессии:** если отчётный endpoint уже вызван (OPU, DDS, Balance, bizinfos) и следующий вопрос пользователя покрывается тем же ответом (например, «а какая выручка?» после уже полученного ОПУ) — брать данные из кэшированного ответа, не вызывать endpoint повторно. В новой сессии кэша нет — вызывать заново.

Новая сессия никогда не наследует `lookup-layer`, `operations-layer` и `report-layer` кэш из предыдущей сессии. В новой сессии агент должен считать кэш пустым, даже если похожий запрос уже выполнялся ранее.

## Внешний `apidoc`

Во внешний `apidoc` нужно идти только для:

- редких endpoint'ов, не описанных в локальном наборе;
- точного payload write-запроса;
- deprecated или legacy-веток;
- спорных параметров и схем ответов, которых нет в локальных документах;
- ошибки схемы, если локальные known caveats не объясняют ответ API.

Во всех типовых read-only business-кейсах сначала опирайся на этот набор.
Если `README.md` и файл по сущности уже содержат endpoint, минимальный набор параметров и runtime caveats, `apidoc` не открывать.

## Границы набора

- Этот набор не заменяет полный `apidoc`.
- Он покрывает типовые read-only business-кейсы и базовые write-flow правила.
- Если кейс редкий, спорный или не покрыт локальным набором, агент должен честно сказать об ограничении и только потом идти во внешний reference.
