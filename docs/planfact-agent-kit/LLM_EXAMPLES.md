# ПланФакт API — LLM Examples

Этот файл содержит только готовые prompt templates. Канонические правила поведения агента находятся в `README.md`, технические детали — в `QUICKSTART.md`, бизнес-трактовки — в `USE_CASES.md`.

Если пример ниже расходится с `README.md` или доменным файлом по маршруту endpoint'ов, считать пример устаревшим: `LLM_EXAMPLES.md` не переопределяет policy, request budget или domain routing.

Использовать примеры как regression tests для поведения агента. Пример должен содержать реалистичный пользовательский prompt, короткий expected route и ориентир по API-budget. Не дублировать в этом файле payload, схемы endpoint'ов и подробные правила из доменных файлов.

Пример не должен вводить новый route, которого нет в `README.md → Intent Router`.

Все шаблоны ниже предполагают одну и ту же базу:

- агент читает ключ только из `.env` в рабочей папке;
- используется переменная `PLANFACT_API_KEY`;
- если `.env` не настроен, агент сначала просит пользователя создать его по образцу `.env.example`.

Общий префикс для всех prompt templates:

```text
Работай по локальной документации ПланФакт API.
Используй PLANFACT_API_KEY из .env.
```

При использовании шаблона добавлять этот префикс перед выбранным примером. Ниже он не повторяется, кроме сценария первого запуска.

В общем префиксе фраза `Работай по локальной документации ПланФакт API` означает:

- не перечитывать весь набор без необходимости;
- сначала открыть `README.md`;
- `QUICKSTART.md` использовать только как index;
- затем открыть доменный файл по сущности запроса;
- `USE_CASES.md` открывать, если формулировка влияет на метод расчёта и область анализа.

Для новых аналитических шаблонов метод расчёта и область анализа фиксируй прямо в тексте prompt, а не только в комментарии ниже. Prompt может просить финансовые выводы, но они должны быть основаны только на данных и методе расчёта, без заранее заданной оценки. В prompt templates используй русские подписи: `Метод расчёта`, `Источник данных`, `Область анализа`, `Результат`.

## Шаблоны промтов

### Первый запуск без конкретной задачи

```text
Работай по локальной документации ПланФакт API.
Используй PLANFACT_API_KEY из .env.
Если .env не настроен, подскажи, как создать его из .env.example.
Если конкретной задачи ещё нет, предложи короткие варианты, с чего можно начать.
```

Бюджет API-вызовов: `0`.

### Текущие остатки и ликвидность по счетам

```text
Покажи текущие остатки по счетам из раздела «Мои счета» с общим итогом.
Отдельно выдели счета с наибольшими остатками и счета с нулевым или отрицательным остатком, если такие есть.
Результат: краткая финансовая сводка по ликвидности на текущую дату.
```

Expected route: `DASHBOARDS.md → Мои счета`; если пользователь явно просит «Показатели» или «Показатели ПРО» — использовать соответствующий dashboard route.  
Бюджет API-вызовов: `1`, максимум `2`.


## Антипаттерны (top mistakes)

Короткие regression-cases: что агент не должен делать, даже если это кажется «безопасной проверкой».

### ❌ Лишний preflight перед write

Запрос: «Создай расходную операцию на 5 000 руб. по статье "Аренда", счёт — Расчётный счёт Сбер».

Не делать: отдельные `/accounts` и `/operationcategories` “для проверки”, если ID уже есть в кэше или достаточно одного filter-first lookup.  
Правильный route: `OPERATIONS.md` + `LOOKUPS.md` только при необходимости ID.  
Бюджет: `1–2`, не `4+`.

### ❌ Delivery провоцирует повторный отчёт

Запрос: «Покажи ОПУ за 2025 год в виде HTML-страницы».

Не делать: повторный `POST /api/v2/reports/opu` “для финальной проверки перед HTML”.  
Правильный route: один отчётный запрос, затем рендер из полученного ответа.  
Бюджет: `1`.

### ❌ Project-layer сверяется с company-level без запроса

Запрос: «Покажи рентабельность проектов методом начисления за 2024 год».

Не делать: автоматически добавлять company-level ОПУ для сверки с проектным рейтингом.  
Правильный route: `REPORTS_V2.md → OPU Projects`; company-level только по явной просьбе или правилу доменного файла.  
Бюджет: `1`, максимум `2` при обоснованном lookup.

### ❌ Full dump `/operations` как первый шаг

Запрос: «Покажи операции за март».

Не делать: начинать с `GET /api/v1/operations` и локально фильтровать месяц.  
Правильный route: `OPERATIONS.md → POST /api/v1/operations/list` с узким периодом.  
Бюджет: `1`; full dump — не default.

### ❌ Budget write считается успешным без контрольного GET

Запрос: «Построй БДР за 1 полугодие 2026 и создай его в аккаунте».

Не делать: считать задачу завершённой только по `isSuccess: true` на `POST` / `PUT`.  
Правильный route: `BUDGETS.md → Fast-path`; после write проверить сохранённую структуру через `GET budget/settings`.  
Бюджет: контрольные `GET` после write обязательны; дополнительные аналитические сверки не нужны.
