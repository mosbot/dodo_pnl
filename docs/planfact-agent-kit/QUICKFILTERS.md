# ПланФакт API — QUICKFILTERS для AI-агента

Этот файл описывает бизнес-логику быстрых фильтров и публичные методы `QuickFilters`.

`README.md` — policy и router.
`QUICKSTART.md` — technical index.
`OPERATIONS.md` — операции и фильтрация списка операций.
`REPORTS_V2.md` — ОПУ и ДДС.

## Когда открывать

- пользователь спрашивает про быстрые фильтры, сохраненные фильтры, вкладку **«Быстрые»** или `QuickFilters`;
- нужно создать, изменить, удалить быстрый фильтр;
- нужно добавить фильтр в избранное, убрать из избранного или изменить порядок;
- нужно понять, почему быстрый фильтр не применился, пропал, стал пустым или дал пустой результат;
- быстрый фильтр относится к **Операциям**, **ОПУ** или **ДДС**.

## Что такое быстрые фильтры

Быстрые фильтры — это сохраненные пользователем наборы условий фильтрации для повторного применения.

Используются:

- в разделе **Операции**;
- в отчетах **ОПУ**;
- в отчетах **ДДС**.

## Публичные методы QuickFilters

Для частых read-only сценариев использовать таблицу ниже и правила применения фильтра в этом файле. Точные query/body-параметры и response shape брать из official `apidoc.planfact.io` только для create/update/delete, спорных параметров или если нужного поля нет в локальной документации.

| Метод    | Endpoint                                         | Назначение                                          | Тип для агента |
| -------- | ------------------------------------------------ | --------------------------------------------------- | -------------- |
| `GET`    | `/api/v1/quick-filters`                          | Получает список быстрых фильтров                    | read           |
| `POST`   | `/api/v1/quick-filters`                          | Создает новый быстрый фильтр                        | write          |
| `GET`    | `/api/v1/quick-filters/directories`              | Получает справочники для быстрых фильтров           | read           |
| `POST`   | `/api/v1/quick-filters/exclude-ids`              | Фильтрация сущностей по исключаемым идентификаторам | read/service   |
| `POST`   | `/api/v1/quick-filters/periods/calculate`        | Расчет гибких периодов быстрого фильтра             | read/service   |
| `PUT`    | `/api/v1/quick-filters/position`                 | Изменяет позиции быстрых фильтров                   | write          |
| `DELETE` | `/api/v1/quick-filters/{quickFilterId}`          | Удаляет быстрый фильтр                              | write          |
| `GET`    | `/api/v1/quick-filters/{quickFilterId}`          | Получает быстрый фильтр по идентификатору           | read           |
| `PUT`    | `/api/v1/quick-filters/{quickFilterId}`          | Обновляет быстрый фильтр                            | write          |
| `PUT`    | `/api/v1/quick-filters/{quickFilterId}/favorite` | Добавляет/удаляет фильтр из избранного              | write          |

Перед write-методами агент должен показать payload или список изменений и дождаться явного подтверждения, если пользователь заранее не дал явную команду на изменение.

Минимальные read-only маршруты:

- список быстрых фильтров → `GET /api/v1/quick-filters`;
- один быстрый фильтр по известному ID → `GET /api/v1/quick-filters/{quickFilterId}`;
- справочники для формы фильтра → `GET /api/v1/quick-filters/directories`;
- расчёт плавающего периода → `POST /api/v1/quick-filters/periods/calculate`.

Не открывать `apidoc` и не делать дополнительные запросы, если задача только объяснить логику быстрых фильтров.

## Применение быстрого фильтра

Действие **«применить быстрый фильтр»** не оформляется отдельным write-методом.

Для агента применение означает:

1. Получить быстрый фильтр через `GET /api/v1/quick-filters` или `GET /api/v1/quick-filters/{quickFilterId}`.
2. Взять сохраненные условия.
3. Передать условия в целевой data source:
   - операции — маршрут операций;
   - ОПУ — `POST /api/v2/reports/opu`;
   - ДДС — `POST /api/v2/reports/dds`.

Не создавать отдельный запрос применения и не искать альтернативный endpoint.

## Общие бизнес-правила

### Принадлежность к разделу

Быстрый фильтр всегда принадлежит одному из разделов: **Операции**, **ОПУ** или **ДДС**. По contract `GET /api/v1/quick-filters` всегда вызывается с обязательным параметром `filter.sectionId`; без него не строить запрос.

### Доступ и область

- Полный доступ: `Владелец`, `Администратор`, `Менеджер`, `Оператор`.
- Для роли `Гость` быстрые фильтры недоступны: вкладка **«Быстрые»** не отображается, доступен только стандартный набор фильтров.
- Быстрые фильтры сохраняются **на уровне пользователя**; общего доступа к фильтрам других пользователей нет.
- Функциональность поддерживается только в **веб-интерфейсе desktop**.
- Быстрые фильтры удаляются при очистке данных аккаунта и не восстанавливаются из backup.

### Действия, избранное и порядок

Пользователь может создать, применить, редактировать, удалить быстрый фильтр, добавить/убрать из избранного и менять порядок внутри блока.

Быстрые фильтры делятся на два блока:

- **Избранные** — отображаются сверху;
- **Обычные** — отображаются ниже.

Перемещение возможно только внутри своего блока. Порядок сохраняется между сессиями.

### Имя и жизненный цикл

- Имя быстрого фильтра должно быть уникальным в рамках пользователя; дубликат имени не сохраняется.
- После создания фильтр появляется в списке, но **не применяется автоматически**.
- Чтобы фильтр начал действовать, пользователь должен отдельно выбрать его во вкладке **«Быстрые»**.
- После применения система переключает пользователя на вкладку **«Общие»**, заполняет стандартные поля сохраненными значениями и перестраивает данные.
- Над стандартными фильтрами отображается тег примененного быстрого фильтра.
- Ручное изменение полей во вкладке **«Общие»** снимает тег.
- Корзина сбрасывает фильтры к значениям по умолчанию и снимает тег.

### Удаление связанных объектов

Если из системы удаляется объект, который использовался в быстром фильтре:

- объект автоматически исключается из критериев;
- если фильтр становится пустым, он удаляется автоматически;
- пользователю показывается информер об изменении или удалении фильтра.

## Быстрые фильтры в «Операциях»

Быстрый фильтр операций сохраняет параметры стандартного фильтра, доступные в аккаунте.

Возможные параметры:

- тип операции;
- дата оплаты;
- юрлица и счета;
- контрагенты;
- статьи учета;
- проекты;
- диапазон суммы;
- другие поля стандартного набора фильтров, если они доступны в текущем аккаунте.

Правила:

- если поле скрыто в стандартной фильтрации аккаунта, его нельзя использовать в быстром фильтре;
- это особенно важно для фильтра по **дате начисления**;
- если скрыты типы операций **Начисление / Отгрузка / Поставка**, они не должны отображаться в форме быстрого фильтра;
- если быстрый фильтр содержит параметр, недоступный по тарифу или настройкам, результат фильтрации пустой и пользователь видит ошибку / информер.

## Быстрые фильтры в ОПУ и ДДС

### Область и поля

- Быстрые фильтры ОПУ и ДДС хранятся раздельно.
- Перенос фильтра между ОПУ и ДДС не предусмотрен.
- Изменения в одном отчете не должны создавать артефактов в другом.
- Для ОПУ и ДДС быстрый фильтр может содержать: период, юрлица и счета, контрагенты, проекты, сделки.
- Для ОПУ дополнительно доступны **виды прибыли**; они не влияют на ДДС.

### Периоды

Период в быстром фильтре отчета бывает **плавающим** или **фиксированным**.

Плавающий период:

- сохраняет пресет, а не диапазон дат;
- пересчитывается относительно текущей даты аккаунта при каждом применении или открытии отчета;
- во вкладке **«Быстрые»** отображается текстом пресета;
- во вкладке **«Общие»** отображается рассчитанными конкретными датами.

Фиксированный период:

- сохраняет конкретные даты начала и конца;
- не пересчитывается автоматически;
- если задана только дата начала, дата конца равна текущей дате;
- если задана только дата конца, дата начала равна дате первой операции аккаунта.

Ограничения периода:

- нижняя дата: `01.01.2000`;
- верхняя дата: текущая дата;
- отчет не строится в будущее;
- если период больше 2 лет, отображение автоматически переключается в режим **«По годам»**;
- если данных нет, отчет строится с нулевыми значениями, без ошибки периода.

### Доступ, тарифы и экспорт

- Если методы `QuickFilters` возвращают `403` или access-error, это означает ограничение по роли, тарифу или правам на текущий сценарий; не трактовать такой ответ как пустой список фильтров.
- Если пользователю недоступны **Проекты** и/или **Сделки**, эти поля не отображаются в **«Общие»** и в форме создания/редактирования быстрого фильтра.
- Если ранее сохраненный фильтр содержит недоступные поля **Проекты** или **Сделки**, эти условия игнорируются, а остальные продолжают работать.
- Если ранее сохраненный фильтр применяется частично из-за недоступных полей, агент должен явно сообщить, что результат построен по частично примененному фильтру.
- Если нет модуля **«Прибыль и ОПУ (P&L)»**, в ОПУ доступен только кассовый метод; создание и применение быстрых фильтров не блокируется.
- При экспорте используется период, фактически примененный на вкладке **«Общие»**: для плавающего периода — текущие пересчитанные даты, для фиксированного — сохраненные точные даты.

## Guardrails для агента

- Не считать быстрые фильтры глобальными или общими: они персональные.
- Не считать, что созданный фильтр применяется автоматически.
- Не переносить быстрый фильтр между ОПУ и ДДС.
- Не строить отчеты в будущее.
- Не отправлять отдельный `apply`-запрос.
- Не заменять QuickFilters generic endpoint'ами операций, отчетов или справочников, если пользователь просит управлять быстрым фильтром.
- Если задача — показать данные по существующему быстрому фильтру, сначала прочитать быстрый фильтр, затем вызвать целевой data source с его условиями.
- Если задача — объяснить логику быстрых фильтров, API-запросы не нужны.
