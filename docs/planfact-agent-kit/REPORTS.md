# ПланФакт API — REPORTS для AI-агента

Этот файл покрывает `BizInfos` (v1) — cash-history по статьям, проектам и счетам, а также бизнес-логику **Платёжного календаря**.
Для **ОПУ** и **ДДС** (v2) открывать `REPORTS_V2.md`.

`README.md` — policy, router, budgets, cache.
`QUICKSTART.md` — короткий technical index.
`USE_CASES.md` — бизнес-семантика и `method + scope`.
`REPORTS_V2.md` — **ОПУ и ДДС**: `POST /api/v2/reports/opu`, `POST /api/v2/reports/dds`.
`OPERATIONS.md` — сырые операции и accrual-реконструкция.
`DOCUMENTS.md` — `invoice-documents`.
`QUICKFILTERS.md` — быстрые фильтры / сохраненные фильтры для Операций, ОПУ и ДДС.
`DASHBOARDS.md` — остатки по счетам (базовый тариф), структура платежей.
`DASHBOARDS_PRO.md` — остатки, ДДС, структура платежей (ПРО-модуль).

## Когда открывать этот файл

- нужна **cash-динамика** по статьям, проектам или контрагентам (time series);
- нужен history endpoint по счету;
- нужна **структура доходов или расходов** за период через `bizinfos` (базовый тариф, без ПРО);
- нужен **Платёжный календарь** — бизнес-логика, метрики, блоки «Доходы и расходы» и «Движение денег».

Если нужны **остатки по счетам** → `DASHBOARDS.md` (базовый) или `DASHBOARDS_PRO.md` (ПРО).
Если нужен **ДДС summary или с детализацией** → `REPORTS_V2.md` (`POST /api/v2/reports/dds`).
Если нужен **полноценный ОПУ или ДДС** — открывать `REPORTS_V2.md`.
Если вопрос про быстрые фильтры ОПУ / ДДС — открывать `QUICKFILTERS.md`.

## Use This / Not This

- `BizInfos` использовать для cash-history, `accountshistory` и базовой UI-структуры доходов/расходов.
- `REPORTS_V2.md` использовать для типового ОПУ и ДДС; не добавлять `bizinfos` или `/operations` для контрольной сверки без явной причины.
- `DASHBOARDS.md` и `DASHBOARDS_PRO.md` использовать только для одноимённых UI-разделов; не подменять ими `accountshistory`.
- `BUDGETS.md` использовать для budget plan/fact; `bizinfos.planValue` не считать планом сохранённого бюджета.

## Что покрывает файл

### `BizInfos`

- `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories`
- `GET /api/v1/bizinfos/incomeoutcomehistorybyprojects`
- `GET /api/v1/bizinfos/incomeoutcomehistorybycontragents`
- `GET /api/v1/bizinfos/accountshistory`

## Как выбирать между reports и operations

- ОПУ или ДДС с детализацией → `REPORTS_V2.md` (`POST /api/v2/reports/opu` / `dds`);
- ДДС summary без детализации → `POST /api/v2/reports/dds` с минимальным filter (детали → `REPORTS_V2.md`);
- остатки по счетам в базовых «Показателях» → `POST /api/v1/dashboards/accountbalance`; текущие остатки в «Мои счета» → `GET /api/v1/bizinfos/accountshistory`; текущие остатки в «Показателях ПРО» → `POST /api/v2/dashboardpro/accountbalance`;
- структура доходов/расходов (базовый) → `bizinfos/incomeoutcomehistorybyoperationcategories`;
- структура доходов/расходов (ПРО) → `POST /api/v2/dashboardpro/paymentstructure-*` (см. `DASHBOARDS_PRO.md`);
- cash-динамика по статьям/проектам/контрагентам → `BizInfos`;
- список сырых операций, accrual-реконструкция, реконструкция долга → `OPERATIONS.md`;
- P&L методом начисления точный → использовать v2 ОПУ с `isCalculation: true`;
- если метод явно `accrual`, `bizinfos` по проектам и статьям не считать источником истины для обычной ОПУ-аналитики.

Исключение: budget plan/fact сценарии из `BUDGETS.md`. Для сравнения с сохранённым бюджетом fact-side продукта берётся из `bizinfos`; не перепроверять этот факт отдельным OPU-запросом без явной причины.

## `incomeoutcomehistorybyoperationcategories`

- Runtime caveat: на рабочем стенде этот endpoint использовать как `GET /api/v1/bizinfos/incomeoutcomehistorybyoperationcategories` с query-параметрами (`filter.periodStartDate`, `filter.periodEndDate`, `filter.isCalculation`, `filter.ignoreMove`, `filter.ignoreAccrual`, `filter.standardPeriod`).
- Не отправлять этот endpoint как `POST` с JSON body по аналогии с `reports/*` или dashboard endpoint'ами.
- использовать для cash-динамики по статьям;
- подходит для помесячных рядов и cash-суммы по одной статье;
- если пользователь просит accrual по статье, переходить на `/operations`.

⚠️ **`planValue` здесь ≠ план бюджета.** Если пользователь сравнивает факт с уже существующим бюджетом — не использовать `planValue` из этого endpoint как плановую сторону. `planValue` здесь — плановые суммы из операций, а не план сохранённого бюджета. Источник плана бюджета → `GET /api/v1/budgets/{budgetId}`. Подробнее → `BUDGETS.md`, раздел «Критичное разграничение».

## `incomeoutcomehistorybyprojects`

- Runtime caveat: использовать как `GET` с query-параметрами, не как `POST`.
- использовать для cash-аналитики по проектам;
- не использовать для accrual;
- для cash-прибыльности проектов policy по `isUndistributed` явно называть в ответе:
  - `cash UI-aligned`
  - `cash strict project-only`

## `incomeoutcomehistorybycontragents`

- Runtime caveat: использовать как `GET` с query-параметрами, не как `POST`.
- использовать для cash-аналитики по контрагентам;
- не использовать как источник истины для отчетов по сырым операциям;
- если нужен точный список операций контрагента, переходить в `OPERATIONS.md`.

## `accountshistory`

- Runtime caveat: использовать как `GET` с query-параметрами, не как `POST`.
- Использовать для вкладки **«Мои счета»** — текущие остатки по каждому счёту.
- `factValue` — подтверждённые операции (isCommitted: true); `planValue` — запланированные (isCommitted: false).
- Для получения абсолютного остатка на дату: **накопить сумму `factValue` по всем датам локально**.
- `accountshistory` — ведущий список счетов для результата: показывать только `accountId`, которые пришли в ответе.
- Если нужны названия / валюта / компания / группа, можно обогатить эти `accountId` через `GET /api/v1/accounts`, но не добавлять счета, которых нет в `accountshistory`.
- Для общего итога в валюте пользователя суммировать накопленный `factValueInUserCurrency`; не использовать `/accounts.remainder`.
- Не использовать `POST /api/v1/dashboards/accountbalance` (для «Показатели») или `POST /api/v2/dashboardpro/accountbalance` (для «Показатели ПРО») для «Мои счета» — эти endpoints для одноимённых разделов.

## Типовые сценарии

- остатки по счетам в базовых «Показателях» → `POST /api/v1/dashboards/accountbalance` — см. `DASHBOARDS.md`
- текущие остатки в «Мои счета» → `GET /api/v1/bizinfos/accountshistory` — накопить сумму factValue локально
- текущие остатки в «Показателях ПРО» → `POST /api/v2/dashboardpro/accountbalance` — см. `DASHBOARDS_PRO.md`
- ДДС summary за период → `POST /api/v2/reports/dds` — см. `REPORTS_V2.md`
- структура доходов / расходов (базовый) → `incomeoutcomehistorybyoperationcategories`
- структура доходов / расходов (ПРО) → `POST /api/v2/dashboardpro/paymentstructure-*` — см. `DASHBOARDS_PRO.md`
- cash-динамика статьи → `incomeoutcomehistorybyoperationcategories`
- cash-рост выручки по проектам → `incomeoutcomehistorybyprojects`
- история изменения по счету → `accountshistory`

## Платёжный календарь — бизнес-логика

Платёжный календарь — **один API-запрос**, который возвращает два логических блока. Не делать отдельный запрос под каждый блок.

Endpoint уточнять в official `apidoc` только если пользователь просит именно Платёжный календарь и локальный набор не содержит достаточного route/contract для вызова. Не открывать `apidoc` из-за обычного запроса ДДС, ОПУ, cash-history или dashboard: для них маршруты описаны в `REPORTS_V2.md`, `DASHBOARDS.md` и `DASHBOARDS_PRO.md`.

Блоки **независимы по логике**: разные типы операций, разные даты, разные метрики. Это не два метода расчёта одного и того же.

### Блок 1: Доходы и расходы

Считается по статьям `Доходов` и `Расходов`. Метод расчёта выбирается пользователем: `cash` (по дате оплаты) или `accrual` (по дате начисления). Если метод не указан — уточнить.

**Типы операций:** Поступления, Выплаты, Начисления. Перемещения не участвуют.
**Начисления:** учитываются только при `isOpuCalculation == true`. В API у Начислений `operationDate == calculationDate`.

Это правило описывает продуктовую логику блока **Платёжного календаря**, а не общий ОПУ методом начисления. Для полноценного ОПУ и accrual P&L использовать `REPORTS_V2.md`.

Полная бизнес-логика кассового метода (статьи, флаг `isOpuCalculation`, нюанс дат у Начислений, факт/план) → `OPERATIONS.md`, раздел «Расчёт прибыли из операций».

#### Факт, план, непроведённые

| Слой          | Какие операции попадают                                                                                                 |
| ------------- | ----------------------------------------------------------------------------------------------------------------------- |
| Факт          | Поступления/Выплаты с подтверждённой релевантной датой + Начисления с `isOpuCalculation == true` и подтверждённой датой |
| План          | Поступления/Выплаты любого статуса + Начисления с `isOpuCalculation == true` любого статуса                             |
| Непроведённые | Операции с просроченной датой (оплаты при cash, начисления при accrual)                                                 |

«Релевантная дата»: дата оплаты при cash (`isCommitted`), дата начисления при accrual (`isCalculationCommitted`).

⚠️ **Ловушка с Начислениями в факте (cash):** у Начисления нет отдельной «даты оплаты» — `operationDate == calculationDate`. Подтверждение проверять по `isCommitted`. Одного флага `isOpuCalculation == true` недостаточно для попадания в факт: нужно ещё `isCommitted == true`.

#### Метрики

| Метрика        | Формула                                                           | Примечание                                     |
| -------------- | ----------------------------------------------------------------- | ---------------------------------------------- |
| Доходы         | ∑ Поступлений, Выплат и Начислений по статьям Доходов             |                                                |
| Расходы        | ∑ Поступлений, Выплат и Начислений по статьям Расходов            | По модулю (без минуса)                         |
| Прибыль        | Доходы − Расходы                                                  | Может быть отрицательной                       |
| Рентабельность | Прибыль / Доходы × 100%                                           | Прочерк, если Расходы > Доходов                |
| Дивиденды      | ∑ Выплат по статьям Капитала (Дивиденды + Дивиденды собственника) | По модулю; отдельная строка, не часть расходов |

Для каждой метрики есть факт, план и непроведённые (кроме рентабельности — только факт и план).

#### Фильтры

- ЮЛ (компания)
- Проекты
- Сделки (если доступен)

### Блок 2: Движение денег

Строится всегда по операциям типа Поступление / Выплата и дате оплаты. Не зависит от метода `cash / accrual`. Не спрашивать пользователя про метод.

#### Структура по видам деятельности

| Деятельность   | Статьи                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------ |
| Операционная   | Доходы/Расходы + Оборотные активы (3.1) + Краткосрочные обязательства (4.1)                |
| Инвестиционная | Внеоборотные активы (3.2)                                                                  |
| Финансовая     | Другие статьи капитала + Вложения учредителей/Дивиденды + Долгосрочные обязательства (4.2) |
| **Общий итог** | Операционный + Инвестиционный + Финансовый                                                 |

#### Факт, план, непроведённые (для каждого вида деятельности)

| Слой          | Какие операции попадают                           |
| ------------- | ------------------------------------------------- |
| Факт          | Поступления/Выплаты с подтверждённой датой оплаты |
| План          | Все Поступления/Выплаты, любой статус дат         |
| Непроведённые | Поступления/Выплаты с просроченной датой оплаты   |

#### Остатки на счетах

Формула: (Поступления + Перемещения-зачисления) − (Выплаты + Перемещения-списания) по всем статьям учёта.

| Вариант остатка | Вчера и ранее  | Сегодня        | Будущее        |
| --------------- | -------------- | -------------- | -------------- |
| Вчера и ранее   | Подтверждённые | Не учитываются | Не учитываются |
| Сегодня (факт)  | Подтверждённые | Подтверждённые | Не учитываются |
| Сегодня (план)  | Подтверждённые | Все            | Не учитываются |
| Будущее         | Подтверждённые | Все            | Все            |

«Подтверждённые» = операции с подтверждённой датой оплаты. «Все» = операции с любым статусом дат.

### Правила для агента

- Блок «Доходы и расходы» и блок «Движение денег» — разные отчёты. Не путать и не объединять.
- `cash` / `accrual` — метод расчёта блока «Доходы и расходы». Блок «Движение денег» всегда по дате оплаты.
- Прибыль ≠ Денежный поток. Прибыль считается по Доходам и Расходам. Денежный поток — по всем денежным движениям.
- Дивиденды — отдельная строка в блоке «Доходы и расходы», не входят в Расходы.
- Перемещения участвуют только в остатках на счетах, не в блоке «Доходы и расходы» и не в строках видов деятельности.
- Непроведённые в блоке «Доходы и расходы» определяются по просроченной дате, выбранной методом расчёта (cash → дата оплаты, accrual → дата начисления). В блоке «Движение денег» — всегда по дате оплаты.
