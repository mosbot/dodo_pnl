# ПланФакт API — LOOKUPS для AI-агента

Этот файл покрывает:

- справочники и `filter-first lookup` (Accounts, ContrAgents, Projects, Companies, OperationCategories, Products, ImportLogs);
- финансовые агрегаты по контрагентам и группам КА (доход / расход / прибыль / дебиторка / кредиторка) через готовые endpoint'ы `contragents/calculated/{id}`, `contragents/groups/calculated`, `contragents/calculations`, `contragents/summary`, `contragents/{id}/additional`;
- финансовые агрегаты по проектам (план из бюджета + факт, дебиторка / кредиторка) через `projects/calculations`, `projects/{id}/summary`.

`README.md` — policy, router, budgets, cache.  
`QUICKSTART.md` — короткий technical index.  
`OPERATIONS.md` — если после lookup нужен список операций или реконструкция долга из сырых операций (нестандартный срез).  
`DEALS.md` — если после lookup нужна сделка.

## Когда открывать

- нужен `accountId`, `contrAgentId`, `projectId`, `companyId`, `operationCategoryId`, `productId`, `productGroupId`, `productUnitId` или `importLogId`;
- нужно найти сущность по названию, ИНН, `externalId`, активности, группе или валюте;
- пользователь дал узкий признак, и надо не грузить весь справочник;
- нужно архивировать, активировать, удалить или объединить сущность справочника;
- нужна **дебиторка / кредиторка / прибыль / доход / расход по контрагенту или списку контрагентов** через готовый endpoint (а не ручной реконструкцией из операций);
- нужен **план из бюджета + факт по проектам** в одном ответе (`projects/calculations`) или **дебиторка / кредиторка по проекту** без загрузки всех операций (`projects/{id}/summary`).

## Быстрый выбор слоя

- `GET /contragents`, `GET /projects`, `GET /accounts`, `GET /operationcategories` — только для ID lookup и справочных атрибутов.
- `GET /api/v1/contragents/calculated/{id}` и `GET /api/v1/contragents/{id}/additional` — стандартный слой для финансов по одному КА.
- `GET /api/v1/projects/calculations` и `GET /api/v1/projects/{id}/summary` — стандартный слой для проектных агрегатов.
- Не добавлять `/operations` или dashboard endpoint только «для проверки» lookup- или aggregate-ответа.

## Общие правила `filter-first`

Канонические правила — в `README.md`, раздел «Справочники и filter-first». Здесь краткий справочник для этого файла.

- если пользователь уже дал название, ИНН, `externalId`, ID, активность, статус, тип, группу, валюту или период — переноси это в server-side фильтры; не делай full dump;
- при lookup одной сущности — сначала фильтрованный endpoint, а не весь справочник;
- для refresh справочников использовать `filter.changesFromDate`, если он доступен;
- если ID уже есть в кэше — повторный запрос справочника не нужен.

**Когда полный список допустим:** пользователь явно просит весь список; server-side фильтра для нужного признака нет в contract; узкий запрос уже доказанно не сужает выдачу.

**Когда полный список недопустим:** поиск одной сущности по имени / ИНН / ID; проверка «существует ли» — справочник вообще не нужен; ID уже в кэше.

## Write-ограничения справочников

Перед любым write-действием по справочнику агент должен проверить:

- активна ли сущность;
- нет ли связанных объектов, которые запрещают действие;
- не повлечёт ли действие каскадные изменения для зависимых сущностей.

Если API возвращает ошибку, агент должен показать пользователю:

- **что мешает действию**;
- **какие сущности нужно сначала отвязать / удалить / активировать**;
- **что делать дальше**.

Не скрывать и не переписывать смысл ошибки.

## Ограничения доступа и partial access

- Если lookup- или summary-endpoint вернул `403` / access-error, считать, что текущая сущность или сценарий недоступны пользователю по правам, роли или тарифу.
- Не трактовать такой ответ как доказательство, что сущность не существует.
- Если endpoint вернул только доступную пользователю часть сущностей или данных, использовать ее как актуальный access-context, но предупреждать о возможной неполноте результата.

## Accounts

- endpoint: `GET /api/v1/accounts`
- использовать только когда нужен справочник счетов или `accountId`
- локально известные server-side фильтры (если нужен спорный или редкий параметр — сверить `apidoc`):
  - `filter.title`
  - `filter.titleOrCompanyTitle`
  - `filter.currencyCode`
  - `filter.externalId`
  - `filter.active`
- `remainder` не использовать как источник истины по остаткам.
- Для остатков в **«Мои счета»** — использовать `GET /api/v1/bizinfos/accountshistory` (накопить factValue локально).
- Для остатков в **«Показателях ПРО»** — использовать `POST /api/v2/dashboardpro/accountbalance`.
- Для остатков в базовом разделе **«Показатели»** — использовать `POST /api/v1/dashboards/accountbalance`.

### Архивация счёта

- Счёт переводится в архив.
- Если к счёту подключены интеграции — они отключаются.

### Активация счёта

- Сначала проверить, активно ли связанное ЮЛ.
- Если ЮЛ архивно — сначала активировать ЮЛ, потом счёт.

### Удаление счёта

Запрещено, если к счёту привязаны:

- операции;
- правила импорта;
- выставленные счета.

Special case: если привязаны только правила импорта (без операций и выставленных счетов) — возможно удаление счёта вместе с правилами.

## Группы счетов

### Удаление группы

- Счета группы не удаляются — перемещаются в группу **«Нераспределенные»**.

### Архивация группы

- Группа архивируется.
- Все активные счета группы архивируются.
- Интеграции отключаются (кроме маркетплейсов).

### Активация группы

- Группа активируется.
- Все архивные счета группы активируются вместе с ней.

## ContrAgents

### Lookup (`GET /api/v1/contragents`)

- endpoint: `GET /api/v1/contragents`
- использовать только для получения ID / названия / ИНН / активности КА — справочник, без финансов;
- локально известные server-side фильтры (если нужен спорный или редкий параметр — сверить `apidoc`):
  - `filter.title`
  - `filter.titleOrContrAgentInn`
  - `filter.externalId`
  - `filter.active`
  - `filter.contrAgentId`
  - `filter.contrAgentGroupId`
- если имя дано в кавычках, сначала пробовать точное название;
- не грузить весь `/contragents` по привычке.

### Удаление КА

Запрещено, если КА привязан к операциям, сделкам или выставленным счетам. При успешном удалении: КА и все его правила удаляются.

### Объединение КА

- Выбирается главный КА; остальные объединяются в него.
- КПП и номера счетов переносятся без дублей.
- Операции, сделки и выставленные счета переходят к главному КА.

### Финансовые агрегаты по КА

Для доходов / расходов / прибыли / дебиторки / кредиторки по контрагенту **использовать готовый точечный endpoint** `GET /api/v1/contragents/calculated/{contrAgentId}`, а не реконструировать из `operations/list`. Для типа долга использовать `GET /api/v1/contragents/{contrAgentId}/additional`. Ручная реконструкция допустима только если готовый endpoint не покрывает фильтр (см. `Когда ручная реконструкция` ниже).

#### `CalculationMethod` — три метода

Финансовые endpoint'ы КА принимают параметр `calculationMethod` с тремя значениями:

| Enum              | Что учитывает                                                                         | Для чего                                         |
| ----------------- | ------------------------------------------------------------------------------------- | ------------------------------------------------ |
| `Cashflow` = 1    | Поступления и Выплаты с `isCommitted = true`. Начисления **не** входят.               | Денежный поток, ДДС                              |
| `Cash` = 2        | Поступления и Выплаты с `isCommitted = true` + Начисления с `isOpuCalculation = true` | Прибыль кассовым методом, ОПУ cash               |
| `Calculation` = 3 | Операции с `isCalculationCommitted = true`.                                           | Прибыль методом начисления, дебиторка/кредиторка |

⚠️ Это **не то же самое**, что двухрежимный `method: cash / accrual` в `POST /api/v2/reports/opu`. В контексте КА три значения, и `Cashflow` ≠ `Cash`: `Cashflow` игнорирует Начисления вовсе, `Cash` подмешивает их по флагу `isOpuCalculation`.

Правило выбора метода:

- пользователь спрашивает **денежный поток / оборот / движение денег** по КА → `Cashflow`;
- пользователь спрашивает **дебиторку / кредиторку** (as-of долги) → `Calculation`;
- пользователь спрашивает **прибыль по КА** без уточнения → задать один уточняющий вопрос (`Cash` или `Calculation`), см. `README.md`, раздел «Уточняющие вопросы»;
- метод явно задан пользователем (cash / начисление / движение денег) — маппить на enum без уточнения.

#### Список КА с финансами

Для получения финансов по списку КА:

1. Получить ограниченный список КА через filter-first lookup (`GET /api/v1/contragents` или `POST /api/v1/contragents/list`, если нужна группировка).
2. Для каждого выбранного КА запросить `GET /api/v1/contragents/calculated/{contrAgentId}`.
3. Запросы можно выполнять параллельно, если список уже ограничен.

Не запускать N+1 по большому списку молча. Если после фильтрации найдено больше `20` КА, сначала предупредить пользователя о количестве точечных запросов и предложить сузить фильтр или подтвердить расчёт.

Этот маршрут считать `expensive path`: запускать только после filter-first сужения списка.

#### Контрагенты: list vs calculated

- `POST /api/v1/contragents/list` — список КА + группировка (`countByGroups`, `contractorsIdsByGroups`). Не финансовый источник.
- `POST /api/v1/contragents/groups/calculated` — финансы, агрегированные по группам КА.
- Если пользователь просит расшифровку до операций — идти в `OPERATIONS.md` и `POST /api/v1/operations/list`.

#### `GET /api/v1/contragents/calculated/{contrAgentId}` — один КА с финансами

- query: `dateStart`, `dateEnd`, `calculationMethod`;
- ответ: полный набор (`allCredit / pastCredit / allDebit / pastDebit / income / outcome / profit / operationsCount`);
- использовать для карточки КА и drill-down из списка.

#### `GET /api/v1/contragents/calculations` — лёгкий агрегат по списку КА

- query: `contrAgentId[]`;
- использовать только как лёгкий расчёт по известным КА;
- не использовать для периода, метода расчёта, дебиторки/кредиторки.

Если нужен период, `calculationMethod`, долги — использовать `GET /api/v1/contragents/calculated/{contrAgentId}` или `GET /api/v1/contragents/{contrAgentId}/additional`.

#### `GET /api/v1/contragents/summary` — список + сводка

- ответ: `items[]` как у `/contragents` + поля `totalIncome`, `totalOutcome`;
- промежуточный вес между `/contragents` (справочник) и `/contragents/calculated` (полные финансы).

#### `GET /api/v1/contragents/{contrAgentId}/additional` — тип долга и метод

- query: `calculationMethod`;
- ответ: `cashDebt`, `nonCashDebt`, `calculationMethod`;
- использовать для уточнения типа долга после `/calculated/{id}`.

#### Таблица выбора endpoint'а

| Что нужно                                                | Endpoint                              |
| -------------------------------------------------------- | ------------------------------------- |
| Только справочник (id, название)                         | `GET /contragents`                    |
| Фильтрованный список КА + count/ids по группам           | `POST /contragents/list`              |
| Список + простой доход/расход                            | `GET /contragents/summary`            |
| Список КА + полные финансы                               | filter-first список КА + серия `GET /contragents/calculated/{id}` |
| Финансы по группам КА                                    | `POST /contragents/groups/calculated` |
| Один КА + полные финансы, включая дебиторку и кредиторку | `GET /contragents/calculated/{id}`    |
| Список + только `income / outcome / profit` (без долгов) | `GET /contragents/calculations`       |
| Доп. инфо по долгу (тип долга, метод расчёта)            | `GET /contragents/{id}/additional`    |

#### As-of date и подтверждение

- бизнес-смысл границы периода — as-of date включительно: операция с датой на эту дату попадает в период, со следующей даты — нет;
- для `GET /api/v1/contragents/calculated/{contrAgentId}` query-параметр этой границы называется `dateEnd`; не подставлять в payload имя `operationDateEnd`, если contract endpoint'а его не содержит;
- каждый метод учитывает только подтверждённые операции (по своему флагу: `isCommitted` или `isCalculationCommitted`);
- неподтверждённые операции и операции без даты в расчёт не входят ни в одном методе.

#### Когда ручная реконструкция

Идти в `OPERATIONS.md` и считать из `operations/list` только если **готовый endpoint не покрывает запрос**:

- нужен нестандартный фильтр результата (например, «КА, у которых дебиторка > 100к»): сначала взять `/calculated`, затем отфильтровать локально;
- нужен нестандартный grouping, которого нет в contract;
- нужен срез по группам КА — сначала `POST /api/v1/contragents/groups/calculated`; ручная группировка КА + операции допустима только если endpoint не покрывает фильтр или нужна расшифровка отдельных операций;
- `calculated` вернул ошибку или edge case, не покрытый doc;
- в ответе пометить, что данные посчитаны вручную из `operations/list`, а не взяты из готового endpoint.

## Projects

### Lookup (`GET /api/v1/projects`)

- endpoint: `GET /api/v1/projects`
- использовать только для ID / названия / активности / группы / признака завершённости — справочник, без финансов;
- локально известные server-side фильтры (если нужен спорный или редкий параметр — сверить `apidoc`):
  - `filter.title`
  - `filter.externalId`
  - `filter.active`
  - `filter.projectGroupId`
- для завершенности проекта использовать `closed`, а не `active`;
- `active` не использовать как основной признак завершенности.

### Финансовые агрегаты по проектам

Для плана из бюджета + факта по проекту, а также для дебиторки / кредиторки по проекту **использовать готовые endpoint'ы** (`projects/calculations`, `projects/{id}/summary`), а не реконструировать из `operations/list` или пары `BUDGETS.md` + `bizinfos`.

#### `CalculationMethod` — три метода (⚠️ отличаются от КА)

У проектов параметр `calculationMethod` имеет **свой набор значений**, не совпадающий с `contragents/calculated*`:

| Enum                              | Что учитывает                                                       | Для чего                   |
| --------------------------------- | ------------------------------------------------------------------- | -------------------------- |
| `Cashflow`                        | Поступления и Выплаты с `isCommitted = true`. Начисления не входят. | Денежный поток по проекту  |
| `IncomeExpensesByOperationDate`   | Подтверждённые операции по дате операции (cash)                     | Прибыль по проекту cash    |
| `IncomeExpensesByCalculationDate` | Операции с `isCalculationCommitted = true` (accrual)                | Прибыль по проекту accrual |

⚠️ **Не путать с КА**: у контрагентов `calculationMethod` использует enum `Cashflow / Cash / Calculation`, у проектов — `Cashflow / IncomeExpensesByOperationDate / IncomeExpensesByCalculationDate`. В payload подставлять строковое имя enum из контекста сущности, не копировать между КА и проектами.

#### `GET /api/v1/projects/calculations` — план из бюджета + факт

- query: `projectId[]`, `calculationMethod`;
- ответ: массив по одному элементу на проект с полями `incomePlan`, `outcomePlan`, `incomeFact`, `outcomeFact`, `entityStatus` (`New / InProgress / Done`);
- `incomePlan` / `outcomePlan` — плановая сторона для plan/fact по проектам без детализации по статьям;
- `incomeFact` / `outcomeFact` — факт по выбранному `calculationMethod`;
- период и стандартные фильтры в этот endpoint не передавать, если они не указаны в актуальном `apidoc`;
- `calculationMethod` для агента обязателен: не полагаться на server default.

⚠️ **Единственный короткий endpoint для плана + факта по проектам в одном ответе.** Ни `POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body, ни `bizinfos/incomeoutcomehistorybyprojects` плана **из бюджета** не возвращают. У `bizinfos` есть `planValue`, но это плановые суммы из операций, а не план сохранённого бюджета (см. `REPORTS.md`, раздел `incomeoutcomehistorybyoperationcategories`). Если нужна детализация строк бюджета по статьям — читать `GET /api/v1/budgets/{budgetId}`.

#### `GET /api/v1/projects/{projectId}/summary` — карточка проекта с долгом

- query: стандартные фильтры;
- ответ: `projectId`, `title`, `closed`, `invoiceBalance` (дебиторка), `invoiceSupplyBalance` (кредиторка) + другие агрегаты;
- использовать для карточки проекта, drill-down из списка, быстрого ответа на «дебиторка / кредиторка по проекту X» без загрузки всех операций.

Аналог `GET /contragents/calculated/{id}` для КА.

#### Таблица выбора endpoint'а

| Что нужно                                                                   | Endpoint                                                                                |
| --------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| Только справочник (id, название, активность)                                | `GET /projects`                                                                         |
| План + факт по проектам (план из бюджета, факт cash/accrual)                | `GET /projects/calculations`                                                            |
| Один проект + дебиторка / кредиторка                                        | `GET /projects/{id}/summary`                                                            |
| ОПУ по проектам с детализацией по статьям, accrual или cash (план не нужен) | `POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body (см. `REPORTS_V2.md`) |
| Cash-динамика по проектам по подпериодам (time-series)                      | `GET /api/v1/bizinfos/incomeoutcomehistorybyprojects` (см. `REPORTS.md`)                |

#### Соотношение с `reports/opu` по проектам

| Параметр                   | `projects/calculations`     | `reports/opu` с `reportGenMethod: "Projects"` |
| -------------------------- | --------------------------- | --------------------------------------------- |
| Метод факта                | `Cashflow` / cash / accrual | cash или accrual (через `isCalculation`)      |
| План из бюджета            | ✅ есть                      | ❌ нет                                         |
| Детализация по статьям     | ❌ нет                       | ✅ есть                                        |
| Разрез по сделкам, группам | ❌ нет                       | ✅ есть (через `reportGenMethod`)              |

Правило выбора:

- нужен **план + факт** по проектам → `projects/calculations`;
- нужен **accrual-ОПУ с разбивкой по статьям** по проектам → `POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body;
- нужна **cash time-series** по проектам → `bizinfos/incomeoutcomehistorybyprojects`;
- нужна **дебиторка/кредиторка** по одному проекту → `projects/{id}/summary`.

#### Запрет: company-level ≠ сумма строк

- **не суммировать** строки `projects/calculations` в company-wide итог и не выдавать эту сумму как «прибыль компании» или «выручка компании»;
- project-level scope не равен company-level по определению: часть операций может быть без проекта (`Не выбран`), а распределения по проектам могут быть частичными;
- для company-level итогов идти в `POST /api/v2/reports/opu` без `reportGenMethod=Projects` или в `POST /api/v2/reports/dds`.

#### Когда ручная реконструкция

Идти в `OPERATIONS.md` и считать из `operations/list` только если готовый endpoint не покрывает запрос:

- нужен нестандартный фильтр результата (например, «проекты, у которых выполнение плана < 50%») — сначала взять `/projects/calculations`, затем отфильтровать локально;
- нужен нестандартный grouping, которого нет в contract;
- `projects/calculations` / `{id}/summary` вернули ошибку или edge case, не покрытый doc;
- в ответе пометить, что данные посчитаны вручную.

## Companies

  - `filter.title`
  - `filter.companyInn`
  - `filter.externalId`
  - `filter.active`
  - `filter.companyId`
- это справочник, не финансовая аналитика.

Запрещено, если к ЮЛ привязаны:

- правила импорта для депозитных счетов.

## OperationCategories

- endpoint: `GET /api/v1/operationcategories`
- локально известные server-side фильтры (если нужен спорный или редкий параметр — сверить `apidoc`):
  - `filter.title`
  - `filter.operationCategoryType`
  - `filter.activityType`
  - `filter.parentOperationCategoryId`
  - `filter.externalId`
- не грузить всё дерево статей без необходимости;
- если нужна готовая группировка статей по типам или поиск нераспределённых статей — использовать `GET /api/v1/operationcategories/group-list`; не грузить и не группировать весь справочник вручную без необходимости;
- для понимания иерархии, типа (денежная/неденежная), ограничений и семантики статей (что входит в выручку, расходы, P&L) — открыть `OPERATION_CATEGORIES.md`.

### Удаление статьи

Запрещено, если статья есть хотя бы в одном бюджете. Показать пользователю, что статью нужно сначала отвязать от бюджетов.

## Products

Products — справочник товаров и услуг. Использовать только когда нужен `productId`, название, тип (`Product` / `Service`), группа, цена из карточки, единица измерения или `externalId` для сделки, счёта или write-payload.

Не использовать Products как источник финансовой аналитики:

- выручка, прибыль, расходы, продажи по периодам → `REPORTS.md` / `REPORTS_V2.md` / `DEALS.md`;
- операции по товару/услуге → сначала проверить, есть ли нужный срез в `DEALS.md` или `OPERATIONS.md`;
- остатки, складской учёт и движение запасов не выводить из `Products`.

### Endpoint routing

| Что нужно                                             | Endpoint                                       |
| ----------------------------------------------------- | ---------------------------------------------- |
| Найти товар/услугу по названию, типу или `externalId` | `GET /api/v1/products`                         |
| Получить одну карточку по ID                          | `GET /api/v1/products/{productId}`             |
| Только количество товаров/услуг по фильтру            | `GET /api/v1/products/count`                   |
| Найти группу товаров/услуг                            | `GET /api/v1/products/groups`                  |
| Только количество групп                               | `GET /api/v1/products/groups/count`            |
| Получить одну группу по ID                            | `GET /api/v1/products/groups/{productGroupId}` |
| Найти единицу измерения                               | `GET /api/v1/products/productUnits`            |

### Filter-first

- `filter.searchString` по товарам/услугам ищет по названию, группе и комментарию;
- `filter.searchString` по группам и единицам измерения ищет по названию;
- если пользователь дал точное название, использовать `filter.needExactMatch=true`;
- для типа использовать `filter.types`: `Product = 1`, `Service = 2`;
- для внешнего ID использовать `filter.externalId`;
- для refresh справочника использовать `filter.changesFromDate`;
- не загружать весь `/products`, если нужен один товар, услуга, группа или count.

### Count endpoints

- `GET /api/v1/products/count` и `GET /api/v1/products/groups/count` возвращают только счётчик;
- не использовать count-endpoint, если нужны ID, названия, цены, группы или расшифровка;
- для вопроса «сколько товаров/услуг/групп подходит под фильтр» использовать count, а не загружать весь список.

### Public write methods

Использовать только если пользователь явно просит создать, изменить или удалить товар/услугу либо группу. Не использовать write-методы для lookup, проверки существования или финансовой аналитики.

| Что нужно                     | Endpoint                                          |
| ----------------------------- | ------------------------------------------------- |
| Создать товар/услугу          | `POST /api/v1/products`                           |
| Изменить товар/услугу         | `PUT /api/v1/products/{productId}`                |
| Удалить товар/услугу          | `DELETE /api/v1/products/{productId}`             |
| Создать группу товаров/услуг  | `POST /api/v1/products/groups`                    |
| Изменить группу товаров/услуг | `PUT /api/v1/products/groups/{productGroupId}`    |
| Удалить группу товаров/услуг  | `DELETE /api/v1/products/groups/{productGroupId}` |

Перед write-действием подтвердить намерение пользователя и брать payload из публичного `apidoc`, не додумывать поля.

Минимальные бизнес-ограничения:

- услуга не может иметь `SKU`;
- у услуги название уникально внутри типа;
- цена не может быть отрицательной;
- если группа товара/услуги не указана, используется служебная нераспределённая группа своего типа;
- удалить товар/услугу нельзя, если он используется в выставленном счёте или сделках;
- удалить группу нельзя, если в ней есть товары/услуги.

## ImportLogs

ImportLogs — журнал завершённых импортов операций. Использовать, когда пользователь просит список загрузок или когда нужен `importLogId`, чтобы затем отфильтровать операции по конкретной загрузке.

- endpoint: `GET /api/v1/importlogs`;
- параметры: только `paging.offset`, `paging.limit`;
- ответ: `items[]` + общий `totalCount`;
- item: `importLogId`, `startDate`, `accountTitle`, `importSource`;
- сервер возвращает только завершённые импорты по доступным пользователю счетам;
- сортировка: новые импорты первыми (`startDate desc`).

Правила:

- для «покажи последние загрузки / логи импорта» использовать `GET /api/v1/importlogs` с осознанным `paging.limit`;
- для «сколько логов импорта» брать `totalCount` из ответа, не грузить все страницы;
- для «покажи операции из загрузки» сначала получить `importLogId`, затем идти в `OPERATIONS.md` и фильтровать операции по `filter.importLogId`;
- не использовать `ImportLogs` как источник сумм, выручки, расходов, остатков или статуса текущей загрузки;
- не передавать фильтры по дате, счёту или источнику в `GET /api/v1/importlogs`: у метода есть только paging.

## Типовые lookup-маршруты

### Операции по контрагенту

Пример: «Покажи операции по клиенту Иванов»

1. `GET /api/v1/contragents?filter.title=Иванов` — получить `contrAgentId`
2. Если ID уже известен из предыдущего запроса или кэша — шаг 1 пропустить
3. Если пользователь дал период или его можно разумно сузить — сначала `POST /api/v1/operations/list` с `contrAgentId` и датами
4. Если в `operations/list` нет нужного server-side фильтра, фильтр игнорируется сервером или уже есть актуальный кэш `/operations` — перейти к `OPERATIONS.md` и локальной фильтрации

Не делать: сначала загрузить весь `/operations`, потом искать по имени контрагента в тексте.

### Операции по проекту

- сначала найти проект через `/projects`;
- затем идти в `OPERATIONS.md`.

### Финансовый срез по проекту

- если пользователь просит **план + факт** по проекту/проектам → сначала `GET /projects/calculations` (см. раздел «Projects → Финансовые агрегаты»), не собирать связку BUDGETS + bizinfos;
- если пользователь просит **дебиторку / кредиторку** по проекту → сначала `GET /projects/{id}/summary`, не реконструировать из операций;
- если пользователь просит **accrual-ОПУ с разбивкой по статьям** по проекту → `POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body (см. `REPORTS_V2.md`);
- если пользователь просит **cash-динамику по подпериодам** → `GET /api/v1/bizinfos/incomeoutcomehistorybyprojects` (см. `REPORTS.md`).

### История по счету

- сначала найти счет через `/accounts`;
- затем идти в `REPORTS.md` или `OPERATIONS.md`.

### Мои счета с текущими остатками

- для списка счетов с текущими остатками и общим итогом использовать `GET /api/v1/bizinfos/accountshistory` (накопить `factValue` локально);
- `accountshistory` — ведущий источник строк результата: показывать только `accountId`, которые пришли в этом ответе;
- `GET /api/v1/accounts` использовать только как lookup для уже полученных `accountId`, если нужен справочный атрибут счёта (название, валюта, активность, группа), которого нет в ответе `accountshistory`;
- не добавлять в таблицу счета из `/accounts`, которых нет в `accountshistory`, даже если они active;
- не суммировать `remainder` из `/accounts` как финансовый итог.

## Known caveats

- не использовать `/accounts` как preflight;
- не использовать `/accounts.remainder` как источник текущих остатков; для «Мои счета» использовать `GET /api/v1/bizinfos/accountshistory`, для «Показателей ПРО» — `POST /api/v2/dashboardpro/accountbalance`, для базовых «Показателей» — `POST /api/v1/dashboards/accountbalance`;
- не грузить полный справочник без необходимости;
- справочный endpoint (`GET /contragents`, `GET /projects`, `GET /accounts`, `GET /companies`, `GET /operationcategories`) не использовать как источник финансовых итогов — для сумм по контрагентам идти в `contragents/calculated/{id}` / `calculations` / `summary` (см. раздел «ContrAgents → Финансовые агрегаты»); для сумм / плана / долга по проектам — `projects/calculations` и `projects/{id}/summary` (см. раздел «Projects → Финансовые агрегаты»); для сумм по статьям — в `REPORTS.md` / `REPORTS_V2.md` / `DASHBOARDS.md`;
- прежде чем реконструировать дебиторку / кредиторку / прибыль по КА из `operations/list`, использовать `GET /contragents/calculated/{id}` — ручной расчёт допустим только при нестандартном фильтре или ошибке готового endpoint;
- прежде чем собирать связку `BUDGETS.md + bizinfos` для «план + факт по проектам», проверить `projects/calculations` — это готовый endpoint для этого кейса в одном запросе;
- `CalculationMethod` у проектов и у КА — разные enum'ы (`Cash` у КА ≠ `IncomeExpensesByOperationDate` у проектов по имени, но семантически обе про cash); при сборке payload подставлять имя enum из контекста сущности, не копировать между ContrAgents и Projects.
