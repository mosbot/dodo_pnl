# ПланФакт API v2 — Отчёты ОПУ и ДДС для AI-агента

Этот файл описывает новые endpoint'ы `/api/v2/reports/` для построения отчётов ОПУ и ДДС.

`README.md` — policy, router.
`REPORTS.md` — `BizInfos` (v1): cash-history по статьям/проектам/счетам, Платёжный календарь.
`USE_CASES.md` — бизнес-семантика, method + scope.
`OPERATION_CATEGORIES.md` — иерархия статей (нужна для интерпретации ответа).
`OPERATIONS.md` — если нужен точный accrual-расчёт по сырым операциям.
`QUICKFILTERS.md` — быстрые фильтры ОПУ / ДДС и публичные методы `QuickFilters`.

## Когда открывать

- пользователь просит ОПУ, отчёт о прибылях и убытках, P&L;
- пользователь просит ДДС, отчёт о движении денежных средств, денежный поток по деятельности;
- нужен отчёт с разбивкой по статьям, проектам или сделкам;
- нужна валовая прибыль, операционная прибыль, EBITDA, EBIT, EBT;
- нужен факт + план с детализацией по периодам.
- нужен быстрый фильтр или сохраненный фильтр в ОПУ / ДДС.

## Technical note для `POST`-запросов

Для `POST /api/v2/reports/*` в shell-safe режиме предпочитай payload-файл:

- `curl ... --data-binary @payload.json`
- не собирать длинный JSON body через inline `--data '{...}'`, если команду формирует агент;
- если API отвечает `Unexpected character encountered while parsing value...`, сначала считать это признаком сломанного JSON-body или quoting error, а не проблемой самого report endpoint.

## Ограничения доступа и трактовка ответа

- Если `POST /api/v2/reports/opu` или `POST /api/v2/reports/dds` возвращает `403` или другой access-error, считать, что текущий штатный отчетный сценарий недоступен по тарифу, правам, методу или scope.
- Если API вернул `200`, `isSuccess = true`, но `data` пустой, считать, что по выбранным параметрам отчетных данных нет.
- Если для ОПУ недоступен метод начисления, а пользователь просит именно начисление, объяснить ограничение и предложить кассовый метод как отдельный доступный сценарий.

## Minimal request shape

Для типовых read-only ОПУ / ДДС не открывать `apidoc`, если хватает полей ниже и known caveats этого файла.

**OPU:**

- `periodStartDate`
- `periodEndDate`
- `isCalculation`
- `reportGenMethod`
- `isPeriodDetail` + `standardPeriod`, если `reportGenMethod = OperationCategory` или нужна динамика

**DDS:**

- `periodStartDate`
- `periodEndDate`
- `reportGenMethod`
- `isPeriodDetail` + `standardPeriod`, если `reportGenMethod = OperationCategory` или нужна динамика
- `isCalculation` не передавать

Правила:

- Для `OperationCategory` итог за период всё равно запрашивать с `isPeriodDetail: true` + `standardPeriod` и агрегировать локально. Это один API-вызов, не повод делать второй.
- Для `Projects` и `Deals` итог за период можно запрашивать с `isPeriodDetail: false`.
- `includeTrendData` и detail-trends не включать без явного запроса тренда / спарклайна.
- Большие JSON-примеры не дублировать в этом файле; при write или редком параметре использовать `apidoc` как contract-reference.

## Глоссарий

| Термин               | Определение                                                                                                                 |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| Метод начисления     | Операции попадают в ОПУ по **дате начисления** (`calculationDate`) — дата отгрузки/поставки/начисления расходов, дивидендов |
| Кассовый метод       | Операции попадают в ОПУ по **дате оплаты** (`operationDate`) — дата фактического движения денег                             |
| `cash`               | Способ расчета факта по дате оплаты; это не синоним `ДДС`                                                                   |
| Денежный поток       | Разница между Поступлениями и Выплатами за период                                                                           |
| Операционный поток   | Денежный поток по операционной деятельности (статьи Доходов/Расходов, Оборотные активы, Краткосрочные обязательства)        |
| Инвестиционный поток | Денежный поток по инвестиционной деятельности (статьи Внеоборотных активов)                                                 |
| Финансовый поток     | Денежный поток по привлечению/возврату финансирования (Капитал кроме дивидендов, Долгосрочные обязательства)                |
| Прибыль              | Доходы за минусом Расходов                                                                                                  |

## POST /api/v2/reports/opu — ОПУ

### Назначение

Полноценный отчёт о прибылях и убытках. Поддерживает кассовый метод и метод начисления, разбивку по статьям / проектам / сделкам, расчёт нескольких видов прибыли.

### Ключевые параметры запроса

> ⚠️ Поля дат (`periodStartDate`, `periodEndDate`) — на рабочем стенде передавать в формате **`YYYY-MM-DD`**, несмотря на то что apidoc указывает `date-time`. Формат с временем вызывает ошибку `invalid`. Подробнее → `QUICKSTART.md`, раздел «Формат дат».

| Параметр            | Тип     | Значения                                    | Описание                                                               |
| ------------------- | ------- | ------------------------------------------- | ---------------------------------------------------------------------- |
| `isCalculation`     | boolean | `false` = кассовый; `true` = начисление     | **Метод расчёта. Обязательно уточнить у пользователя, если не указан** |
| `reportGenMethod`   | string  | `OperationCategory` / `Projects` / `Deals`  | Способ построения: по статьям (дефолт), по проектам, по сделкам        |
| `isPeriodDetail`    | boolean | true/false                                  | Разбить на подпериоды                                                  |
| `standardPeriod`    | string  | `Day\|Week\|Month\|Quarter\|HalfYear\|Year` | Гранулярность при `isPeriodDetail=true`                                |
| `periodStartDate`   | string  | `YYYY-MM-DD` ⚠️                             | Начало периода                                                         |
| `periodEndDate`     | string  | `YYYY-MM-DD` ⚠️                             | Конец периода                                                          |
| `userCurrencyCode`  | string  | `RUB\|USD\|EUR\|...`                        | Валюта ответа                                                          |
| `accountId`         | array   | ID счетов                                   | Фильтр по счетам                                                       |
| `contrAgentId`      | array   | ID контрагентов                             | Фильтр по контрагентам                                                 |
| `projectId`         | array   | ID проектов                                 | Фильтр по проектам                                                     |
| `sellingDealId`     | array   | ID сделок продаж                            | Фильтр по сделкам продаж                                               |
| `purchaseDealId`    | array   | ID сделок закупок                           | Фильтр по сделкам закупок                                              |
| `isGrossProfit`     | boolean | true/false                                  | Считать валовую прибыль                                                |
| `isOperatingProfit` | boolean | true/false                                  | Считать операционную прибыль                                           |
| `isEbitda`          | boolean | true/false                                  | Считать EBITDA                                                         |
| `isEbit`            | boolean | true/false                                  | Считать EBIT                                                           |
| `isEbt`             | boolean | true/false                                  | Считать EBT (прибыль до налогов)                                       |
| `withOther`         | boolean | true/false                                  | Показывать прочие проекты/сделки под тегом «Остальные»                 |
| `includeTrendData`  | boolean | true/false                                  | Включить данные тренда (для спарклайнов)                               |
| `aggregationMode`   | string  | `AutoAggregate\|KeepOriginal`               | Режим агрегации                                                        |

### Формулы ОПУ

Перемещения **не участвуют** ни в одном методе ОПУ.

**Метод начисления (`isCalculation: true`):**

Типы операций: Поступления, Выплаты, Начисления, **Отгрузки, Поставки**.

- Доходы = ∑ Поступлений, Выплат, Начислений, Отгрузок и Поставок по статьям **Доходов** с подтверждённой датой начисления (`isCalculationCommitted == true`)
- Расходы = ∑ тех же типов по статьям **Расходов** с подтверждённой датой начисления

**Кассовый метод (`isCalculation: false`):**

Типы операций: Поступления, Выплаты, Начисления. **Отгрузки и Поставки не участвуют.**

- Факт: операции с подтверждённой датой оплаты (`isCommitted == true`)
- План: те же типы операций, статус подтверждения не важен

Полная бизнес-логика кассового метода (статьи, флаг `isOpuCalculation`, нюанс дат у Начислений) → `OPERATIONS.md`, раздел «Расчёт прибыли из операций».

**Итоговые показатели (всегда):**

- Прибыль = Доходы − Расходы
- Рентабельность = (Прибыль / Доходы) × 100% (прочерк, если Расходы > Доходов)
- Нераспределённая прибыль: дополнительно включать операции по статьям **Дивидендов**; не распространять это правило на весь состав Расходов ОПУ

### ОПУ по проектам (`reportGenMethod: Projects`)

- Строки = проекты и их группы
- Для каждого проекта: Доходы, Расходы, Прибыль, Рентабельность
- «Без проекта» отображается как отдельная группа в конце списка
- Используются те же фильтры: период, валюта, метод, ЮЛ/счета, КА, проекты

⚠️ **План из бюджета этот endpoint не возвращает** — только факт. Если пользователь просит «план + факт по проектам» (например, «выполнение бюджета проекта»), идти в `LOOKUPS.md`, раздел «Projects → Финансовые агрегаты», и использовать `GET /api/v1/projects/calculations`: план там из сохранённого бюджета, факт — в одном из трёх методов расчёта.

Разграничение:

- нужен **план + факт** по проектам → `projects/calculations`;
- нужен **accrual-ОПУ по проектам с разбивкой по статьям** (план не нужен) → этот раздел (`POST /api/v2/reports/opu` с `reportGenMethod: "Projects"` в body);
- нужна **cash-динамика по подпериодам** (time-series) → `GET /api/v1/bizinfos/incomeoutcomehistorybyprojects` (см. `REPORTS.md`).

### Profit-метрики в ОПУ

| Флаг                | Метрика              | Формула                                       |
| ------------------- | -------------------- | --------------------------------------------- |
| `isGrossProfit`     | Валовая прибыль      | Выручка − Прямые расходы (тип «Прямой»)       |
| `isOperatingProfit` | Операционная прибыль | Валовая прибыль − Косвенные расходы           |
| `isEbitda`          | EBITDA               | Операционная прибыль + Амортизация            |
| `isEbit`            | EBIT                 | EBITDA − Амортизация (= Операционная прибыль) |
| `isEbt`             | EBT                  | EBIT − Проценты по кредитам                   |

Если пользователь не просит конкретную метрику прибыли, не передавать эти флаги (или ставить false).

## POST /api/v2/reports/dds — ДДС

### Назначение

Отчёт о движении денежных средств. Строится по **операциям движения денег** и дате оплаты (`operationDate`). Поддерживает разбивку по статьям / проектам / сделкам.

Для ДДС не выбирать метод `cash / accrual`: это не способ расчета прибыли, а отдельный отчет по деньгам.

### Ключевые параметры запроса

Те же, что у ОПУ, кроме:

- **Нет `isCalculation`** — у ДДС нет выбора между `cash` и `accrual`; он строится по операциям движения денег
- **Нет profit-флагов** (`isGrossProfit`, `isEbitda`, и т.д.)

| Параметр           | Тип     | Значения                                    |
| ------------------ | ------- | ------------------------------------------- |
| `reportGenMethod`  | string  | `OperationCategory` / `Projects` / `Deals`  |
| `isPeriodDetail`   | boolean | Разбивка по подпериодам                     |
| `standardPeriod`   | string  | `Day\|Week\|Month\|Quarter\|HalfYear\|Year` |
| `periodStartDate`  | string  | `YYYY-MM-DD` ⚠️                             |
| `periodEndDate`    | string  | `YYYY-MM-DD` ⚠️                             |
| `userCurrencyCode` | string  | `RUB\|USD\|EUR\|...`                        |
| `accountId`        | array   | Фильтр по счетам                            |
| `contrAgentId`     | array   | Фильтр по контрагентам                      |
| `projectId`        | array   | Фильтр по проектам                          |
| `withOther`        | boolean | Показывать «Остальные»                      |
| `includeTrendData` | boolean | Данные тренда                               |
| `aggregationMode`  | string  | `AutoAggregate\|KeepOriginal`               |

### Структура ДДС (по статьям)

| Строка                      | Статьи                                                                                     |
| --------------------------- | ------------------------------------------------------------------------------------------ |
| Операционная деятельность   | Доходы/Расходы + Оборотные активы (3.1) + Краткосрочные обязательства (4.1)                |
| Инвестиционная деятельность | Внеоборотные активы (3.2)                                                                  |
| Финансовая деятельность     | Другие статьи капитала + Вложения учредителей/Дивиденды + Долгосрочные обязательства (4.2) |
| **Общий денежный поток**    | Операционный + Инвестиционный + Финансовый                                                 |
| Остатки на начало периода   | (ПОСТ. + ПЕРЕМ.зачисл.) − (ВЫПЛ. + ПЕРЕМ.списан.) до начала периода                        |
| Остатки на конец периода    | Остатки на начало + Общий денежный поток                                                   |

Для ДДС использовать не только статьи `Доходов` и `Расходов`, а весь денежный слой: операционные, инвестиционные и финансовые движения, которые реально влияют на деньги.

### ДДС по проектам (`reportGenMethod: Projects`)

- Денежный поток = Поступления − Выплаты (только с подтверждённой датой оплаты)
- Поступления = ∑ Поступлений + Перемещений-зачислений
- Выплаты = ∑ Выплат + Перемещений-списаний
- **Начисления, Отгрузки, Поставки НЕ включаются**
- Остатки на начало = ∑ по проекту до начала периода (подтверждённая дата оплаты)
- Остатки на конец = Остатки на начало + Денежный поток
- **Остатки по счетам при разбивке по проектам не отображаются**

### ДДС по сделкам (`reportGenMethod: Deals`)

- Структура: Операционный поток (Поступления, Выплаты) + Инвестиционный поток (Поступления, Выплаты) + Общий поток
- Финансовый поток в разбивке по сделкам не выводится

## Быстрые фильтры ОПУ и ДДС

Если пользователь спрашивает про быстрый фильтр, сохраненный фильтр или вкладку **«Быстрые»** в ОПУ / ДДС, открывать `QUICKFILTERS.md`.

Короткое правило:

- быстрые фильтры ОПУ и ДДС хранятся раздельно;
- перенос фильтра между ОПУ и ДДС не предусмотрен;
- общий набор полей: период, юрлица и счета, контрагенты, проекты, сделки;
- для ОПУ дополнительно доступны виды прибыли;
- период может быть плавающим или фиксированным;
- при экспорте используется период, фактически примененный на вкладке **«Общие»**;
- для управления быстрыми фильтрами использовать методы `/api/v1/quick-filters*` из `QUICKFILTERS.md`, а отчётные данные строить через `POST /api/v2/reports/opu` или `POST /api/v2/reports/dds`.

## Trend endpoints

### POST /api/v2/reports/detail-trends/opu

### POST /api/v2/reports/detail-trends/dds

Используются для спарклайн-поповеров в UI — детализированные тренды по одному показателю.

**Когда нужны:** только если пользователь явно просит sparkline или интерактивный попап тренда по одному показателю.

> Не вызывать detail-trends дополнительно к основному report-endpoint. Слова «дашборд», «визуализация», «отчёт» — не триггер для этих endpoints. Они удваивают количество API-вызовов без добавления данных для стандартной аналитики.

Параметры:

- `type`: `Income | Outcome | Main`
- `section`: `Moves | Dividend | Profit | Proceed | Direct | Indirect | IncomeOther | OutcomeOther | ...`
- `comparisonMode`: `None | PreviousPeriod | PreviousYear`
- `filter`: основные параметры фильтрации (период, счета, проекты и т.д.)

## Правила для агента

### Выбор endpoint

- Для ОПУ и ДДС по умолчанию использовать **v2 endpoints**, а не собирать вручную из `/operations`.
- Для любого ДДС — summary или с детализацией — использовать `POST /api/v2/reports/dds`.
- Не строить ОПУ вручную из `/operations`, если v2 endpoint покрывает запрос.

### Метод расчёта

- ДДС — не отчёт с выбором метода `cash / accrual`. Не спрашивать пользователя про метод; строить по операциям движения денег и `operationDate`.
- `cash` в пользовательском вопросе про прибыль = способ расчета ОПУ по дате оплаты, а не синоним ДДС.
- ОПУ — **обязательно уточнить метод**, если пользователь не указал:
  - «Кассовый» → `isCalculation: false`
  - «Начисление» / «Акрual» → `isCalculation: true`
  - Не угадывать молча.

### reportGenMethod

- По умолчанию: `OperationCategory` (по статьям).
- `Projects` — если пользователь явно просит «по проектам».
- `Deals` — если пользователь явно просит «по сделкам».

### standardPeriod и isPeriodDetail

- Если нужна динамика по месяцам: `isPeriodDetail: true`, `standardPeriod: Month`.
- Если нужен итог за период без разбивки:
  - для `Projects` и `Deals`: `isPeriodDetail: false`;
  - для `OperationCategory`: всё равно задавать `isPeriodDetail: true` и `standardPeriod`, а итог без разбивки собирать локально.
- Если пользователь просит «помесячно», «поквартально» — установить соответствующий `standardPeriod`.

### Сравнение двух периодов

Если пользователь просит «сравни Q1 2025 с Q1 2024» или «рост расходов год к году»:

- **Один вызов**, а не два: задать общий диапазон (`periodStartDate` = начало раннего периода, `periodEndDate` = конец позднего), `isPeriodDetail: true`, `standardPeriod` = `Quarter` / `Month` / `Year`.
- Сервер вернёт колонки по каждому подпериоду — дельту и процент изменения считать **локально**.
- Для Баланса: один вызов `POST /api/v1/accounting/balance/table` с двумя датами-колонками, разницу считать локально.
- Не делать два отдельных вызова OPU/DDS на каждый период.

### Profit-метрики

- Если пользователь просит «прибыль» без уточнения — по умолчанию просто Доходы − Расходы (не включать `isGrossProfit` и т.д.).
- Если пользователь просит валовую прибыль → `isGrossProfit: true`.
- EBITDA → `isEbitda: true`.
- Не включать несколько флагов прибыли, если пользователь просил только один.

### Смешение методов

- Не включать `isCalculation: true` при запросе ДДС (его нет).
- Не смешивать ДДС и accrual-ОПУ в одной сводной цифре без явной маркировки.
- Если пользователь просит «ОПУ и ДДС вместе» — строить раздельно: ОПУ с явным методом, ДДС как отчёт по движению денег.

### Сравнение двух методов расчёта

Если пользователь хочет увидеть ОПУ и кассовым, и начислением — это два отдельных вызова `POST /api/v2/reports/opu` с разными значениями `isCalculation`.

Альтернатива — локальный расчёт из кэша: допустим при одновременном выполнении двух условий: (1) операции уже загружены в кэш `/api/v1/operations` и кэш актуален; (2) нужно сравнить оба метода в рамках одного запроса пользователя. В этом случае оба варианта считаются локально — один v2-вызов экономится. Во всех остальных случаях использовать v2 endpoint, а не строить ОПУ вручную из операций.

### Known caveats

- `reportGenMethod: "OperationCategory"` **требует** `isPeriodDetail: true` и `standardPeriod`. Без них и ОПУ, и ДДС возвращают ошибку «Статьи считаются только по периодам». Если `isPeriodDetail: true`, но `standardPeriod` не передан — ошибка «Nullable object must have a value». Поэтому для итогов по статьям без пользовательской просьбы о динамике всё равно запрашивать period detail и агрегировать итог локально. Для `Projects` и `Deals` ограничение не действует — `isPeriodDetail: false` работает.
- Если раздельные запросы по юрлицам (через `accountId`) покрывают все юрлица — итог считать локально, не делать дополнительный запрос без фильтра.
- Справочник, полученный ранее в сессии, не перезапрашивать — парсить первый ответ локально.
- ДДС по проектам не включает начисления, отгрузки и поставки — только реальное движение денег.
- ДДС по проектам не показывает остатки по счетам.
- ОПУ по проектам: «Без проекта» = отдельная группа в конце, не исключать автоматически.
- Рентабельность = прочерк, если Расходы > Доходов (не 0%, не отрицательное).
- Перемещения не участвуют в ОПУ, но участвуют в ДДС как движение денег.
