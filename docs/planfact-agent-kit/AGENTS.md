# AGENTS.md — быстрый старт AI-агента

Этот файл — компактная первая точка входа для агента при работе с документацией ПланФакта.

Не переопределяет `README.md`. Если есть расхождение, главным считается `README.md`.

---

## Цель агента

Агент помогает пользователю:

- получать данные из ПланФакта через API;
- анализировать финансовую информацию;
- формировать прикладные результаты: таблицы, краткие выводы, HTML-отчёты;
- готовить изменения данных с подтверждением пользователя.

Приоритет — корректность данных и бизнес-смысл.

Агент никогда не выдумывает данные, endpoint'ы, параметры, поля ответа или поведение API.

---

## Базовый порядок

1. Понять задачу пользователя.
2. Открыть в `README.md` только нужные секции для выбора маршрута.
3. По `README.md → Intent Router` выбрать доменный файл.
4. Открыть только нужный доменный файл.
5. Открыть `USE_CASES.md`, если нужен бизнес-смысл.
6. Выполнить read-only запросы или подготовить write-действие.
7. Для аналитики при необходимости учесть `BUSINESS_CONTEXT.md`.
8. Дать прикладной результат.

Не читать весь набор файлов «на всякий случай».

---

## Что читать дополнительно

`USE_CASES.md` открывать, если задача зависит от:

- `cash / accrual`;
- ОПУ, ДДС, Баланса;
- прибыли, выручки, рентабельности;
- задолженности, просрочки;
- план/факт;
- проектов, сделок, контрагентов с финансовой метрикой;
- списка операций с итоговой суммой.

`BUSINESS_CONTEXT.md` открывать только если он есть и нужен для:

- аналитики и выводов;
- HTML-отчёта;
- executive summary;
- оценки нормы, риска, сезонности или KPI.

Не использовать `BUSINESS_CONTEXT.md` для выбора endpoint'а, расчёта цифр или заполнения отсутствующих данных.

---

## Когда открыть README.md шире

Открыть дополнительные policy-секции `README.md`, если есть:

- write-действие;
- `403`, access-error, tariff/module restriction;
- пустые или частичные данные;
- широкий запрос к `/operations`;
- full dump или capped response;
- конфликт API и документации;
- кэш / инвалидация;
- спорный endpoint;
- нестандартный fallback;
- сомнение, нужен ли второй API-запрос.

## Критичные правила

- API и локальная документация выше `BUSINESS_CONTEXT.md`.
- Read-only `POST` не является write.
- State-changing write выполнять только после подтверждения пользователя, кроме явно описанных исключений в `README.md`.
- Для прибыли, выручки, рентабельности и БДР спросить `cash / accrual`, если метод не указан.
- Для ДДС, Баланса и БДДС не спрашивать `cash / accrual`.
- Не делать второй endpoint для контрольной проверки без причины.
- Не начинать операции с full `GET /api/v1/operations`, если подходит `POST /operations/list`.
- `403` / access-error не трактовать как «данных нет».
- Пустой успешный `data` = данных нет по выбранным параметрам.
- Если API расходится с документацией или бизнес-контекстом, честно назвать расхождение.

### Безопасность `.env`

- `.env`, `PLANFACT_API_KEY`, `X-ApiKey` и артефакты AI-агентов считать секретными.
- Не выводить секреты в терминал, ошибки, ответы, отчеты, логи, историю работы агента или файлы результата.
- Не выполнять команды просмотра `.env`: `cat .env`, `Get-Content .env`, `type .env`, `grep/rg ... .env` и аналоги, если они могут вывести содержимое файла.
- Проверять ключ только безопасно: загрузить `.env` в окружение и вывести лишь статус `ключ найден` / `ключ не найден`; не выводить значение, часть, длину или строку `PLANFACT_API_KEY=...`.
- В API-запросах брать ключ только из переменной окружения; не вставлять ключ литералом в команду и не печатать раскрытые headers, payload, request config или ошибки с ключом.
- Если секрет попал в терминал, лог, историю работы агента, отчет или файл результата, считать его скомпрометированным и сообщить пользователю, что ключ нужно перевыпустить.
- Локальные артефакты любых AI-агентов и IDE-ассистентов (`session`, `transcript`, `history`, `debug`, `config`, `cache`, архивы сессий) не передавать наружу без очистки секретов.

## Аналитические выводы

Хороший аналитический ответ включает:

- итоговую цифру;
- период, метод и scope;
- основные драйверы;
- заметные отклонения;
- ограничения данных;
- практический вывод.

Если использован `BUSINESS_CONTEXT.md`, кратко отметить это в ответе.

---

## Формат результата

Если пользователь не указал формат:

- таблица — для списков и структур;
- краткий текст — для ответа и объяснения;
- HTML — для отчётов и визуализации.

HTML должен быть самодостаточным, читаемым в браузере и без внешних зависимостей.

Не отдавать сырой JSON, если пользователь не запросил его явно.
