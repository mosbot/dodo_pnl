# План: фоновый прогрев immutable-данных Финансов

Цель — убрать «долгое первое открытие» в Финансах. Принцип: **неизменные
(закрытые) данные подгружаются фоново и единоразово**, пользователь никогда не
ждёт живой фетч из Dodo IS / PlanFact для истории. Реализацию пока НЕ делаем —
это план.

## Что вообще immutable (и уже кэшируется)

Закрытый полный месяц (вне live-окна `is_period_in_live_window`, глубина
`planfact_keys.live_months_window`) неизменен. Уже есть слои:

| Слой | Таблица | Режим | Чем наполняется |
|---|---|---|---|
| P&L-агрегаты закрытых месяцев | `cache_history` | Full (PlanFact) | `_build_pnl_for_period` (cache_mode="save") |
| Выручка+каналы закрытых месяцев | `lite_revenue_cache` | Lite | `_lite_revenue` (cacheable=True) |
| Ops-метрики месяца | `ops_metrics` | оба | `_run_ops_sync` |
| LY полный месяц (для прогноза) | `monthly_revenue_history` | оба | `get_or_fetch_ly_full_month` |
| Имена пиццерий | `dodois_units_cache` | оба | `_get_or_refresh_unit_names` (TTL 24ч) |

Прогрев = заранее, в фоне, прогнать **те же builder'ы** для нужных закрытых
месяцев. Builder'ы идемпотентны: если месяц уже в кэше — no-op. Прогрев лишь
переносит первый (дорогой) проход с «момента открытия пользователем» на «фон
после входа».

## Какие сценарии Финансов закрываем

Все сценарии читают одни и те же месячные слои, поэтому прогрев месяца закрывает
их разом:

1. **Месяц** — `cache_history` / `lite_revenue_cache` + `ops_metrics` за месяц.
2. **Период** (диапазон) — те же слои, помесячно.
3. **График истории выручки** (12 мес) — `cache_history` / `lite_revenue_cache`
   за каждый месяц окна (+ LY-окно при включённом сравнении).
4. **LFL / сравнение год к году** — те же слои за месяцы прошлого года.

Целевой набор месяцев для прогрева:
`closed_months = [now−1 … now−N] минус live-окно` (N по умолчанию 12), плюс их
LY-двойники `[now−12−1 … now−12−N]` (ещё 12) — покрывает и график, и LFL.
**Текущий/частичный месяц и live-окно НЕ прогреваем** (меняются → всегда live).

## Оркестратор

Новый модуль `app/warmup.py` (или функции в main):

```
async def warm_financials(session, user|pf_key_id, *, months_back=12,
                          include_ly=True, with_ops=True): ...
```

Логика:
1. Режим тенанта — `_tenant_has_planfact` (Full vs Lite).
2. Считаем `closed_months` (+ LY при include_ly), отбрасываем live-окно.
3. Для каждого месяца, которого ещё нет в кэше:
   - Full: `_build_pnl_for_period(..., cache_mode="save")` → пишет `cache_history`.
   - Lite: `_lite_revenue(..., cacheable=True)` по проектам → пишет `lite_revenue_cache`.
   - Ops (если `with_ops`): `_run_ops_sync(period=m)` → пишет `ops_metrics`.
4. `monthly_revenue_history` — попутно через существующий ленивый путь, либо
   отдельным проходом, если нужен прогноз без визита.

Порядок — **выручка/P&L сначала** (дёшево, ~1.5с/мес, делает графики и
страницу мгновенными), **ops потом** (дорого, ~45с/мес, фоном, ниже приоритет).

## Защита и нагрузка

- **Только закрытые месяцы** (immutable) — инвариант, проверяем
  `is_period_in_live_window`.
- **Идемпотентность** — builder'ы skip'ают закэшированное; повторный прогрев
  дёшев. Запись через ON CONFLICT (уже так в `lite_revenue_cache`/cache).
- **Анти-долбёж Dodo IS** — месяцы прогреваем **последовательно** (или
  семафор ≤2), внутри уже batched ≤30 юнитов / ≤6 параллельных
  (`dodois_client._MAX_PARALLEL`). Не запускать ops-прогрев параллельно с
  пользовательским ops-синком того же месяца (переиспользовать
  `_OPS_SYNC_INFLIGHT`).
- **Inflight-гард на тенант** — set `_WARM_INFLIGHT[(pf_key_id)]`, как
  `_OPS_SYNC_INFLIGHT`, чтобы конкурентные входы не плодили прогревы.
- **Дебаунс** — `_warm_last_run[pf_key_id]` (in-memory, TTL напр. 6ч): не
  прогревать чаще раза в N часов. После рестарта кэш в БД жив → повторный
  прогрев всё равно дёшев (skip).
- **Токен** — через брокер sa; нет токена / нет `dodo_unit_uuid` → грейсфул
  выход, без падений.
- **Мульти-воркер** — гард per-process; худший случай — 2 прогрева, builder'ы
  дедупят на уровне БД. При желании — Postgres advisory-lock на (pf_key).

## Триггеры (как запускается)

1. **On-demand при первом заходе в Финансы** (основной путь):
   в `get_pnl` после возврата ответа — `asyncio.create_task(warm_financials(...))`
   (fire-and-forget, как текущий ops-trigger), задебаунсенный `_warm_last_run`.
   Пользователь видит текущий месяц сразу, история догревается в фоне.
2. **(Опц.) Ночной прогрев всех тенантов** — у pnl нет Celery (есть у sa).
   Вариант: внутренний `POST /internal/warm-all` в pnl (под `X-Admin-Token`),
   который дёргает sa-Celery-beat по расписанию. Тогда даже первый дневной
   визит — уже тёплый. Включаем после п.1.
3. **Ролловер месяца** — отдельного триггера не нужно: 1-го числа только что
   закрывшийся месяц станет immutable, и ближайший on-demand прогрев его
   подхватит (он окажется не в live-окне и не в кэше).

## Наблюдаемость / UX

- Лог по месяцам: `warm <pf_key> <month>: revenue 1.4s / ops 44s`.
- (Опц.) `GET /api/financials/warm-status` → фронт может тихо показать
  «прогреваю историю…» на графике, не блокируя страницу. Но базово —
  молча: данные просто появляются.

## Конфиг / флаги

- `enable_financial_warmup: bool = False` — общий killswitch.
- `warm_months_back: int = 12`, `warm_include_ly: bool = True`,
  `warm_with_ops: bool = True`.
- `warm_min_interval_sec: int = 21600` (6ч дебаунс).

## Фазы внедрения

1. **Фаза 1 — прогрев выручки/P&L** (cache_history / lite_revenue_cache) за
   12 закрытых + 12 LY месяцев, on-demand при входе. Дёшево, убирает 30-сек
   ожидание истории. (Самый большой эффект.)
2. **Фаза 2 — прогрев ops** (`ops_metrics`) за 12 месяцев, фоном, ниже
   приоритет (тяжёлый). Убирает 45-сек ops-ожидание на первый просмотр месяца.
3. **Фаза 3 — ночной прогрев всех тенантов** (sa-Celery → pnl internal),
   чтобы первый дневной визит был мгновенным.

## Чего НЕ прогреваем

- Текущий и live-window месяцы (меняются).
- Тенанты без `dodo_unit_uuid` (нечего тянуть).
- Board/Пульс-окна — отдельный механизм (#2, success-флаг fetch); этот план
  только про Финансы.

## Переиспользуемые куски (минимум нового кода)

`_build_pnl_for_period`, `_lite_revenue`, `_run_ops_sync`,
`store.is_period_in_live_window`, `_tenant_has_planfact`,
`pnl_module.month_range`, `_month_range_dates`, `_OPS_SYNC_INFLIGHT`.
Новое: `warm_financials` + гард/дебаунс + триггер в `get_pnl` (+ опц. internal
endpoint для ночного прогона).
