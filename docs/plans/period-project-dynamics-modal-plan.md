# План: динамика метрик по пиццерии в режиме «Период» (Вариант B)

Цель — в режиме **Период** дать ценность «смотреть динамику метрик по
конкретной пиццерии»: карточки остаются компактными (сумма за период), по клику
открывается **модалка** с помесячной детализацией одной точки — график тренда +
таблица метрики×месяцы со спарклайнами.

## UX

1. В режиме Период на каждой карточке пиццерии — аффорданс «↗ помесячно»
   (подсказка-ссылка). Селектор «Суммарно/Сравнение» НЕ нужен: карточки и так
   суммарные, а модалка = сравнение по месяцам. (Если позже захотим — добавим
   глобальный тумблер, но в B он избыточен.)
2. Клик по карточке (или по «↗ помесячно») → модалка «<Пиццерия> · помесячно»:
   - **Сверху — график тренда** по выбранной метрике (переключатель-чипы:
     Выручка / Заказы / KC / Доставка / РК / …). Бары или линия по месяцам.
   - **Снизу — таблица метрики×месяцы**: строки — метрики (Выручка, Заказы,
     каналы, ops), столбцы — месяцы периода. В каждой строке метрики —
     **мини-спарклайн** тренда (динамика видна по всем метрикам сразу).
   - Месяцев много (>8) → горизонтальный скролл таблицы (как «Детализация»).
   - (Опц., Фаза 3) колонка «Δ к прошлому году» по каждой метрике.
3. Модалка переиспускает существующую инфраструктуру (`.modal`/`.modal-content`,
   drill-модалка). График — Chart.js (уже на странице).

## Данные

Нужна матрица **по одной пиццерии: метрика × месяц**. Источники (всё помесячно
по точке уже доступно или дёшево собирается):

| Метрика | Источник per (project, month) |
|---|---|
| Выручка + каналы (Доставка/Ресторан) | `/finances/sales/units/monthly` за каждый месяц (как Lite/`_lite_revenue`); для full — он же или cache_history |
| Заказы (+каналы) | тот же monthly-эндпоинт, поле `ordersCount` (как `_fetch_monthly_orders`) |
| Ops: KC/DC, AOT, готовка, ср.доставка, РКО, РС, РК | таблица `ops_metrics` — УЖЕ помесячно по проекту (DB-чтение, быстро) |

Полные accrual-строки PlanFact (EBITDA, чистая прибыль) per (project, month) —
тяжелее (помесячные PlanFact-сборки). Выносим в Фазу 2; в Фазе 1 берём то, что
дёшево: выручка/каналы/заказы (Dodo) + все ops-метрики (DB). Это уже покрывает
операционную динамику, ради которой всё и затевалось.

## Бэкенд

Новый эндпоинт (на одну точку, по клику — не на каждый /api/pnl):

`GET /api/pnl/project-monthly?project_id=&date_start=&date_end=&method=&compare=`

Ответ:
```json
{
  "project": {"id": "899038", "name": "Кубинка-1"},
  "months": ["2026-01","2026-02","2026-03","2026-04"],
  "metrics": [
    {"code":"REVENUE","label":"Выручка","format":"rub","direction":"higher",
     "values":[4210000,4480000,4050000,4710000],
     "prev_values":[...]   // если compare
    },
    {"code":"ORDERS","label":"Заказы","format":"int","values":[5946,...]},
    {"code":"KC_LIVE","label":"KC","format":"pct","direction":"lower","values":[31.2,...]},
    {"code":"AVG_DELIVERY","label":"Доставка","format":"mm_ss","values":[1983,...]},
    {"code":"CUST_RATING","label":"РК","format":"num2","values":[4.73,...]}
  ]
}
```
- `months` = `_months_between(date_start, date_end)`.
- Метрики формируем из:
  - REVENUE/каналы/ORDERS — цикл по месяцам: revenue через `_lite_revenue`
    (для Lite) либо per-month revenue полного пути; orders через
    `_fetch_monthly_orders`. Оба уже умеют один проект/диапазон → вызвать на
    каждый месяц (или, где есть, читать кэш).
  - ops-метрики — `store.list_ops_metrics(pf_key, period_month=m, project_id=...)`
    по каждому месяцу окна; мету/формат берём из `store.OPS_METRICS`
    (переиспользуем формат-флаги mm_ss/digits/direction, что уже есть на фронте).
- `compare` → второй проход по LY-месяцам, `prev_values`.
- Перф: ops — быстрые DB-чтения; revenue/orders — N запросов в Dodo (N=число
  месяцев, ≤ ~12, батч 1 юнит) → ~секунды на клик; кэш monthly уже есть
  (`_ORDERS_CACHE`/`lite_revenue_cache`). Прогрев истории (уже сделан) делает
  большинство месяцев тёплыми.
- Гейт capability `finance` (как остальные pnl-эндпоинты).

## Фронтенд

- Режим Период: на карточке пиццерии добавить ссылку «↗ помесячно»
  (и/или весь клик по карточке). Только когда `state.mode === 'period'`.
- Новая функция `openProjectMonthlyModal(projectId)`:
  - `GET /api/pnl/project-monthly?...` с текущим диапазоном (+ compare, если
    включён LFL).
  - Рендер модалки: чипы метрик → Chart.js (bar/line по месяцам); таблица
    метрики×месяцы; в каждой строке — `<canvas>`-спарклайн (Chart.js,
    отключённые оси/легенда) или лёгкий inline-SVG.
  - Форматирование значений переиспользуем (mm:ss, %, ₽, num2) — те же
    форматтеры, что в плитках.
- CSS — переиспользуем `.modal`, `.dense-table`/`drill-table`, добавим класс
  для спарклайн-ячейки.

## Фазы

1. **MVP**: модалка по клику; метрики Выручка + каналы + Заказы (Dodo) + все
   ops-метрики (DB); таблица метрики×месяцы + спарклайн в строке. Без графика-
   переключателя (или сразу простой bar по Выручке).
2. **График-переключатель**: чипы метрик → Chart.js тренд выбранной метрики.
3. **LFL-колонка** (Δ к прошлому году) + accrual-строки PlanFact (EBITDA и т.п.)
   через помесячные сборки полного пути.

## Открытые вопросы

- Клик по всей карточке vs отдельная ссылка «↗ помесячно» (чтобы не мешать
  возможным будущим действиям на карточке). Предлагаю — вся карточка кликабельна
  в режиме Период + видимая подсказка.
- Набор метрик в модалке по умолчанию (все ops + выручка/заказы) и их порядок.
- Спарклайн: Chart.js (тяжелее, но единообразно) vs inline-SVG (легче). Склоняюсь
  к inline-SVG для строк (их много) + Chart.js для большого графика сверху.
