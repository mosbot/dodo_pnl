# План миграции P&L на `POST /api/v2/reports/opu`

Дата: 2026-06-10. Основания: `planfact-api-efficiency.md` (находка №1),
`v2-reports-opu-poc.md` (net profit сошёлся до копейки, ~350× меньше трафика).

## Цель и ожидаемый эффект

Заменить источник агрегата для P&L: вместо `GET /operations` (30–50 МБ
сырых операций за месяц + хрупкий `_fetch_ops_recursive` date-split) — один
`POST /api/v2/reports/opu` (~80–200 КБ готового агрегата).

Эффект на загрузку дашборда (live-месяцы; закрытые и так из cache_history):

| Сценарий | Сейчас | После |
|---|---|---|
| Месяц, ключ PiX | ~30 МБ, recursive-split, 5–15 с | 1 POST, ~1–2 с |
| Месяц, ключ XFood (30+ точек) | десятки МБ × split | 1 POST |
| «Период» 12 мес | 12 × full-fetch | 12 лёгких POST |
| Пересборка снапшота закрытого месяца («Обновить») | full-fetch | 1 POST |

Drill-down `/api/operations` НЕ трогаем — там нужен список операций, объёмы
малы (limit/offset работает у `list_operations`).

## Ключевая архитектурная идея

v2 используется **только как источник сумм `(project_id, category_id) → amount`**.
Классификация остаётся нашей: `cat_totals` прогоняется через существующий
`_build_category_index` (template_path_to_code → leaf → эвристика), из него
выводятся `totals` (pid|pnl_code) и `revenue_by_channel`. Это снимает
расхождения из POC (Revenue/MARKETING — артефакты v2-шной группировки):
v2-группировку мы просто не используем. Шаблон, метрики, формулы, таргеты,
visibility — нетронуты.

Совместимость уже есть: `build_pnl` умеет режим `cached_aggregates`
(`pnl.py:545`), формат `{"totals": {pid|code}, "cat_totals": {pid|cid},
"revenue_by_channel": {...}, "active_project_ids": [...]}` — адаптер v2
собирает ровно его.

### Исходный план этапа (для истории)

Скрипт `/tmp/probe_v2_opu.py` на VPS (по образцу POC), проверяем на ключе PiX:

1. **Структура ответа**: есть ли в `reportGenMethod: "Projects"` категорийная
   разбивка per-project и присутствуют ли **`operationCategoryId`** (не только
   title). POC-файлы `_scratch/v2-*.json` удалены — переснять.
   - Если ID есть → адаптер тривиален.
   - Если только path/title → маппинг через `path_str` (он у нас уже
     канонизирован в `_build_category_index`); риск коллизий имён — пометить.
   - Если Projects не даёт категорий → fallback-вариант:
     `reportGenMethod: "OperationCategory"` + `projectId: [pid]` per-project
     (N лёгких POST; для OperationCategory обязательны `isPeriodDetail: true`
     + `standardPeriod` — REPORTS_V2.md:343).
2. **Клиент без настроенного шаблона ОПУ** (открытый вопрос POC): создать
   «голый» PF-ключ, дёрнуть v2 → defaults / пусто / 403. Определяет
   детект-условие для fallback.
3. **Большие периоды**: 12 мес × все проекты XFood-ключа — время ответа,
   размер, rate-limit headers (`X-RateLimit-*`).
4. **Кассовый метод**: `isCalculation: false` на одном месяце, сверить с
   текущим `method=cash`.
5. **План-операции**: в accrual v2 берёт только `isCalculationCommitted` —
   совпадает с нашим `require_committed=True`. Проверить крайний случай
   текущего месяца (плановые операции будущих дат).

Выход этапа: короткий апдейт этого файла (формат ответа + решение по
fallback-условию). Если п.1 даёт «категорий нет вообще» — миграция
останавливается, остаёмся на raw.

## Этап 2 — адаптер + интеграция (~1 день)

`app/pnl.py` (или новый `app/v2_adapter.py`):

- `v2_to_aggregates(report, cat_index) -> dict` — flatten листьев v2 в
  `cat_totals {pid|cid: amount}` (+ знак: расходные категории в v2 уже
  с нужной семантикой — проверить в этапе 0, у нас расходы хранятся
  положительными в `cat_totals` и знак даёт `PNL_CODES`-семантика);
  затем из `cat_totals` × `cat_index` вывести `totals` и
  `revenue_by_channel` (код переиспользовать из основного цикла агрегации —
  вынести общую функцию `_aggregate_from_cat_totals`).
- `main.py::_build_pnl_for_period`: в live-пути вместо
  `_fetch_period(...operations...)` →
  ```
  projects, categories = gather(list_projects, list_operation_categories)
  report = await pf.report_opu(...)
  aggregates = v2_to_aggregates(report, cat_index)
  build_pnl(..., cached_aggregates=aggregates)
  ```
  `unclassified`-диагностика: считается из cat_totals (категории без
  pnl_code) — в отличие от cache-path тут данные есть.
- Снапшоты закрытых месяцев (`cache_mode="save"`): сохранять агрегат,
  полученный из v2, в той же структуре — формат `cache_history` не меняется,
  миграция БД не нужна.

## Этап 3 — feature-flag + fallback (~0.5 дня)

## Этап 4 — shadow-mode (~2–3 календарных дня)

- Режим `'shadow'`: основной ответ строится из raw (как сейчас), параллельно
  тянется v2, адаптируется, и в лог пишется дельта по инвариантам:
  `revenue`, `net_profit`, число категорий, max |Δ| по строкам.
  Допуск: |Δ| ≤ 0.01 ₽ (копейка на округлении).
- Включить shadow на PiX-ключе; через 2–3 дня живой эксплуатации
  (включая смену месяца и кнопку «Обновить») собрать лог:
  `journalctl | grep v2-shadow`.
- Отдельно прогнать shadow на «Период» (12 мес) и на cash-методе.

### Результаты валидации (2026-05, оба ключа, оба метода)

**PiX / accrual — ПОЛНОЕ СОВПАДЕНИЕ** lines/template_lines/channels после
фикса адаптера (PF позволяет вешать операции на родительские категории —
адаптер берёт own-вклад узла = totalValues − Σ детей, а не только листья).
Единственная разница в category_breakdown: Capital-категория «Корректировка»
(−2.1M) — v2 балансовые категории кроме дивидендов не отдаёт by design,
на строки P&L не влияет. Live-подтверждение: `v2-shadow 2026-06 accrual:
OK — 22 строк сходятся` на реальном MTD-запросе.

**PiX / cash — v2 ПРАВИЛЬНЕЕ raw.** Δ ровно 927 945.59 в RENT — это
6 НЕПОДТВЕРЖДЁННЫХ (`isCommitted=false`, т.е. плановых) оплат коммуналки
на 31 мая: Электроэнергия Немчиновка-1 (161 664.26 — ТРИ дубля!),
Алексин-1 (226 909.80), Протвино-1 (151 960.00), Вода Протвино-1
(64 083.01). PF cash-факт = только подтверждённые операции
(REPORTS.md:130) — v2 так и считает; старый raw-путь брал всё подряд и
показывал план как факт. NB: дубли Немчиновки (3 × 161 664.26) — мусор
в самом PlanFact, стоит почистить до их подтверждения.

**Xfood — НЕ МИГРИРОВАТЬ без отдельного решения.** Разбор 2026-06-10
(diag5/diag6) показал ДВЕ независимые причины расхождений ±15–26M:

1. **Депозиты-овернайты в выручке raw.** Операции «Переводы в пути» на
   служебном проекте 977879 («Служебный проект для нераспределенной части
   операции»): «Перечисление средств для открытия депозита… UNV-…» /
   «Возврат средств по депозитной сделке…» по 7–10.5M ежедневно (примеры:
   op 272851060 на 10.5M, op 272851053 на 10M). PF в ОПУ переводы не
   включает (REPORTS_V2.md: «Перемещения не участвуют»); наш raw-классификатор
   относил их в REVENUE/канал other — **35.4M фиктивной «выручки» за май
   в текущем проде Xfood**. Тут v2 безусловно правильнее.

2. **PF применяет правила распределения на лету в отчётах.** Контрольный
   пример — точка ЧИЗЗИ (1206346) за май: ПРЯМАЯ постраничная выгрузка
   операций PF по filter.projectId даёт income 209 154 ₽ (121 частей;
   fetch_all_operations даёт ровно столько же — наш сплит НИЧЕГО не
   теряет), а v2-ОПУ показывает на той же точке ~4.16M. Т.е. PlanFact
   при построении отчёта перераспределяет суммы между проектами по
   настроенным правилам, которые НЕ материализованы в operationParts.
   Отсюда же зеркальные дельты LC (raw 16.9M vs v2 1.9M) и OTHER_OPEX
   (raw 50M vs v2 24M).

Вывод: v2 = managerial-отчёт, как Xfood настроил его в PlanFact UI;
raw = бухгалтерский разрез сырых операций. Вопрос «какой из них —
правильный P&L по точке» должен решить владелец данных Xfood, сверив
одну точку (например ЧИЗЗИ за май) со своим отчётом в PF UI. Если их
эталон — PF UI (почти наверняка), переключаем на v2; raw в любом
случае искажён депозитами (п.1).
